Knowledge and documentation

AGENTS.md is a pilot's checklist, not a style guide. It lands in the system prompt every turn, so it's the single highest-leverage configuration point. Keep it short and earn each line: every rule traces to a specific past failure or hard external constraint, or it's noise and exhausts the context window.

The session log is the substrate of recoverability. An append-only event log of every thought, tool call, and observation is what makes a long run debuggable, auditable, and resumable. Without one, a container failure is a session failure — what you have is a long-running shell script that happens to call an LLM, not a long-running agent.

Context and defensibility

Whatever an agent does not know about your business at the moment of execution, it makes up. Most of the time the made-up answer is plausible enough that you don't notice — until you do, and the cost is whatever was built on top of the bad assumption. A finite context window plus a "looks plausible" filter is exactly the wrong combination for high-stakes work.

Specs and knowledge bases are the same idea at different scales. A spec captures the durable intent of one piece of work. A knowledge base captures the durable intent of one business. Together they're how a finite context window can act with the company's actual taste, instead of generic competence.

Intent and specification

Bugs are deviations from intent - but it is worth defining prioritization of intent across product, user, operator, regulator, and security. This is something to be addressed at the spec level of the system.

The unit of correctness is the spec, not the code. Code executes; agents execute. Wrongness in software is almost never "the machine made a mistake" — it's that the intent the machine was acting on was incomplete, ambiguous, or never written down. The first question when something goes wrong is not what is the agent doing weird? but what did I tell it to do, and did that description capture what I wanted?

The spec and the test are the work; the prompt and execute steps are the cheap part. Code generation has compressed to near-zero; what hasn't compressed is the precision of the description the code is generated from. Vague sketch + no tests + many prompt cycles produces software whose correctness no one can verify.

A spec is a living product contract; specs are never finished. A spec is the most up-to-date statement of what the work is trying to achieve, not a deliverable produced once at the start. When the code changes, the relevant specs change in the same commit. A spec that drifts from the system it describes is worse than no spec, because it gives false confidence.

If the agent could plausibly produce ten different valid implementations, you need a technical spec to pick the one you want. Most product surfaces don't need a technical spec — any reasonable implementation of a message thread will do. Systems with non-trivial technical depth (livestreaming pipelines, custom video players, real-time sync engines) do.

The appreciating skill is writing specs that survive contact with an autonomous executor. Defining work crisply enough that an agent can run on it for a day is harder than doing the work yourself. That definition is the human's job and isn't going away.

Verification

Write the test first, use it to drive the interface, then refactor. The Continuous Integration tradition sharpens it further: tests, linters, and other verifiers are the automated health check of the codebase.

Use the test pyramid, paired with Software Engineering at Google's practical advice to skew toward maintainable unit tests while still keeping broader-scoped tests; the Google rule of thumb is roughly 80% unit and 20% broader tests.

Even formal verification does not save you if the spec is incomplete, because perfect verification can only establish conformance to the specification, not that the specification itself is right.

Grade outcomes, not trajectorial theater. Recent agent-evals practice lands on the same lesson: over-specifying the path an agent must take leads to brittle evals. Evaluate what matters in the environment, then read transcripts to see whether the eval is fair.

Use the least expensive reliable signal that matches the risk. Interaction tests, larger tests, and inferential graders all have legitimate uses; the real rule is matching signal cost to risk.

The loop must refuse to let the agent stop early. Models declare "task complete" too soon. A wrapper that re-injects the goal into a fresh context window when the agent claims it's done is the simplest defense (the Ralph pattern).

Surface correctness is not systemic correctness. Code that compiles, lints, and passes tests still hides ordering changes, edge-case defaults, and decisions the human never made. The "looks plausible" filter is exactly the wrong filter when the author is a model.

The test spec is the human's seat at the table. Code-level tests (unit, integration, smoke) can be regenerated by the agent at any time; what stays human-readable is the plain-English test plan that says these are the conditions we're verifying. When the descriptive spec changes, the test spec changes; when the test spec changes, the code-level tests are regenerated.

Modularity and the unit of change

Decompose around volatility and language boundaries, not around execution order or presentation layers. When the point of modularity is safe change, modules should hide the decisions most likely to vary, because the whole value of decomposition is to contain blast radius. Start from difficult design decisions and likely changes, and hide those behind module boundaries.

Vertical-slice intuition within the scope of a productized agency is canonically sound if the slice owns a coherent concept end-to-end. In practice that means each slice should be promptable, reviewable, and testable as a unit, with contracts at its edges and no hidden cross-slice magic.

Anti-patterns: layer-first decomposition, temporal decomposition, and "we'll share the same model name everywhere."

A change should do one thing and be small enough for a reviewer to understand end-to-end. Ship small diffs if you want any realistic chance of inspecting them for correctness.

This becomes dramatically more important once an agent can generate a five-file refactor, a migration, and three incidental cleanups in a single turn. Harnesses should enforce scope budgets, change descriptions, and maybe even "one primary intent per diff."

The unit of review is the unit of comprehension. A change too large to actually read is a change you will ratify rather than review. This is the human-side rationale behind Google's ~100-line PR norm; the agent-side rationale is the same — once a diff stops being readable, "approved" stops meaning anything.

Decouple Brain, Hands, and Session. The model + harness loop (Brain), the sandboxed execution environment (Hands), and the append-only event log (Session) should be independently replaceable. Coupling them means a stale assumption forces the whole system to change at once; decoupling them lets sandboxes be cattle, brain crashes recoverable, and harness pieces removable as the model grows into them.

Capabilities and inventory

A capability is a vertical slice that owns every layer it touches. Database tables, API routes, shared types, frontend components — all in one folder. The unit of change is the capability, not the codebase.

Folder copy is the portability test. A capability is portable if dropping its folder into a new repo applies the schema, mounts the routes, exposes the types, wires up the screens, and declares the env vars — without further surgery. The test is a physical constraint, not a design rule. There is nothing to debate.

A capability is not done until it is substrate inventory. Every finished client feature is rewritten — as part of the same goal — into a portable capability in the shared library. Skip that step and the inventory doesn't grow; the system goes back to being a normal codebase that gets harder over time.

Build for one, deploy to many. A capability that ships for client A and is useful for client B only if A's specifics are scrubbed first is not reusable — it's a snowflake. Folder-copy portability is what keeps the inventory honest.

Codebases grow by accretion; capability libraries grow by inventory. Accretion entangles features over time, so the whole codebase becomes the unit of change. Inventory adds independent, self-contained slices that travel together across every layer. The fiftieth client engagement starts with forty-nine capabilities the first one didn't have.

The human-machine boundary

Humans own intent, boundaries, harnesses, and acceptance; agents own searchable labor.

Context engineering for coding agents is curating what the model sees.

The real discipline is harness engineering, not prompt engineering: specs, rules, examples, fast tests, structural checks, code review surfaces, transcripts, outcome-based evals, intervention paths, and explicit handoff artifacts across sessions.

Mature oversight looks less like approving every action and more like observing traces and intervening quickly when the system drifts.

The default behavior of any AI coding agent is the shortest path to "done." The reward signal points at "task complete," not "task complete and the design doc exists." Senior-engineer work — surfacing assumptions, sizing changes, writing the spec, leaving evidence, refusing to merge what can't be reviewed — is exactly what gets skipped unless skipping is made impossible.

Small human surface area, large agent surface area. The human belongs only where context, judgment, and accountability matter — defining intent, prioritizing work, accepting what ships. Everything else belongs to the loop. Anything more elaborate is overhead, and overhead is what stops systems like this from scaling.

Harness engineering

Agent = Model + Harness. A raw chat model is not an agent. It becomes one when a harness gives it state, tool execution, feedback loops, and enforceable constraints. If you're not the model, you're the harness.

A decent model with a great harness beats a great model with a bad harness. Most leverage lives on the harness side of the equation, not the model side. The gap between what current models can do and what you see them doing is largely a harness gap.

The skills are the prompts. Mature agent systems don't prompt; they invoke skills. Each repeatable workflow lives in a versioned file with triggers, instructions, and edge cases; the harness routes to the right skill; the skill remembers the lessons the prompt window forgets. If you're hand-crafting prompts at the keyboard every session, the workflow isn't yet substrate — and the next person (or the next session) starts from scratch. (After Garry Tan.)

Skills compose; improving one improves every workflow that uses it. A complex workflow is a chain of focused skills, each doing one thing well. When cross-modal-eval gets sharper, every skill that calls it gets sharper for free. Composability is the unit of expressiveness; per-skill quality is the unit of leverage. Ten focused skills beat one mega-skill for the same reason ten focused tools beat fifty overlapping ones.

Models are interchangeable; the skill decides which model to call. Precision tasks call one model; recall tasks call another; creative work calls a third; latency-critical work calls a fourth. The skill encodes the routing; the harness doesn't care. "Which model is best?" is the wrong question — every frontier model is best at something, and the substrate is what decides what. This is also why the harness has to stay thin: model selection is a per-task decision that lives next to the work, not in a global router.

Failures are configuration problems, not model problems. The blame-the-model default is wrong by default. Most observed failures are legible and fixable in the harness — call it the "skill issue" reframe and fix it where it lives.

Every harness component encodes an assumption about what the model can't do on its own. When a component can't be traced to a specific behavior the model would otherwise fail at, it shouldn't exist. When the model gets good at the behavior, the component should come out.

Ratchet, don't brainstorm. Add a constraint only after a real failure; remove it only when a capable model has made it redundant. Every line of AGENTS.md, every hook, every rule should trace to a specific thing that went wrong or a hard external constraint.

Expect v1 to be terrible; the fix ratchets into the skill. First-pass skills make embarrassing mistakes — factual errors, missed edge cases, generic-sounding output. That's the point. Every failure is what you ratchet the skill against, so every future invocation gets the lesson baked in. The system gets better not because the model improves but because the substrate absorbs what went wrong; the hundredth invocation is good because the first ninety-nine taught the skill something. (After Garry Tan.)

Work backwards from behavior. Start from the behavior you want (or want to fix) and derive the harness piece that delivers it. If you can't name the behavior a component exists to deliver, drop it.

Success is silent; failures are verbose. Hooks and checks should say nothing on success and inject actionable error text on failure. This makes the feedback loop nearly free in the common case and self-correcting when it isn't.

Harnesses don't shrink as models improve; they move. Old scaffolding becomes dead code; the ceiling rises and exposes new failure modes that need new scaffolding. The right harness is shaped by your failure history — it can't be downloaded.

Tools and external systems

When an agent talks to an external service, wrap it. Three things every wrapper should do: keep a local copy of the data the agent will keep coming back to; let the agent ask compound questions in one shot; return small structured output sized for an agent's context, not a human's screen. Whether you call the wrapper an agent-native CLI, an MCP server, or a script in bin/ matters less than the shape.

Local mirror beats remote API call. State out of the context window, queryable offline, replayable across sessions. The 8,000 lines of JSON belong on disk; the agent should see the table.

Compound commands beat ten round trips. A single command that does the work of ten — search, merge, sort, filter — keeps the context budget intact and the latency low. Raw HTTP is what you'd do with curl; agent-native tooling is what you'd do if you were designing the interface for the agent.

Tool descriptions are trusted text the model reads. Each tool's name, description, and schema gets stamped into the system prompt every request. Ten focused tools beat fifty overlapping ones, and a sloppy or malicious tool description can prompt-inject your agent before you've typed anything.

Long-running execution

State must live outside the model's context window. The agent is amnesiac; the filesystem, the session log, and the memory store are not. Every long-running design is, at its core, a way of moving state out of context and back in on demand.

Three walls bound every long-running agent: finite context, no persistent state, no self-verification. Every architecture in this space is an answer to those three problems. If a design doesn't address all three, it isn't long-running — it's just a long single-session run.

Compaction alone is insufficient for very long jobs. Past a certain horizon you need full context resets — tearing down the session and rebuilding it from a structured handoff file. Closer to onboarding a new engineer than to "memory."

Bet on the patterns, not the products. Google, Anthropic, and Cursor have arrived at roughly the same shape from very different directions: Brain / Hands / Session, planner / worker / judge, durable session log, sprint contract, Ralph-loop continuation. Surface area differs across products; the underlying patterns converge.

Compounding systems and substrate

The five-layer compounding-system framework, the durability hierarchy, and the cross-domain principles for substrate vs. iterating layers live in core-principles.md and compounding-system.md. The principles below are the software-specific applications.

Frontier labs will keep producing better models and better harnesses. Both. The model gets better; the meta-skills router, the sandbox, the generic tool wrappers, the observability layer — they all get better too, on the labs' cadence. What labs won't produce is the captured intent of your work, the captured operating mind of your business, and the workflows that only matter inside your domain. Plan accordingly. Garry Tan compresses the same distinction into a slogan worth borrowing: fat skills, fat code, fat data, thin harness — invest where it compounds, rent the routing layer.

The substrate tells the agent three things: where to look, what shape the answer should take, and what counts as done. Where to look is the knowledge layer and the conventions. What shape is the spec. What counts as done is the test spec and the sprint contract. A substrate missing any of the three still produces wandering output, no matter how capable the model — perfect knowledge with no spec for shape produces directionless work; a perfect spec with no done-condition produces work that won't stop. (This is the principle that distinguishes substrate from documentation: documentation describes; substrate directs.)

Time-dependence

Sustainable software is software that can keep responding to change. Smaller batches are higher quality over time, frequent upgrades make later upgrades easier, and deprecation is part of the job because obsolete systems have carrying cost.

A good harness, good docs, good examples, and a sane platform turn model improvements into leverage. A weak system turns the same models into faster accumulation of drift.

A spec-driven, capability-based system has positive time-dependence. Three flywheels turn at once: specs sharpen each iteration; capabilities compound into a library that doesn't deprecate the way a normal codebase does; models improve on their own schedule. This is one of the few kinds of software system that gets better while you sleep.

The interface to the agent is the unit of stability. Because the system meets the agent at the spec, the test, the prompt, and the skill — and because all four are stable abstractions — model improvement is absorbed automatically. No rewrites, no migration projects. The same prompts produce better work.

The delivery loop

"Spec → test → prompt → execute" is a strong inner loop. The outer loop is review, integration, rollout, observation, and deprecation. The outer loop is equally or more important to success over time. The general loop-as-recovery and recursive-loop principles live in core-principles.md; the four-step inner loop named here is the software-specific instantiation.
