The Compounding System

The framework. operating-spec.md is one instance of it.

Source: compounding-system.md
Generated: HTML companion
Status: Mirrors Markdown

How to read this

This document defines the structure I use to build systems whose value grows with use. It is the framework — a description of the layers any compounding system has, what design rule each layer obeys, and how to tell whether each layer is healthy. operating-spec.md is one application of this framework: a compounding system for productized-agency software delivery. Future compounding systems (knowledge management, marketing, sales, operations) follow the same shape with different contents.

The framework is settled in its shape (five layers, the durability hierarchy, the one-shot regeneration test). The contents inside each layer are domain-specific and iterate with the work.

If you are an agent reading this, this is the structure of the system you operate inside. Read §1 for the shape, §2-§6 for the layer-by-layer design rules, and §7 for how the framework applies beyond software.

├── Intake layers          (where work originates — human, monitoring, feedback, ambient)

Each layer has a different durability profile, a different design rule, and a different relationship to "what survives across model generations." Investments in different layers are not interchangeable.

The defining property of a compounding system is that value grows with use rather than decays with neglect. Most software systems decay — code rots, dependencies drift, edge cases accumulate, the team gets steadily slower at making the same kind of change. Capability libraries, knowledge bases, and skill libraries compound. A compounding system is whatever has the second property — and the architecture below is the structure that produces it.

2. Substrate

The durable, repository-shaped component that survives across model generations.

The substrate is everything in the repo that captures work no frontier lab will ever do for you, arranged so that a more capable model and a better harness can do strictly better work off the same artifacts over time. Six categories, today:

  • Specs (descriptive, technical, test) — the precision of intent for each piece of work.
  • Capability inventory — the shared library of vertical slices, each portable by folder copy.
  • Knowledge layer — the captured operating mind of the business, queryable per question.
  • Domain-specific skills and tools — workflows that only matter inside the domain.
  • Repository conventions — folder layout, naming, frontmatter contracts, the loop-as-contract.
  • Decision records — the captured why behind paths taken and roads not taken.

The defining test for substrate (one-shot regeneration): A future model with a better harness, reading only the substrate, could rebuild a better working version of the system from scratch. Whatever fails this test is not substrate.

Design rule. Build only what frontier labs won't build for you. Edit substrate continuously; treat each edit as an investment that pays out across every future engagement.

Lifecycle. The substrate iterates — entries are edited, categories are added, the implementation beneath it is replaced. But the substrate is what survives iteration. As the labs ship better models and harnesses, the substrate stays; the iterating layers around it churn.

Healthy. Spec count grows. Capability inventory grows. Knowledge layer is reviewed on cadence. Decision records exist for every settled path. Conventions are explicit and validated.

Unhealthy. Substrate categories are sparse or empty. Specs drift from the code they describe. Knowledge entries go stale. Conventions exist as oral tradition rather than written contracts. Future agents have to re-litigate settled decisions.

See docs/education/definitions.md "The substrate entry, expanded" for the full taxonomy and open-category candidates.

3. Iterating layers

The components rented from frontier providers and replaced on their cadence.

Iterating layers are everything in the system that someone else (Anthropic, OpenAI, Google, Cursor, AWS, framework maintainers) will ship a better version of within a year. Useful while it's the best available; rented leverage rather than owned asset.

Common iterating layers:

  • The general harness / runtime — OpenClaw, Cursor, Claude Code, the meta-skill router.
  • Framework choices — Expo, Fastify, Postgres, Cognito, AWS. Replaceable by definition.
  • Standard tool wrappers — MCP servers for popular SaaS (Linear, Stripe, Notion). Someone will ship a better one.
  • Vendor SDK wrappers — Slack, Sentry, HubSpot integrations. Same logic.

Design rule. Use the best available; do not invest in building your own. The litmus test before building anything: would a frontier provider plausibly ship a better version within a year? If yes, lean toward not building it. If you do build it, name it explicitly as a short-term advantage with a known maintenance cost the day their version arrives.

Lifecycle. Replaced as soon as something better ships. The expectation is that this layer turns over completely every 18-36 months. Anything you build here is on a clock from day one.

Healthy. Thin. Whatever you're using is the current best-available version. Switching costs are low. The iterating layer is recognized as iterating, not mistaken for substrate.

Unhealthy. Heavy custom investment in the wrong layer. "We built our own version of X" when X is a frontier-provider roadmap item. Maintenance burden grows faster than value delivered. The iterating layer is treated as if it were substrate, and gets defended past its expiration date.

4. Operational shape

Who does what — which humans hold which decisions, which agents do which work, which queues route what to where.

The operational shape is the organizational structure of the compounding system. It names the human roles, the agent roles, and the routing surface that connects them. Distinct from substrate (the artifacts) and iterating layers (the runtime), the operational shape is who sits where in the loop.

Three components:

  • Human roles — who holds intent, who prioritizes, who accepts. The places where context, judgment, and accountability matter.
  • Agent roles — who plans, who executes, who reviews. The searchable-labor surface.
  • Routing — the queue, the board, the metadata schema that lets the system prioritize and attribute work.

Design rule. Small surface area for the human, large surface area for the agent. Put the human only where context, judgment, and accountability matter — defining intent, prioritizing work, accepting what ships. Everything else belongs to the agent. Anything more elaborate is overhead, and overhead is what stops compounding systems from scaling.

Lifecycle. The roles and routing evolve with the work and with the team. Generally more stable than iterating layers, less stable than substrate. Operational shape changes when the team scales, when the work mix shifts, or when a new intake source comes online.

Healthy. Each human decision is high-leverage. Routing is unambiguous — every piece of work has a clear owner and a clear next step. Agents do not idle waiting for human approval on low-stakes calls.

Unhealthy. Humans are bottlenecks on routine work. Routing is opaque. Multiple humans each "kind of" own the same decision. Agents can't tell what's pending and what's blocked.

For this repo's productized-agency context, the current operational shape is described in operating-spec.md §7.

5. Intake layers

Where work originates — the surface where the outside world becomes work the system processes.

The intake layer is everything that turns events into tasks. Without it, the system runs only on what humans manually request; with it, the system runs on a continuously refreshed task surface.

Four common sources:

  • Human inputs — direct requests from you or your team, conversations in Slack, feedback in standups.
  • Monitoring — Sentry-class signals, alerts, anomaly detection. Bugs and incidents that the system catches before users report them.
  • User feedback — in-app reports, feature requests, support tickets. Direct user-to-task pipelines.
  • Ambient ingestion — background pulls of client SaaS, calendar, transcripts, metrics. Garry Tan's 100 crons. Keeps the substrate (especially the knowledge layer) warm between agent runs.

All four ideally land on the same task surface — the board — with metadata that lets the rest of the system prioritize, route, and attribute. Source, severity, client, type, urgency, related entity. The metadata schema is substrate; the wrappers feeding into it are iterating layer.

Design rule. The intake layer should be additive — new sources land on the board with the right metadata, and the rest of the system picks them up without changes. If adding a new intake source requires rewriting routing or prioritization logic, the metadata schema is wrong.

Lifecycle. New intake sources arrive as the business grows (new clients, new monitoring tools, new user feedback channels). The metadata schema (substrate) is meant to be stable; the wrappers feeding into it (iterating layer) churn.

Healthy. All work, regardless of source, lands on one task surface with consistent metadata. The board is the single source of truth for what the system is doing. Adding a new intake source is a config change, not a rewrite.

Unhealthy. Multiple parallel queues that don't talk to each other. Work falls between sources. The board is incomplete. Some intake is silently lost or deprioritized because it didn't land in the right place.

6. The delivery loop

The contract that turns intake into shipped work and substrate updates.

The delivery loop is the protocol every piece of work follows. In this repo's vocabulary, the inner loop is spec → test → prompt → execute; the outer loop is review, integration, rollout, observation, and deprecation. The four steps of the inner loop are the contract; the structure inside them is open.

The loop is recursive: large goals contain smaller goals, each running its own spec/test/prompt/execute cycle. An agent is free to break a top-level goal into any number of inner loops, in any order, using whatever skills fit.

Design rule. When a step fails, go back to the smallest correction — usually the spec — update it, regenerate the downstream artifacts, re-run. Don't restart from scratch. The loop is the recovery mechanism, not just the build mechanism.

Lifecycle. The four-step contract is settled. The skills, tools, and workflows that run inside the contract iterate continuously. The outer loop (review → ship → observe → deprecate) is also stable in shape; the implementation evolves with the team and the iterating layers.

Healthy. Every piece of work passes through all four steps with evidence at each. Failures route to the smallest correction. The loop is recursive in practice, not just in theory.

Unhealthy. Steps get skipped under time pressure. Agents prompt-and-execute without specs or tests. Work is "done" without verification. Loop iterations re-litigate things that should have been settled in the spec.

For the principles that govern the loop, see sw-principles.md §The delivery loop and §Intent and specification.

7. Domain-agnostic application

The framework is not specific to software. The same five-layer shape applies to any domain where (a) durable artifacts can be captured, (b) the artifacts can be referenced by a person or agent, and (c) there's a loop that produces work and feeds back into the artifacts. If a domain can't be structured this way, it doesn't compound, and the framework doesn't apply.

Same shape every time. The substrate contents differ; the structure of a compounding system doesn't.

This is why the framework is durable: the domain-specific contents iterate, but the layered shape and the durability hierarchy stay constant. A marketing compounding system built on this framework today will, in two years, still have substrate / iterating / operational / intake / loop as its layers — even though every piece of content inside each layer will have turned over.

8. The cross-layer principles

A few rules of thumb apply to compounding systems regardless of which layer you're working in.

01
Intake creates work
Signals arrive through people, monitoring, users, and ambient ingestion.
02
The loop ships work
Each task moves through a repeatable contract instead of an improvised run.
03
Substrate absorbs learning
The next run starts from sharper specs, stronger context, and better examples.

Investments are not interchangeable across layers. An hour spent on substrate compounds across every future engagement. An hour spent on iterating-layer customization is rented leverage on someone else's roadmap. An hour spent on operational shape clarifies who decides what. An hour spent on intake schema unblocks every future intake source. Different layers, different returns. Budget time accordingly.

The substrate is the durability lever, not the value generator. The intake brings work in. The operational shape routes it. The loop processes it. The iterating layers execute it. The substrate is what makes today's good work still be good work in two years. Different jobs.

Healthy compounding systems get more valuable with use; unhealthy ones get more painful. Time is either compounding asset value or compounding maintenance liability; there is no neutral default. The five-layer architecture is one answer to "what does it look like when value compounds." A system whose substrate is sparse and whose iterating layers are heavy will compound liability instead.

The framework absorbs improvement. When a frontier provider ships a better harness, the iterating layer is replaced; the substrate is unchanged. When a new intake source comes online, the metadata schema (substrate) absorbs it; routing (operational shape) picks it up. The compounding system gets better with each external improvement, because the iterating layers are designed to be replaced and the substrate is designed to absorb.

What is settled vs. what is iterating

Settled (the framework itself):

  • The five layers (substrate / iterating / operational / intake / loop).
  • The durability hierarchy (substrate is what survives one-shot regeneration; iterating layers are rented; operational shape is structural; intake is the world-facing surface; the loop is the contract).
  • The cross-layer rules (investments are not interchangeable; substrate is the durability lever; healthy systems compound asset value; the framework absorbs improvement).
  • The domain-agnostic generalization (the same shape applies to software, marketing, sales, knowledge management, operations).

Iterating (the contents of any one compounding system):

  • Which categories exactly belong in the substrate. Today: six. Likely to grow.
  • Which iterating layers we currently rent. Today: OpenClaw, Expo, Fastify, Postgres, Cognito, AWS, vendor SDKs. Will turn over completely on a 18-36 month cadence.
  • The operational shape for the productized agency. Currently described in operating-spec.md §7.
  • The intake sources currently feeding the system. Today: human inputs and a planned client-side feedback panel. Will grow to include monitoring, ambient ingestion, etc.
  • The specific skills, tools, and conventions inside each layer.
