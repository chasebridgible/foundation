System Philosophy

Chapter one: the highest-level description of how intent becomes results through systems, context, actors, evidence, and revision.

Format: HTML-native
Authoring: edit this HTML file
Status: Draft working spec

How to read this

This document is the root description of what Foundation makes explicit: how intent becomes results through goals, capabilities, jobs, processes, actors, context, evidence, evaluation, and revision.

This philosophy comes from business work, where people turn intent into reality through offers, distribution, fulfillment, data, tools, and money. The same pattern applies to any goal a person has: name the result, define the steps, act with enough context, use evidence, and revise.

The core model is simple. A goal needs capabilities. Capabilities are produced by jobs. Jobs run through processes. Processes are performed by actors using context. Evidence is evaluated against intent, then the system updates. In business, the recurring goal is to create value, deliver value, and make money.

Capability

A capability is the system ability needed to produce a class of result reliably under real constraints.

Running example
Create value through software

The business needs the ability to turn product intent into software behavior that gives users a useful outcome.

Product behavior: the user can do the thing the offer promises.
Reliable result: the behavior works under real constraints.
Step 03
Job

A job is the named body of work required for a capability to exist, operate, or improve. A title can own many jobs; a job is performed through processes.

Build the experience Match product intent Verify behavior
Running example
Build the software experience

The job is to design and implement the user experience so it matches the intended product behavior and can be checked against it.

Input: product intent, context, constraints, and examples.
Output: a working part of the experience.
Step 04
Process

A process makes the job repeatable through actions, decisions, checks, and handoffs.

Clarify intent Design Build Test Review Revise
Running example
Run the build path

The process is the sequence someone follows to complete the job: clarify intent, design the interaction, build it, test it, review the rendered result, and revise.

Decision: which behavior belongs in this pass?
Check: does the rendered experience match the intent?
Step 05
Actors + Context

Actors are the people, agents, software, tools, vendors, or workflows that act. Context is what they need to act well.

Product owner Agent Software Specs Product context
Running example
Agent plus product context

An actor could be an agent, engineer, workflow, or software tool. Context is the intent, constraints, current code, product promise, standards, and evidence needed to act well.

Actor: the person, agent, or software performing the work.
Context: the information that lets the actor make the right move.
Step 06
Evidence + Evaluation

Evidence is the signal from reality. Evaluation is the judgment that compares the signal against intent.

Evidence Eval spec Test Screenshot Review
Running example
Evaluate the experience against intent

Evidence can be a test result, rendered screenshot, trace, review result, or user signal. Evaluation decides whether that evidence satisfies the intent.

Evidence: the UI renders and the behavior can be observed.
Evaluation: the eval spec decides pass, revise, continue, or stop.
Step 07
Revision

Revision updates the system after evidence comes back, so the next cycle starts from sharper context.

Update intent Update spec Update design Update code Update context
Running example
Revise the system

Revision updates the spec, design, process, context, or implementation so the next job starts from what the system learned.

If evidence passes: preserve the pattern and choose the next job.
If evidence misses: update the missing part before adding more effort.

Software development, the Business Operating System, context substrate, and Orderly Systems apply this root model to different domains of work.

This document separates durable principles from current operating choices. Principles are the parts I expect to survive changes in tools, models, teams, companies, and markets. Operating choices are Foundation's current best way to express those principles. The words will sharpen as the system runs. The structure is the thing being protected.

1. The root claim

Goals require intent-driven systems.

A goal becomes real when intent is clear enough to guide action, context is available to the actors doing the work, capabilities exist to produce the outcome, jobs are defined, processes make the jobs repeatable, actions and decisions touch reality, and evidence tells the system whether reality matched intent.

This is the core of the philosophy. Business is one durable mapping: a business is an intent-driven system for building value, delivering value, and capturing money.

The common failure mode is hidden machinery. Work happens behind closed doors, inside heads, inside tools, inside meetings, inside customer conversations, inside code, inside dashboards, and inside habits. A system improves when that machinery becomes visible.

The leverage move is to make the system legible: what goal is being pursued, which capability is missing, which job creates it, which process performs the job, who or what acts, what context they need, which decisions matter, and what evidence proves improvement.

Operating rule

When a business is not achieving its goal, decompose its system before adding effort. Find the missing capability, job, process, action, decision, context, actor, or evidence.

The visual carries the loop. The important distinction is timing: intent and context come before action; evidence comes after action; evaluation judges the evidence; revision updates the system so the next loop starts with better context.

Knowledge is stored understanding: facts, history, decisions, definitions, examples, data, principles, and notes. Context is knowledge prepared for action: the relevant subset of knowledge plus the current goal, constraints, state, actor, tools, risks, examples, decision criteria, and evidence needed for the current job.

Data is recorded fact or event history. Evidence is the part of reality selected to judge whether intent became result. Evaluation compares evidence against intent. Revision updates the system so the next action starts from what was learned.

Specs preserve intent, context, process, and evidence at the layer where work happens. The shared spec types are system, capability, job, technical, and eval. A job spec is the contract for a body of work and contains its process, SOP, checklist, or workflow by default. A technical spec defines software, data, automation, infrastructure, or integration implementation details. An eval spec defines how evidence will be judged. A test is one way to collect evidence for an eval; evaluation is the broader category.

This vocabulary is a diagnostic tool. When a result misses, it gives the operator a smaller layer to inspect: spec clarity, capability, job ownership, process clarity, action quality, actor fit, context, tool fit, evidence, evaluation, or revision.

Capability and job are different layers. A capability names what the system can reliably do. A job names the work required for that capability to exist, operate, or improve. For example, pay employees accurately and on time is a capability; run payroll for this pay period is a job; approving hours, calculating pay, resolving exceptions, and recording proof are process steps and actions inside the job.

Specs carry intent at every layer. Use a spec wherever a layer needs explicit intent, context, process, and evidence. System specs orient the whole system. Capability specs define reliable outcomes. Job specs define bodies of work and include the process, SOP, checklist, or workflow by default. Technical specs define implementation contracts when software, data, automation, infrastructure, or integrations are involved. Eval specs define how evidence is judged.

Interfaces are where the system is exposed or operated. In software, an interface can be a screen, route, command, schema, workflow file, dashboard, or document. In business, an interface can be an offer, sales call, intake form, fulfillment handoff, customer dashboard, invoice, SOP, metric report, or decision meeting. An interface shows where an actor can act or where evidence can be observed; the process then names the action, actor, tool, context, and eval spec for that place. Older backfill documents may use surface for this idea, but the broader system vocabulary is interface.

4. Business is a recurring mapping

The root system is goal-to-result work. Business is a recurring application of that system with a durable top-level goal: build value, deliver value, and capture money.

Because many businesses share that top-level goal, they also share a recurring capability shape. Most businesses need some version of: create an offer, reach buyers, sell, fulfill, support customers, hire, pay people, manage cash, comply with obligations, measure performance, and improve. A franchise makes this explicit because the same capability map repeats across locations, but the pattern exists in every operating business.

The working inventory for that recurring shape lives in Business Capability Map. It lists the capability families a business generally has to solve and the jobs that make each capability real.

When entering a business, classify each recurring function by asking how the system solves it today: what capability it supports, what job performs it, who or what acts, what process they follow, what tools they rely on, what context they need, and what evidence proves the result happened.

Paying employees is a useful example. At the system level, pay employees accurately and on time is a capability because it names a reliable outcome the business must be able to produce. A specific payroll run is a job. Approving hours, calculating pay, submitting payroll, resolving exceptions, and recording evidence are process steps and actions inside that job.

An entrepreneur improves the business system by improving the jobs and capabilities inside it. That can mean creating an offer, clarifying positioning, building distribution, selling, fulfilling, hiring, designing operations, managing cash, building software, deploying agents, creating context substrate, or learning the fundamentals required to do the next job better.

Language matters because precise vocabulary makes improvement smaller and visible. An entrepreneur improves the business system by describing it clearly enough to change one capability, job, process, action, actor, tool, context, evidence source, or evaluation at a time.

5. Software development as a visible system

Software development is useful in this philosophy because it makes the system visible. Intent becomes a spec. Context becomes repo state, principles, examples, and prior decisions. Jobs become named bodies of work such as designing the interface, implementing behavior, reviewing the change, testing the result, and preparing release. Processes become workflows or skills. Actors use tools to change the system. Tests, screenshots, traces, CI, and production behavior become evidence. Eval specs and reviews judge that evidence against intent.

The value of the work is the capability created afterward: a user can complete a task, an operation runs faster, an internal tool removes repeated work, a failure mode is handled, or the business can deliver value more reliably.

The general rule is the same everywhere: name the result, define the capability, choose the job, run the process, give the actor context and tools, collect evidence, and revise the system.

6. Capability is the unit of performance

A capability is the ability to reliably produce an outcome. In software delivery for Orderly Systems, a capability is often portable implementation inventory: schema, API, types, UI, and tests traveling together so the same capability can be delivered to another client. In business, a capability is broader: the organization's ability to create an offer, reach buyers, sell, fulfill, retain, hire, measure, or improve. The software definition is a domain-specific instance of the broader definition.

A business can build something valuable and still fail when it lacks the capability to distribute it, sell it, package it, price it, fulfill it repeatedly, or capture money from it. An entrepreneur can work hard and still fail when effort misses the missing capability. Effort is real, but capability is what changes the system's ability to win.

The primary business capability is broader than "create value." A more complete statement is: build value, deliver value, and capture money. That top-level capability decomposes into at least three subordinate capabilities:

  1. Value creation: identify a painful or valuable problem and create something that improves the customer's reality.
  2. Value delivery: get the value to the customer reliably, repeatedly, and at a quality level that fulfills the promise.
  3. Value capture: package, price, sell, collect, retain, and expand the exchange so the business makes money.

Distribution likely sits above sales. Distribution is the capability of making the offer reachable, visible, credible, and available to the right market. Sales is one job family inside that larger capability: turning qualified attention into a buying decision. Marketing, partnerships, channels, audience, content, outbound, referrals, product-led motion, and reputation can all be distribution mechanisms.

An offer is the promise that connects value to money. Offer principles therefore belong in the capability system: who the promise is for, what outcome it names, what pain it removes, what proof supports it, what risk reversal exists, what urgency or scarcity is real, how it is priced, and how fulfillment makes the promise true.

7. Processes are skills

A process is the repeatable path for doing a job. In software development, a skill is the same kind of artifact: a workflow an agent can follow when a trigger appears.

A useful skill names the job, the steps, the decisions, the tools, the context, the output, and the evidence required to stop. That is why a skill belongs in the process layer of the system.

Goal: working video-player experience
Capability: user can open, control, recover, and exit playback
Job: define and implement the open-player process
Process: inspect current player interfaces, define states, implement behavior, verify rendered UX
Skill: the reusable workflow that tells an agent how to run that process
Actions: read files, identify state model, edit component, run tests, capture screenshots
Evidence: tests pass, playback states render, gestures work, errors recover
Evaluation: compare that evidence against the eval spec and decide pass, revise, continue, or stop

Job specs, eval specs, technical specs, and skills work together. The job spec names intent, current handling, proposed handling, and process. The eval spec names how evidence will be judged. The technical spec names implementation contracts when software or automation is involved. The skill names the reusable workflow an agent can follow. Together they tell an actor how to do a job in pursuit of a capability.

8. Context and intake

Knowledge is stored understanding: facts, history, decisions, definitions, examples, data, principles, and notes.

Context is knowledge prepared for action. It is the relevant subset of knowledge, plus the current goal, constraints, state, actor, tools, risk, and evidence needed for a specific job. Knowledge becomes useful when the system can turn it into context for action.

A knowledge base is one implementation of context substrate. Its job is to preserve the operating mind of the business so future actors can act with the right goals, promises, constraints, standards, history, decisions, and examples in view.

Orderly Systems starts by asking questions because intake is how hidden work becomes visible. The intake process discovers the business system: what the owner is trying to accomplish, what the business promises, which capabilities matter, which jobs produce them, who does the jobs today, what process is followed, what context is missing, what decisions happen, what evidence is trusted, and where the system fails.

That discovery process should become operating substrate: capability maps, job specs, decision rules, metric definitions, offer context, actor assignments, evidence requirements, and update rules. Substrate is the durable memory layer inside the system. It is not the whole system: the system also includes actors, tools, intake, execution, evidence, evaluation, and revision. The exact artifact names should fit the business domain. The function is stable: make context reusable so the next human, agent, or system performs better.

A document entry is useful when it changes future action. A metric is useful when it changes a decision. An intake answer is useful when it updates a job, process, actor assignment, offer, or capability map. Otherwise the business has collected information without improving the system.

9. Learning updates the system

Learning is the revision function of an intent-driven system. Evidence and evaluation show where intent and reality diverge. Learning turns that judgment into clearer definitions, principles, specs, processes, capabilities, and context.

Intelligence is the speed at which a system learns. A smart system turns a miss into an update quickly: it sees the evidence, evaluates it, revises the right artifact, and changes the next action. A system that repeats the same mistake is stupid in the operational sense: evidence reached the system, but revision did not happen fast enough to change behavior.

A system has learned when future action changes. A note, conversation, or insight becomes useful when it updates an artifact another person, agent, or software system can use.

01
Observe
Use evidence to see where the current system missed the intended result.
02
Name
Clarify the principle, definition, boundary, job, action, or decision that became visible.
03
Update
Revise the spec, process, skill, capability, context, decision rule, or eval spec.

At this level, learning is system revision: reality teaches the system, and the system preserves that lesson by updating the artifacts that guide future work.

What is settled vs. what is iterating

Settled (the foundation of the model):

  • Goals become real through intent-driven systems.
  • Real systems produce outcomes through real actions and decisions.
  • The useful vocabulary is goal, spec, capability, job, process, action, actor, tool, context, evidence, evaluation, and revision.
  • Business is a recurring application of the root system: build value, deliver value, and capture money.
  • Software development makes the system visible through specs, context, capability-producing jobs, workflows, tools, evidence, evaluation, and revision.
  • The shared spec types are system, capability, job, technical, and eval.
  • A job spec is the contract for a body of work; its process lives inside the job spec unless a separate linked view would make the work easier to run.
  • A capability is the ability to reliably produce an outcome under real constraints.
  • A skill is the software-development form of a process or workflow.
  • Knowledge becomes useful when it becomes context for action.
  • Learning updates the artifacts that guide future work.
  • Interfaces show where a system is exposed or operated; process/action maps name the actions, actors, context, tools, and evidence connected to those interfaces.

Iterating (current best implementation):

  • The remaining migration path from older spec IDs and backfill layer names to system, capability, job, technical, and eval language.
  • The final vocabulary for distribution, sales, marketing, offer creation, value capture, and business performance capabilities.
  • The document structure for capability maps, offer artifacts, distribution artifacts, learning logs, and business-stack registries.
  • How much of this vocabulary should move into sw-definitions.html, Business Operating System specs, and future skills.
  • The first practical workflow that turns System Philosophy into daily execution and revision.

This document should be updated whenever the vocabulary becomes clearer, a new domain maps cleanly onto the model, or real execution proves that one of the layer boundaries needs revision.

Document roles and reading order

The core documents should have clear jobs. This document defines the root model and vocabulary. Domain documents should apply the model without redefining it. Principle documents should hold durable rules for doing work well. Spec system documents should define how durable specs are authored, linked, checked, and kept current.

Naming convention proposal: use plain noun names for root concepts, *-principles.html for durable rules, *-map.html for inventories, and the spec-system convention of {subject}.html, {subject}-technical.html, and {subject}-eval.html for spec families.

  1. Root model: system-philosophy.html, compounding-systems.html, and definitions/sw-definitions.html.
  2. Company and domain applications: orderly-systems.html, business-operating-system.html, and software-operating-system.html.
  3. Inventories and maps: name the domain first and the artifact second, as in business-capability-map.html.
  4. Principles: use {domain}-principles.html, with core-principles.html reserved for cross-domain rules.
  5. Specs: use system, capability, job, technical, and eval specs. Job specs are the canonical work-scope specs.

Recommended reading order: start with System Philosophy, then Orderly Systems, The Compounding System, Definitions, Business Capability Map, Business Operating System, Software Operating System, Core Principles, domain principle documents, and finally the Spec System documents.
