System Philosophy

Chapter one: the highest-level description of how intent becomes results through systems, context, actors, evidence, and revision.

Format: HTML-native
Authoring: edit this HTML file
Status: Draft working spec

How to read this

This document is the root description of the thing Foundation is trying to make explicit. It is not a software-development document first and a business document second. It is the system philosophy underneath goal-directed work itself.

The audience is founders. A founder is responsible for turning intent into reality through people, agents, software, tools, processes, offers, distribution, fulfillment, data, and money. A technical founder happens to meet that responsibility through software sooner than most, but the deeper problem is not coding. The deeper problem is learning how to see, describe, operate, and improve intent-driven systems.

Software development has already made some of this discipline visible through specs, tests, skills, CI, code review, and deployment. Business has the same underlying structure, but the vocabulary is less settled. This document names the shared structure first. The software operating spec, Business OS, context substrate, backfill process, and Orderly Systems offer are mappings of this root philosophy.

Two kinds of claim show up here. Principles are the parts I expect to survive changes in tools, models, teams, companies, and markets. Operating choices are the current best way to express those principles inside Foundation. The words will sharpen. The structure is the thing.

1. The root claim

Goals require intent-driven systems.

A goal does not become real because it is wanted. It becomes real when intent is clear enough to guide action, context is available to the actors doing the work, capabilities exist to produce the outcome, jobs are defined, processes make the jobs repeatable, actions and decisions touch reality, and evidence tells the system whether reality matched intent.

This is the core of the philosophy. Everything else is a mapping. Business is one durable mapping: a business is an intent-driven system for building value, delivering value, and capturing money. Software development is another mapping: it builds value through software, whether that value is internal leverage or an external product. The context substrate, spec system, Business OS, backfill process, and Orderly Systems offer are all expressions of the same root pattern.

The first failure mode is invisibility. Work happens behind closed doors, inside heads, inside tools, inside meetings, inside customer conversations, inside code, inside dashboards, and inside habits. The engineer does not see sales. The salesperson does not see design. The founder does not see every process clearly enough to improve it. The system still runs, but it runs as hidden machinery.

The leverage move is to make the machinery legible: what goal is being pursued, which capability is missing, which job creates it, which process performs the job, who or what acts, what context they need, which decisions matter, and what evidence proves improvement.

Operating rule

When a goal is not being achieved, do not first ask for more effort. Decompose the system until the missing capability, job, process, action, decision, context, actor, or evidence becomes visible.

2. Intent, context, action, evidence

The deep loop is simple:

Intent -> context -> execution -> evidence -> revision

Intent names what should be true. It can be a product behavior, customer outcome, business result, offer promise, process standard, or founder goal. Intent is not useful until it is clear enough that another actor can act on it.

Context is the operating information the actor needs to make the right move: goals, constraints, history, customer promises, domain facts, prior decisions, data, examples, definitions, principles, standards, and known failure modes. Everything is context-driven because actors act from what they can see.

Execution is where the system touches reality through actions and decisions. A person sends the email. An agent edits the code. Software routes the event. A founder chooses the offer. A customer buys or does not. Until an actor acts or decides, the system is still potential.

Evidence is reality answering back. Tests pass or fail. Revenue appears or does not. A customer gets the promised outcome or does not. A handoff succeeds or breaks. A dashboard changes. A reviewer accepts the work. Evidence without update is reporting. Evidence with update is learning.

In this system, a spec is any artifact that captures intent, context, and evidence well enough that a person, agent, software system, or team can act against it. The word may or may not appear in business-facing language. A system can have a spec. A capability can have a spec. A job can have a spec. A process can have a spec. An action or decision can have a spec when the stakes justify it.

A job can also have a test spec. In business language that may be called an eval, acceptance criteria, a quality gate, an audit, a scorecard, or a metric definition. The name should fit the domain. The function is the same: define how the system will know whether intent became reality.

The useful vocabulary is a stack. Each layer exists because the layer above it needs a reliable way to happen.

The stack is not a bureaucracy. It is a diagnostic tool. If a goal is not being reached, the system gives you somewhere smaller to look. The missing piece might be capability, job ownership, process clarity, action quality, decision criteria, actor fit, context, tooling, or evidence.

Spec is not a single layer. Spec is the intent-and-evidence artifact that can attach to any layer where clarity matters. A system spec, capability spec, job spec, process spec, action spec, and test spec all do the same deeper work at different scopes.

Surface is not action. A surface is where a behavior is exposed or observed: a screen, route, command, workflow file, schema, document, dashboard, or physical station. A surface tells you where action can happen. The action itself is named later, inside the process or Process / Action Map. Calling surface identification "action identification" is close, but it skips one layer: first name the place where behavior exists; then name what actors do there.

4. Business is a recurring mapping

The right split is not business versus software. The root system is goal-to-result work. Business is a recurring application of that system with a durable top-level goal: build value, deliver value, and capture money.

A founder's job is to improve the business system. That can mean creating an offer, clarifying positioning, building distribution, selling, fulfilling, hiring, designing operations, managing cash, building software, deploying agents, creating context substrate, or learning the fundamentals required to do the next job better. These are not separate philosophical categories. They are jobs and capabilities inside one intent-driven system.

That is why the same vocabulary should apply across Orderly Systems, software development, context-substrate work, and business operations. The business has goals. Goals require capabilities. Capabilities are made real by jobs. Jobs are performed by processes. Processes require actors, actions, decisions, context, and evidence. The business learns when evidence updates the system.

Business system
Intent becomes value and money
Goals, capabilities, jobs, processes, actors, context, and evidence turn founder intent into customer outcomes and economic results.
Software development
One value-building mapping
Specs, tests, skills, agents, tools, reviews, and deployment are the software-development mapping of the same intent/context/evidence/revision loop.

This is why language matters. A founder cannot improve a system they cannot describe. The more precise the vocabulary, the smaller the unit of improvement becomes.

5. Software development is a mapping

Software development is valuable because it builds value through software. Sometimes the value is external: a product behavior a customer pays for. Sometimes it is internal: leverage, automation, reliability, speed, data, or operating clarity. Either way, software is built because someone expects the system to produce a better result afterward.

The software-development system makes the general philosophy executable. A software spec says what is intended and how the system will know whether the result works. A test gives executable evidence. A skill gives a repeatable process. A repo gives context. A codebase exposes surfaces. A PR packages the work for review. CI supplies automated evidence. A deployment turns candidate work into operating reality. A backfill process turns existing code into legible substrate.

Those ideas should not stay trapped in software because they were never software-only ideas. They teach any intent-driven system how to operate: write intent clearly, give actors the right context, define the job, run the process, verify with evidence, and update the substrate.

Business goal: build value, deliver value, and capture money
Software-development capability: build value through software
Job: turn intent into shipped software value
Process: spec -> test -> prompt -> execute -> review -> deploy -> learn
Actors: founder, agent, subagent, tests, browser, CI, reviewer, users
Context: principles, definitions, specs, code, context substrate, prior decisions
Evidence: tests, screenshots, traces, diffs, customer behavior, production signals
Learning: update specs, skills, capability inventory, definitions, and decisions

The practical result is that specs are agnostic. The file may be called job-spec.html, process-spec.html, offer-map.html, or something else. The name should be clear enough for the people using it. The function is stable: preserve intent and context well enough that a capable actor can act, then name the evidence that decides whether the action worked.

6. Capability is the unit of performance

A capability is the ability to reliably produce an outcome. In software delivery, a capability is often a portable vertical slice: schema, API, types, UI, and tests traveling together. In business, a capability is broader: the organization's ability to create an offer, reach buyers, sell, fulfill, retain, hire, measure, or improve. The software definition is not wrong. It is a domain-specific instance of the broader definition.

A business can build something valuable and still fail because it lacks the capability to distribute it, sell it, package it, price it, fulfill it repeatedly, or capture money from it. A founder can work hard and still fail because the work is not creating the missing capability. Effort is real, but capability is what changes the system's ability to win.

The primary business capability is not just "create value." A more complete statement is: build value, deliver value, and capture money. That top-level capability decomposes into at least three subordinate capabilities:

  1. Value creation: identify a painful or valuable problem and create something that improves the customer's reality.
  2. Value delivery: get the value to the customer reliably, repeatedly, and at a quality level that fulfills the promise.
  3. Value capture: package, price, sell, collect, retain, and expand the exchange so the business makes money.

Distribution likely sits above sales. Distribution is the capability of making the offer reachable, visible, credible, and available to the right market. Sales is one job family inside that larger capability: turning qualified attention into a buying decision. Marketing, partnerships, channels, audience, content, outbound, referrals, product-led motion, and reputation can all be distribution mechanisms.

An offer is the promise that connects value to money. Offer principles therefore belong in the capability system: who the promise is for, what outcome it names, what pain it removes, what proof supports it, what risk reversal exists, what urgency or scarcity is real, how it is priced, and how fulfillment makes the promise true.

7. Processes are skills

In agent development, a skill is a workflow the agent can follow when a trigger appears. In business, a process or SOP is the same shape. In human work, a checklist is the same shape. Different names, same structure: when this situation appears, follow these steps, make these decisions, use these tools, produce this output, and verify it this way.

This reframes agent instruction. A prompt is not just a request. It is the current way the human gives the actor a job, the process for doing it, the context needed to act, and the evidence required to stop. A weak prompt asks for an outcome while hiding the job. A strong skill names the job and gives the actor enough process to perform it.

The old failure mode was describing the desired app behavior in the language available at the time, then watching the agent take invisible engineering actions that were not understood, constrained, or verified. The way forward is not to become every kind of engineer at implementation depth. The way forward is to understand the job structure at the level needed to give better intent: what decisions must be made, which actions must happen, what risks matter, what evidence proves the work, and which pieces belong in reusable substrate.

Goal: working video-player experience
Capability: user can open, control, recover, and exit playback
Job: define and implement the open-player process
Process / skill: inspect current player surfaces, define states, implement behavior, verify rendered UX
Actions: read files, identify state model, edit component, run tests, capture screenshots
Evidence: tests pass, playback states render, gestures work, errors recover, spec matches behavior

That is why specs, tests, and skills are tightly connected at the systems level. They are not identical artifacts, but they form one process substrate. The spec names intent. The test or eval names evidence. The skill names the workflow. Together they tell an actor how to do a job in pursuit of a capability.

8. Context and intake

Context is higher than knowledge. Knowledge is what the system knows. Context is what an actor needs in order to act well: goals, constraints, promises, history, data, examples, taste, standards, permissions, definitions, tools, current state, and known failure modes.

A knowledge base is one implementation of context substrate. Its job is to preserve the operating mind of the business so future actors can act with the right goals, promises, constraints, standards, history, decisions, and examples in view.

Orderly Systems starts by asking questions because intake is how hidden work becomes visible. The intake process discovers the business system: what the owner is trying to accomplish, what the business promises, which capabilities matter, which jobs produce them, who does the jobs today, what process is followed, what context is missing, what decisions happen, what evidence is trusted, and where the system fails.

That discovery process should not end as a pile of notes. It should become operating substrate: capability maps, job records, process maps, decision rules, metric definitions, offer context, actor assignments, evidence requirements, and update rules. The exact artifact names should fit the business domain. The function is stable: make context reusable so the next human, agent, or system performs better.

A document entry is useful when it changes future action. A metric is useful when it changes a decision. An intake answer is useful when it updates a job, process, actor assignment, offer, or capability map. Otherwise the business has collected information without improving the system.

9. Learning is a job

There is a personal version of the same system. To build better software systems, business systems, and agent systems, the operator has to learn the fundamentals of each domain well enough to connect them. That learning is not background activity. It is a job in the system.

The job is not "read more." The job is: identify missing vocabulary, learn the principles, observe real work, decompose jobs, write definitions, create process artifacts, test them in execution, and revise the substrate when reality teaches something. Learning compounds only when it updates future action.

This matters because the limiting factor was not grit, effort, suffering, pain tolerance, or perseverance. Those can keep a system running, but they do not make the system correct. The missing piece was fundamentals: knowing what is real, what matters, what work actually has to happen, and what evidence would prove it.

There may not always be one objectively correct choice. But real constraints, real actions, real outcomes, and real consequences exist. A system succeeds or fails in the world, not in the story told about it. The operator's job is to keep improving the match between thought, language, action, and reality.

01

Find the missing word, boundary, principle, or job that is making the system hard to see.

Documenting the process is therefore part of the process. The system learns when the operator's improved understanding becomes an artifact another human or agent can use next time.

10. Backfill translated

The Foundation backfill process already follows this vocabulary, even where the names are still software-specific. The building analogy is useful because it makes invisible structure visible, but the durable output should be clear operating vocabulary.

The naming rule is simple: use verbs for process steps and nouns for artifacts. Inventory before interpretation, surfaces before actions, capabilities before jobs, context before execution, and verification before acceptance. Current executable backfill specs, file paths, and scripts can keep legacy implementation keys for compatibility; durable prose, skill routing, and user-facing workflow language should use the canonical process and artifact names.

11. Principles

Goals require intent-driven systems. A meaningful goal needs intent, context, capabilities, jobs, processes, actors, actions, decisions, evidence, and revision. Otherwise it is a wish with effort attached.

Name things as clearly as possible. Name a thing as if you were explaining the real work to a capable person with no hidden context. Prefer words that expose the job, actor, action, decision, evidence, and outcome. If a sentence sounds impressive but cannot change what someone does next, it is not yet operating language.

Use analogies to reveal structure, then translate them back into operating language. Analogies help people understand invisible systems. "Walk every room" makes file inventory intuitive. But the durable artifact should eventually use precise names the system can execute against.

Specs live wherever intent lives. A system, capability, job, process, action, decision, and test can all have specs. The artifact name can change by domain, but the function stays the same: preserve intent, context, and evidence so actors can execute and improve.

Context is higher than knowledge. Knowledge is what the system knows. Context is what the actor needs to act well. A knowledge base is useful when it becomes context substrate for future decisions and actions.

Define the job before choosing the actor. A human, agent, software system, vendor, or hybrid can only be chosen intelligently after the work is understood. Choosing the actor first turns delegation into guessing.

Actions happen in reality, not in abstractions. Every process eventually becomes someone or something doing something: opening a file, making a call, changing a value, approving a decision, sending a message, shipping a product, collecting money, handling an exception. If the action cannot be named, the process is not fully understood.

Capabilities are built from jobs. A capability is not a slogan. It is the system's repeatable ability to produce an outcome because the underlying jobs, processes, actors, tools, and evidence are strong enough.

Titles are not jobs. CEO, founder, engineer, designer, operator, and salesperson are titles or roles. Each contains jobs. Performance improves when the role is decomposed into the jobs and capabilities it is accountable for.

Software is a job-solving tool. A tech stack is not the business system. It is one layer of tools that perform or support jobs. The business stack is broader: offers, capabilities, jobs, actors, tools, data, processes, distribution, money, and learning.

Humans and agents both stop short unless the process closes the loop. A capable actor can still be lazy, overloaded, under-contexted, or satisfied too early. The process must force evidence, review, and revision instead of relying on effort or confidence.

What can be measured against intent can be improved. Measurement is useful when it is tied to the goal, the job, and the decision it should change. A metric detached from intent becomes decoration or distortion.

The context layer has jobs. It preserves operating context, makes work retrievable, reduces rediscovery, improves recommendations, routes decisions, and helps the owner see what the system knows and how it runs. If it does not change future work, it is storage rather than operating memory.

Learning must update substrate. Reading, thinking, and discussion are inputs. The compounding output is a sharper definition, principle, skill, spec, job record, offer, checklist, metric, decision record, or capability.

Verification closes every layer. An action needs evidence. A process needs an exit criterion. A job needs acceptance. A capability needs performance signal. A business needs money and value delivered. A learning system needs changed future behavior.

What is settled vs. what is iterating

Settled (the foundation of the model):

  • Goals require intent-driven systems; they do not become real through desire alone.
  • Real systems produce outcomes through real actions and decisions.
  • The useful decomposition is goal, capability, job, process/skill, action/decision, actor/tool, context, and evidence.
  • Business is a recurring mapping of the root system: build value, deliver value, and capture money.
  • Software development is a value-building mapping of the root system; it builds internal or external value through software.
  • Specs are layer-agnostic intent-and-evidence artifacts; a system, capability, job, process, action, decision, or test can have a spec.
  • A capability is the ability to reliably produce an outcome; a software vertical slice is one domain-specific form of capability.
  • A process, workflow, SOP, checklist, and agent skill are the same kind of thing at the systems level.
  • Context is higher than knowledge and is the condition for correct execution; actors perform better when the system gives them better context.
  • Humans and agents are both actors that need enough context, process, evidence, and revision pressure to avoid stopping short.
  • Document and artifact mapping applies beyond codebases; business document sets can be inventoried, surfaced, grouped into capabilities, and turned into context packs and specs.
  • Learning compounds only when it updates future action through substrate.
  • Surface identification is not action identification; surfaces expose behavior, while process/action maps name actions.

Iterating (current best implementation):

  • The exact names for stack-layer artifacts, especially where to use spec directly versus a business-facing synonym.
  • The final vocabulary for distribution, sales, marketing, offer creation, value capture, and business performance capabilities.
  • The document structure for job specs, process specs, capability maps, offer artifacts, distribution artifacts, learning logs, and business-stack registries.
  • How much of this vocabulary should move into sw-definitions.html, Business OS specs, and future skills.
  • Which executable backfill step names should migrate now, and which should keep their implementation names until a deliberate compatibility pass.
  • The first practical workflow that turns System Philosophy into daily learning and daily execution.

This document should be updated whenever the vocabulary becomes clearer, a new domain maps cleanly onto the stack, or real execution proves that one of the layer boundaries is wrong.
