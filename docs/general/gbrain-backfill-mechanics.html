GBrain Mechanics for Foundation Backfill

This note extracts reusable mechanics from Garry Tan's GBrain for the Foundation backfill and development substrate. It is not a formal spec. It is a working analysis of what should guide the Artifact Inventory, Surface / Function Map, Capability Map, queue, Context Pack, and evaluation layers.

Status: analysis draft
Created: 2026-05-25
Source reviewed: local clone of garrytan/gbrain

Core Read

GBrain should be treated as a compounding system, but not because it is a magic memory product. It compounds because new inputs can update durable knowledge, derived indexes, graph links, skills, eval baselines, and operational procedures. Model improvement then makes the latent parts better while the deterministic substrate keeps the system stable.

For Foundation, the important lesson is not "install GBrain." The lesson is that a good knowledge layer separates canonical durable artifacts from derived retrieval structures, routes every read and write through explicit boundaries, and makes improvement measurable with replayable gates.

The shortest translation: GBrain is useful as a reference implementation for how a knowledge substrate can evolve. Foundation should copy the mechanics that make evolution safe, not the whole product surface.

Reusable Mechanics

Mechanic 01
Canonical files, derived indexes
GBrain's markdown repo is the system of record; the DB is a derived cache. Foundation should make backfill JSONL/HTML artifacts canonical and treat any DB/search/vector index as rebuildable.
Mechanic 02
Explicit source boundaries
GBrain's brain/source axes prevent silent context mixing. Foundation needs analogous repo/run/source/slice boundaries so a backfill row can always cite where it came from and where it is allowed to write.
Mechanic 03
Operation contracts
GBrain routes CLI, MCP, HTTP, and subagent calls through shared operations. Foundation backfill should expose each layer as a small set of deterministic operations plus skills, not as one long prompt.
Mechanic 04
Resolvers over giant manuals
GBrain uses resolver tables to load the right skill at the right time. Foundation should keep AGENTS files small and route layer work through resolver/skill entries keyed by artifact and failure mode.
Mechanic 05
Deterministic where possible
GBrain uses zero-LLM link extraction, check scripts, schema validation, and source scoping for trust. Foundation should use latent judgment only after deterministic inventory, parsing, hashing, and coverage checks are exhausted.
Mechanic 06
Capture and replay
GBrain captures retrieval candidates and replays against baselines. Foundation should capture backfill decisions and replay/evaluate them when skills, rubrics, or models change.
Mechanic 07
Trust boundary as data
GBrain distinguishes trusted local callers from remote/untrusted callers. Foundation should distinguish deterministic checker output, agent draft output, human-reviewed output, and promoted canonical state.
Mechanic 08
Queues with resumable state
GBrain's Minions queue is durable, idempotent, bounded, and audited. Foundation's Job / Spec Queue layer should become a real state model, not just prose inside the report.

Layer Implications

Artifact Inventory

The Artifact Inventory should follow GBrain's source-of-record pattern. The JSONL registry is canonical. Hashes, sizes, language detection, and generated-file classification are deterministic facts. Any search index, graph, or UI is derived and must be rebuildable from the registry plus repo files.

The inventory rows should probably carry more than the proposed V1 fields. Add registry_version, repo_id, run_id, path_kind, language, is_generated, source_status, mapping_status, responsibilities, surface_refs, review_flags, confidence, mapped_by, mapped_at, and evidence. This makes later layers less dependent on re-reading raw files.

Surface / Function Map

GBrain's retrieval system shows the value of layering cheap deterministic extractors before model judgment. The Surface / Function Map should first use deterministic detectors for routes, commands, screens, tables, workflows, tests, docs, generated artifacts, jobs, and infra resources. Latent extraction should fill gaps, classify intent, and flag ambiguity.

Every surface should cite the Artifact Inventory rows that produced it. That creates a chain from file -> surface -> capability -> spec -> test. Without that chain, the backfill system will know it read files but will not know which files prove which user-facing behavior.

Capability Map

GBrain's graph and source-aware citations are more relevant than its embeddings here. A capability is a graph node with evidence edges, not just a paragraph. The row should carry actor, outcome, domain object, actions, states, rules, experience, backing contracts, failure/recovery, and evidence links back to surfaces and files.

The Capability Map should also include negative evidence: files inspected that looked relevant but did not prove the capability. This helps future agents avoid rediscovering the same dead ends.

Define Spec Jobs

This is where GBrain's job queue is most useful. Broad capability rows let agents escape. A Job / Spec Queue should include slice id, parent capability id, acceptance boundary, required artifact outputs, Context Pack id, status, attempts, blocked reason, reviewer state, budget, and last checker/eval result.

Use deterministic split checks before semantic review: row too broad, too many actors, too many unrelated surfaces, no backing contracts, no failure modes, no evidence, or no observable done condition.

Context Pack

GBrain's retrieval modes and token budgets suggest the Context Pack should be a scoped retrieval product, not a dump. It should gather the smallest sufficient set of file rows, surface rows, code excerpts, docs, tests, and prior decisions for one slice. It should include why each item was included and what question it answers.

Make Context Packs reproducible. A future agent should be able to re-run pack generation from the queue row and get a comparable result, then inspect differences.

Process / Action Map

GBrain's diarization pattern matters here: read many documents, produce one structured judgment artifact. Process / Action Map should read the Context Pack and produce states, rules, permissions, edge cases, empty/loading/error behavior, and visible/operator behavior. The output should be structured enough to evaluate and specific enough to author descriptive specs.

Author Specs

GBrain's thin-harness/fat-skills split says Author Specs should live mostly in skills, while the harness only manages artifacts and gates. Keep the descriptive spec skill architecture-agnostic. Keep the technical spec skill contract-first. Use deterministic validators to enforce metadata, owned paths, related specs, evidence links, and coverage.

Evaluate Job Slices And Revision

GBrain's eval gate is a better pattern than one-time judging. Foundation should require create -> evaluate -> revise -> acceptable. The evaluator should write structured receipts that include scores, findings, sampled evidence, model/rubric versions, and what changed after revision.

Evaluate System Coherence

GBrain's rebuild promise is the hard version of the final question. Foundation's graph evaluation should ask whether the spec graph can rebuild the system's intent without source access. The practical test is: hide the source, give a future agent the specs and evidence summaries, and see whether it can describe the system, generate tests, and identify implementation boundaries.

Implementation Guidance

Foundation need GBrain pattern Suggested implementation
Canonical state Markdown repo is source of truth; DB is derived cache. Make JSONL/HTML artifacts canonical. If a SQLite/vector/graph index is added later, it must be rebuildable from artifacts and repo files.
Routing Brain/source axes with explicit citations. Add repo/run/source/slice identifiers to every row. Require downstream artifacts to cite upstream row ids.
Layer execution Shared operation contracts serve CLI/MCP/HTTP. Define operations like file_registry_init, surface_extract, capability_split, and evidence_pack_build. Wrap them with skills rather than embedding logic only in prompts.
Quality gates Check scripts enforce invariants before semantic review. For each layer, ship a deterministic checker before an LLM evaluator. The checker proves structure/currentness; the evaluator judges meaning.
Semantic regression Capture, publish baseline, replay, gate. Capture mapped rows and evaluator receipts. Replay representative slices when models, prompts, or rubrics change.
Cost control Search modes and explicit user confirmation for expensive defaults. Every layer that can spend model tokens should have mode, budget, batch size, and stop conditions recorded in the run log.
Agent context Resolver loads the right skill; large manuals are compressed to pointers. Use a backfill resolver keyed by layer and failure mode. Keep the active report concise and move state into machine-checkable artifacts.
Durable work Minions queue has bounded, resumable job state. Move Job / Spec Queue out of prose and into Job / Spec Queue rows. Treat attempts, blocks, and approvals as first-class state.

What Not To Copy

  • Do not make a DB canonical by accident. GBrain is careful that the DB is a cache. Foundation should keep that discipline if it adds indexes, embeddings, or a local service.
  • Do not build the orchestration layer first. GBrain's job queue is useful because it sits on top of clear operations and state. Foundation should finish file/surface/capability/checker contracts before adding heavy orchestration.
  • Do not let the active report become the database. The report should summarize state. Registries, queues, run logs, and eval receipts should carry state.
  • Do not copy GBrain's complexity wholesale. Brain/source routing, OAuth, MCP, minions, schema packs, and hybrid retrieval solve GBrain's product constraints. Foundation needs the mechanics, not all surfaces.
  • Do not let giant memory files become the resolver. GBrain's own ethos warns against huge CLAUDE-style manuals. Foundation should keep context load progressive and route to narrow skills/specs.

Knowledge Base Extension

The same mechanics apply to Business Operating System knowledge bases. Canonical business artifacts should be files or structured records: offer map, job records, role records, SOPs, metric dictionary, decision log, incidents, customer promises, and update rules. Search, graph traversal, dashboards, and recommendations should be derived from those artifacts.

GBrain's source boundaries map naturally to clients, departments, functions, or operating domains. Foundation's job records can become the business equivalent of capabilities. Context Packs can gather the facts needed to improve one job. Eval/replay can test whether recommendations still improve causal drivers rather than vanity metrics.

Business Operating System translation: a company compounds when measured reality updates durable operating memory. GBrain shows one way to make the memory retrievable. Foundation still needs the operating protocol that turns retrieval into better definitions, decisions, actions, and learning.

Recommended Order

  1. Finalize the Artifact Inventory artifact and checker as source-of-record infrastructure.
  2. Add artifact category rules: canonical, derived, and runtime-only exceptions.
  3. Build the Surface / Function Map with deterministic detectors first, LLM fill second.
  4. Make row ids and evidence edges mandatory across file, surface, capability, queue, Context Pack, and spec artifacts.
  5. Add semantic eval receipts and replay before the Capability Map becomes too large.
  6. Move Job / Spec Queue into durable JSONL state with explicit acceptance boundaries.
  7. Add Context Pack generation with token budgets and reproducibility receipts.
  8. Only then consider a GBrain-like retrieval/index layer for faster context assembly.

Bottom Line

GBrain supports the direction of the Foundation backfill system. It suggests that the right V1 is not a smarter one-shot backfill prompt. The right V1 is a stack of durable, source-cited, machine-checkable artifacts with deterministic gates, semantic eval receipts, and replayable improvement loops.

The Artifact Inventory is the bottom of that stack. It proves the repo was seen. The Surface / Function Map, Capability Map, queue, Context Pack, specs, and graph evaluation prove the system was understood. GBrain's most useful contribution is showing how to keep those layers canonical, scoped, routed, indexed, and improvable over time.
