Building Software with Agents:
A Working Spec

For me, and for the agents I work with.

How to read this

This document is itself a spec. It describes how I want to build software with agents, and how the agents I work with should operate inside the system. It is written in the form it teaches: short, declarative, principle-first, kept current as I learn.

Two kinds of claim show up in here. Principles I am confident in — I would defend them in conversation and do not expect them to change as my technical understanding improves. Operating choices are my current best implementation of those principles. Those will change. When they do, this doc is updated in the same commit as the system that changed.

If you are an agent reading this, this is your environment. Read §7 (the operating shape) for what to do, and read everything else for why.

1. Bugs are deviations from intent.

Code executes. Agents execute. Wrongness in software is almost never "the machine made a mistake" — it is that the intent the machine was acting on was incomplete, ambiguous, or never written down. A bug is a gap between what was supposed to happen and what was specified to happen.

That is not a semantic move. It changes where you look when something goes wrong. The first question is not what is the agent doing weird? It is what did I tell it to do, and did that description capture what I wanted? Almost always, the answer is no — the deviation is the agent faithfully executing an under-specified intent.

The implication for how to work with agents is direct. The unit of correctness is the spec, not the code. If you can describe what should happen with enough precision that a test could verify it, the rest of the work compresses. If you cannot, no amount of prompting or model upgrade will reliably get you there.

This claim has a corollary worth stating out loud. As models improve, raw technical capability gets cheaper. What does not is articulated intent. The bottleneck in software development is moving from can the machine build this to can the human describe what should be built well enough that the machine knows when it is done. The leverage move is to get good at articulating intent.

2. The loop: spec, test, prompt, execute.

Every piece of agentic work runs the same four-step cycle.

01
Spec
State the intended outcome clearly enough that a test could verify it.
02
Test
Turn the intent into code-level checks and a human-readable test plan.
03
Prompt
Load the spec, tests, rules, skills, and relevant context into the run.
04
Execute
The agent builds against the contract; evidence decides whether it is done.
  1. Spec. Describe the intended user experience and outcome with enough specificity that a test could verify it.
  2. Test. Derive tests from the spec — both code-level (unit, integration, smoke) and an English-readable test plan a human can review for intent.
  3. Prompt. Compose a prompt that references the spec, the tests, the rules of the system, the available skills, and the relevant context.
  4. Execute. The agent runs against the prompt, produces work, and the tests verify whether the work matches intent.

When a step fails, you do not restart from scratch. You go back to whichever step is the smallest correction — usually the spec — update it, regenerate tests, re-prompt, re-execute. The loop is the thing.

What this loop replaces is the older mode I used to work in: prompt and execute, repeatedly, with a vague sketch instead of a spec and no tests at all. That mode burns context, generates work that drifts further from intent each round, and produces software whose correctness no one can verify. The lesson from doing it that way for too long is that the spec and the test are not optional preliminaries. They are the work; the prompt and execute steps are the cheap part.

The loop is also recursive. A large goal contains smaller goals, each running its own spec/test/prompt/execute cycle. The agent is free to break a top-level goal into any number of inner loops, in any order, using whatever skills fit. The four steps are the contract; the structure inside them is open.

3. Specs evolve. So do the tests they generate.

A spec is a living product contract. It says: this is how the system should look and behave from the user's side, and these are the verifiable conditions that protect that behavior. Specs are never finished. They are the most up-to-date statement of what the work is trying to achieve.

There are three kinds, and they answer different questions.

Descriptive specs describe how the application should look and feel to a user — the full user flow, conditional behavior, edge cases. This is where the experience the user actually has gets articulated, and where vagueness gets discovered. A descriptive spec that says "the popup should appear when the user saves a clip" is too thin. One that says "a centered modal slides up over 200ms with a confetti burst, a haptic tap on iOS and Android, the title 'Saved' in 18pt semibold, the subtitle 'Your highlight is in your camera roll' in 14pt regular, dismissed on tap-anywhere or after 1.5 seconds" is a spec.

Technical specs describe how the system functions internally — what should happen, in what sequence, with what timing. They are required for every software task. The level of detail scales with the work: a simple message-thread change may need a short technical note that names touched modules, data flow, and non-goals; a livestreaming pipeline, custom video player, or real-time sync engine needs a deeper contract for sequencing, timing, failure modes, and boundaries. The rule is: the technical spec is never skippable, but it should be only as deep as the task requires.

Test specs describe in plain English how the system will be verified. They are the human's seat at the table. The code-level tests can be regenerated by the agent at any time, but the test spec is what the human reads to decide whether the right things are being checked. When the descriptive spec changes, the test spec changes; when the test spec changes, the underlying code-level tests are regenerated.

The discipline that makes all of this work is straightforward. When the code changes, the relevant Specs must also change. A spec that drifts from the system it describes is worse than no spec, because it gives false confidence. Spec/code consistency is part of every loop iteration, not an afterthought.

4. Capabilities — vertical slices that compound.

Most software gets harder to change as it ages. The unit of change keeps growing — a new feature touches the database, the API, the contracts, the frontend, three shared utilities, and the dev environment, and any of those can break the others. The codebase becomes the unit of change, and the codebase is the whole system.

The fix is to make the unit of change much smaller and much more self-contained. A capability is a single product feature that owns every layer it touches: its database tables, its API routes, its shared types, its frontend components. All in one folder. The capability moves between client repos by folder copy. If it does not move cleanly that way — schema applied, routes mounted, types available, screens wired up, env vars declared — the capability is not done.

Folder copy is the portability test because it is a physical constraint, not a design rule. The capability either drops into a new repo and works, or it does not. There is nothing to debate.

This is what makes the system compound. A normal codebase grows by accretion: features tangle, the whole thing becomes one big thing. A capability library grows by inventory. Every feature built for one client becomes a reusable asset across every layer at once — a new screen and a new route and a new schema slice, traveling together. The next client either uses the capability as-is, extends it, or contributes a new one back.

The discipline that makes this real is the corollary of the principle. A capability is not done until it is substrate inventory. Every finished client feature is rewritten — by the agent, as part of the same goal — into a portable capability in the shared library. Skip that step and the inventory does not grow, and the system goes back to being a normal codebase that gets harder over time.

5. The knowledge base — durable context.

A context window is finite. Whatever an agent does not know about your business at the moment of execution, it makes up. Most of the time the made-up answer is plausible enough that you do not notice — until you do, and the cost is whatever piece of work was built on top of the bad assumption.

Context is the moat in agent-based development. The defensible thing is not the agent or the model; both are commodities. The defensible thing is the captured operating mind of the business — what it sells, who it sells to, what it promises, what it does not, who decides what, what the SOPs are, what the constraints are. That is the knowledge base.

A knowledge base is structured so an agent can read exactly the part it needs at exactly the moment it needs it. It contains the company's operating model (strategy, positioning, offerings), control (roles, decisions, systems, metrics), and execution (SOPs, checklists, workflows). It is edited continuously — by humans and by agents — and validated against a small set of rules so it stays trustworthy.

The knowledge base and the spec serve the same function at different scales. A spec captures the durable intent of one piece of work. The knowledge base captures the durable intent of one business. Together they are how a finite context window can act with the company's actual taste, instead of generic competence.

A business that articulates its operating mind well will outcompete one that does not, even at the same level of model capability. That is not a future claim. It is already true.

6. The system gets better as the world gets better.

Most software systems decay. Code rots, dependencies drift, original authors leave, edge cases pile up, and the team gets steadily slower at making the same kind of change. The default time-direction of software is downhill.

A spec-driven, capability-based system goes the other way. Three flywheels turn at once.

Specs sharpen over time. Every iteration of the loop teaches you something about the work — an edge case you had not considered, a behavior you took for granted that needed to be explicit. The next version of the spec is more precise than the last. Compounded over a year, the spec for any given feature is dramatically more useful than the day it was written.

Capabilities compound. Every engagement contributes new vertical slices to the shared library. The library does not deprecate the way a normal codebase does, because each capability is self-contained — old ones do not block new ones. The fiftieth client engagement starts with forty-nine capabilities of inventory the first one did not have.

Models improve on their own schedule. The agent operating against this system in a year will be smarter than the one operating against it today. Because the system's interface to the agent is the spec, the test, the prompt, and the skill — and because all four of those are stable abstractions — the system absorbs model improvement automatically. No rewrites. No migration projects. The same prompts produce better work.

The limit case of all three turning together is the hypothetical one-shot: a perfect spec, a strong enough model, a rich enough capability library, and the system produces the right outcome in a single pass. We are not there yet. But the trajectory is toward there, and every piece of discipline in this document is an investment in moving along that curve.

The strongest version of the claim is this. This is one of the few kinds of software system that has positive time dependence. It gets better while you sleep.

7. The operating shape.

The system that runs the loop is deliberately small.

Operating shape
small human surface · large agent surface
intent
Me
Confirm specs, prioritize work, accept what ships.
operator
Shipmate
Watches the board, reports blocks, runs deployment commands.
gate
Now / Later
A simple queue decides what is authorized to execute.
labor
Agent
Runs the loop against the substrate and produces a PR.

Me. I confirm specs, prioritize development tasks, and decide what is good enough to ship. Three high-leverage decisions, repeated.

Shipmate. My CTO and operator. Watches the task board, notifies me on Slack when something needs attention, kicks off agent runs when I green-light work, reports back when work finishes or blocks, and runs deployments when I tell it to. Shipmate does not write code. It makes sure the right work happens at the right time with the right context.

The task board. A simple two-lane queue: Later and Now. New work — from me, from clients, from in-app feedback, from crash reports — lands in Later. Work moves to Now only when I green-light it. That move is the only execution gate in the system. No agent ever starts work on something still in Later, regardless of who created it.

The agent. One worker at a time, across all active repos. Receives a goal from the board, runs the spec/test/prompt/execute loop against the substrate (knowledge base, capabilities, templates), and produces a pull request. Spawns subagents when a goal benefits from parallel inner loops. Uses whatever skills fit the work; the workflow phases (plan, architect, design, build, review, push, deploy) are tactics inside the loop, not a fixed sequence.

That is the whole shape. Anything more elaborate is implementation detail. The reason this works is that it puts the human only in the places where context, judgment, and accountability matter — defining intent, prioritizing, accepting work — and delegates everything else to the loop.

The operating choices inside this shape (which board, which orchestration framework, which skills, which validation gauntlet) are an iterating layer. The shape itself is settled.

8. The business — productized agency.

The same principles, at company scale.

Traditional agency
Effort resets each engagement
Client work ships, but little returns to the next client. Revenue scales with labor.
Productized agency
Inventory compounds each engagement
Specs, context, and capabilities become reusable substrate for the next run.

A traditional agency consumes effort per engagement that does not return. Each client is a fresh start; the work that ships for one is rarely useful for the next. Labor cost scales linearly with revenue, and margin compresses as the team grows.

A productized agency built on the system above looks different. The substrate (the shared templates, the knowledge base structure, the operating shape) is built once. The capability library grows with every engagement and is available to every future engagement. The agent does the work the same way each time, against the same substrate, using the same loop. What is customized per client is the spec — the intent — and the small surface of capabilities the client actually needs. Most of the work is not bespoke.

That is the gap the business model lives in. The client perceives a custom application built specifically for them, because the spec, the knowledge base, and the selected capabilities really are theirs. The delivery cost looks more like a product than a service, because the substrate carries most of the load. Service-grade outcomes with high revenue at product-grade cost. The margin lives in the gap, and the gap widens as the capability library grows.

Three things this depends on, all of which are claims I am willing to make.

Context is the moat. The defensible asset is each client's captured business context, the agency's accumulated capability inventory, and the business knowledge and acumen of the agency operators. None of these are replicable from outside.

Build for one, deploy to many. A capability that ships for client A and is useful for client B only if A's specifics are scrubbed is not reusable; it is a snowflake. Folder-copy portability is what keeps the inventory honest.

The offer compounds with the inventory. What an agency built on this system can promise a new client in year three is qualitatively different from what it can promise in year one — not because the team is more skilled, but because the substrate they ship onto is richer. The offer becomes uniquely positioned because the inventory backing it has no equivalent elsewhere.

The business shape this produces is a small team with a large surface area, building custom client systems as a service, but delivering with the speed that a product would offer. The bet is that a system like this allows for increased speed, capability, and the opportunity to scale beyond traditional limits.

What is settled vs. what is iterating

Settled (will not change as I learn more):

  • The principles in §1 (intent is the bottleneck) and §2 (the four-step loop).
  • The three spec types in §3 and the discipline of co-evolution.
  • Capabilities as vertical slices, with folder-copy portability as the test.
  • The knowledge base as durable context.
  • The three flywheels and positive time-dependence.
  • The basic operating shape: me → OpenClaw → board → agent.
  • The productized agency thesis.

Iterating (current best implementation; will keep evolving):

  • Which skills the agent invokes inside the prompt and execute steps, and how they are packaged.
  • How OpenClaw orchestrates exactly — the wrapper, the failure handling, the run-state machine.
  • How the board ingests external requests when the contribute panel ships.
  • How the validation gauntlet is wired per client.
  • The opinionated stack itself (Expo, Fastify, Postgres, Cognito, AWS) — settled for now, but the principles work on any stack.
