Foundation Backfill Author Specs Technical Spec

This technical layer makes Foundation Backfill Author Specs deterministic enough for scripts and structured enough for agent judgment.

Status: active
Updated: 2026-06-07
Parent: foundation.backfill-author-specs.job

Implementation must preserve the source-of-record split: JSONL is canonical for layer rows, check artifacts, and eval receipts, while HTML summaries and report sections are generated for human reading. Each command must be resumable from artifacts and must not rely on chat memory.

Contracts

ContractRuleReason
Upstream completenessfoundation:author-specs:init starts only from a Process / Action Map handoff whose check, eval, summary, and optional report are current and outstanding.Later synthesis must not hide earlier discovery gaps.
Canonical JSONLLayer rows and eval receipts are JSONL; HTML summaries are derived.Agents need durable machine state and users need readable review surfaces.
Exact fill loopfoundation:author-specs:fill accepts one --process-map-id, one job spec path, and one technical spec path. It rejects --all, --batch-size, generated payload files, bulk JSON, multiple IDs, out-of-order targets, and unresolved prior row evals.Long-running work survives context compaction and cannot close coarse slices.
Deterministic checkervalidates row schema, upstream freshness, one row per active Process / Action Map row, target-owned spec IDs, spec metadata, graph metadata, owned paths, canonical sections, required job language, required technical contract language, and report state.Structural drift should fail mechanically before semantic evaluation.
Semantic evalscores each row for upstream traceability, job intent coverage, technical contract coverage, graph traceability, adequacy readiness, material row fidelity, spec-only rebuild readiness, and generic-compression rejection.Correctness includes meaning, specificity, evidence quality, and spec-only continuation readiness.
Material fidelityRow eval compares the authored job/technical text against the current Process / Action Map row and blocks if actor or role, trigger, intended outcome, domain object, actions, states, transitions, permissions, rules, visible/operator behavior, edge cases, recovery paths, evidence refs, explicit gaps, blockers, or human decisions are omitted.Author Specs must preserve upstream intent, not only produce valid-looking spec files.
Rebuild readinessRow eval blocks if the spec pair does not answer what behavior must exist, required contracts, current evidence, flexible versus constrained implementation space, failures/recovery, row-specific verification, and remaining uncertainty.A future agent must be able to continue from specs without rediscovering source.
Row continuation signalA row-targeted eval that passes writes next action as selecting the next Author Specs target; only non-target all-row eval can direct handoff.A single outstanding row must not be mistaken for whole-layer completion.
Handoff gateeach active Process / Action Map row has a ready job/technical spec pair or explicit blocker; every Author Specs row preserves child/sole capability refs and rejects parent, needs-split, blocked, or non-queueable refs as active work; checker passes; every non-pending row has a current outstanding eval receipt; --next returns no target; HTML summary exists; report names next layer as Evaluate Job Slices.Each layer is a reproducible correct unit before the next layer consumes it.

Checker and fill loop

  1. Read active report, run log, Process / Action Map artifact, current Author Specs artifact, and current eval receipts.
  2. Select the next target with --next; the target is either pending, needs revision, blocked but unevaluated, or missing a current outstanding receipt.
  3. Inspect only that target's Process / Action Map row and referenced Context Pack evidence.
  4. Author one job/descriptive spec and one technical spec, then record their paths for the current --process-map-id.
  5. Run the checker; repair structural failures before semantic eval.
  6. Run row-targeted eval; revise the same target until the row receipt is outstanding, material row fidelity is complete, and rebuild-readiness findings are gone.
  7. After no target remains, run handoff check, all-row eval, and report.
