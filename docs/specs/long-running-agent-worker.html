Long-Running Agent Worker

This job gives agents a durable way to work for a long time: define the goal, preserve state, choose one bounded unit, produce evidence, check quality, and hand off clearly when the exit criteria are met.

Intent

Make long-duration agent labor reliable enough that another actor can inspect progress, resume the work, and judge whether the run honored its standard of completion.

Trigger

A user request, scheduled automation, subagent assignment, backfill, research pass, scout run, or other task expected to span long wall-clock time, context compaction, restart, or handoff.

Current handling

  • Current actors: Codex workers, subagents, scheduled automations, and watchful supervisors.
  • Current path: long-running behavior has been spread across Research, Backfill, Shift Clock, and supervisor instructions.
  • Need: the broad long-running worker contract should own the worker and supervisor skills, while domain jobs own their source schemas, validators, and final artifacts.

Proposed handling

  • Worker path: load the governing rules, create or confirm the goal, set run state, work one bounded unit at a time, record evidence, validate material changes, and hand off from durable state.
  • Supervisor path: observe durable state and thread evidence, steer only when needed, and debrief behavior against the workflow contract.
  • Shift-clock path: when a clock is used, treat it as committed work time and an exit gate with sparse checks tied to real progress.

Process

  1. Define the run: name the goal, standard of completion, run ID, branch or worktree, expected artifacts, validators, and handoff location.
  2. Load bounded context: read the local rules, owning skill, active spec, run state, queue, and only the source material needed for the next unit.
  3. Choose one unit: select one source, row, file, queue item, route, test failure, quality audit, or search family.
  4. Produce durable progress: write one accepted finding, rejection, no-result record, source status, code/spec patch, validation receipt, or handoff update.
  5. Check and revise: run the cheapest relevant check after material changes and revise the current unit until it meets the job standard or has a named blocker.
  6. Continue or close: select the next useful unit until the exit criteria are met, the shift clock permits closure, or a concrete blocker is recorded.
  7. Hand off: report run ID, branch, changed artifacts, validations, blockers, clock state when present, unresolved risks, and the exact continuation point.

Revision

When a long-running run loses state, damages artifact boundaries, skips checks, hands off vaguely, or shows a better pattern, revise the smallest durable owner: AGENTS rule, worker skill, supervisor skill, shift-clock primitive, domain job spec, validator, or eval.
