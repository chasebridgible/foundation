Business Graph Is Visible

Foundation should make a business inspectable: what it can do, which jobs make those outcomes real, who or what handles the work, how work moves, what tools and evidence support it, and where the operating system is fragile or stale.

Status: draft
Updated: 2026-06-01
Parent: foundation.knowledge-data-layer.system
Confidence: medium

Intent

The Business Graph Is Visible turns Foundation's HTML-native business substrate into a navigable graph for humans and agents. The canonical artifacts remain reviewable HTML specs and structured metadata. Visual canvases, cards, filters, and app views are derived surfaces that help a client see the business instead of reading a long document set. The broader principle is defined in Visible Business Foundation: Foundation should make business architecture inspectable and connected to reality.

The graph should treat software and non-software work the same way: a person, vendor, agent, spreadsheet, SaaS workflow, or custom application may all implement jobs that support a capability. The durable object is the business outcome and the work required to produce it reliably, not the current implementation choice.

Foundation itself is the first required graph-compatible corpus. Every Foundation HTML spec must carry valid graph-metadata, and new specs must be scaffolded with graph metadata by default.

FieldContractEvidence or source
Reliable outcomeGoals, capabilities, jobs, actors, evidence, gaps, ownership, and next work are visible as a graph that links back to source HTML specs.Foundation capability remap and graph renderer contracts.

Boundary

This spec defines the business graph contract and visualization requirements. It does not select a web framework, canvas library, database, embedding strategy, authentication model, or client portal implementation. Those belong in future technical and job specs once a pilot proves the graph shape.

Graph contract

The graph is a derived view over canonical substrate artifacts. Each visible node should resolve back to an HTML artifact, metadata object, source row, or explicit gap.

Node typePurposeRequired connections
CapabilityNames a reliable business outcome the organization must produce.realized-by job edges, evidence, confidence, owner or owner gap, source coverage, revision state.
JobNames the recurring body of work that makes one or more capabilities exist, operate, or improve.supports capability edges, actors, process, inputs, outputs, tools, states, rules, evidence, failure/recovery.
ProcessShows how a job is actually performed through steps, decisions, checks, handoffs, and exceptions.Job, actors, tools, evidence points, decision rules, open questions.
ActorRepresents a role, person, vendor, system, agent, customer, or partner that performs or influences work.Jobs performed, capabilities depended on, authority, handoffs, overload or single-point dependency signals.
Tool or systemRepresents software, hardware, data source, spreadsheet, custom app, automation, or operational asset used by work.Jobs supported, actors using it, data or evidence produced, access or failure risks.
EvidenceRepresents the source, metric, observation, review, or data that supports a graph claim.Source registry row, capability/job/process claim, confidence, freshness, access boundary.
GapRepresents missing, conflicting, stale, fragile, or low-confidence operating knowledge.Owning capability/job/process, source of uncertainty, next review or remediation path.
Core invariant: a client-facing graph must never imply more certainty than the substrate supports. Unknown ownership, partial answers, stale evidence, and contradictory sources are first-class graph states.

Metadata contract

HTML remains the durable reading surface, but each artifact must embed enough structured metadata for graph extraction and agent routing. V1 uses a dedicated graph-metadata JSON script block alongside spec-metadata.

ArtifactEmbedded metadata should exposeNotes
Capability mapCapability ID, name, outcome, family, status, confidence, realized-by job edges, evidence refs, owner refs or owner gap, strategic importance, health signals.Capabilities stay high-level and should not absorb process details.
Job specJob ID, owning or supported capabilities, supports capability edges, actors, process refs or embedded process sections, inputs, outputs, states, rules, tools, evidence, metrics, failure/recovery, open questions.Job specs carry most operational detail and must be inspectable from their owning capability.
Role or actor registryActor ID, actor type, responsibilities, jobs performed, decision rights, capacity/load notes, access boundaries, succession or dependency risks.Actor can be human or non-human.
Source registrySource ID, source type, location, owner, freshness, trust level, access boundary, related claims, related capabilities/jobs/processes.Sources do not become truth until promoted into reviewed substrate.
Metric or eval specMetric or evaluation ID, definition, evidence source, interpretation rules, target, review cadence, revision target.Metrics explain graph health but do not replace human review.

The graph extractor should prefer stable IDs over visible headings, and should preserve source links so every surfaced claim can be inspected. Derived canvas state may cache layout positions, filters, and user preferences, but it must be rebuildable from canonical artifacts plus optional view-state metadata.

Operating loop

  1. Register sources: transcripts, recordings, existing docs, SOPs, dashboards, SaaS exports, spreadsheets, screenshots, and observations enter the source registry with owner, freshness, trust, and access fields.
  2. Extract evidence: agents convert sources into structured answer rows and signal rows for capabilities, jobs, actors, processes, tools, metrics, risks, and gaps.
  3. Synthesize substrate: reviewed signal groups become capability maps, job specs, actor registries, process sections, metric definitions, and eval rules.
  4. Extract graph: a graph builder reads HTML metadata and source registry rows, validates links, and produces a derived graph model.
  5. Render views: humans and agents inspect capabilities, jobs, actors, processes, tools, evidence, and gaps through focused graph views.
  6. Evaluate reality: outcomes, source changes, metric evidence, human review, and operational observations update graph health and confidence.
  7. Revise substrate: changes route back to the smallest canonical artifact: source row, capability, job spec, process section, actor registry, metric definition, or eval spec.

View contracts

The graph should be visible through a primary capability map because clients need one stable place to explore the business. The canvas is an exploration surface; the canonical source remains the substrate.

The canvas must optimize for inspectability, not graph-edge exhaustiveness. Metadata can contain every valid relationship, but the visual surface should draw only the local path the user has intentionally revealed. Global support, containment, dependency, evaluation, and provenance relationships remain available through details and source links unless the active view needs them to answer its primary question.

ViewPrimary questionRequired behavior
Capability mapWhat can the business do, and how is each capability handled?Render top-down from centered system nodes to one horizontal capability band; do not wrap capability cards into multiple rows; the first viewport centers the graph stage even when the canvas is narrower than the full graph; suppress global overview edges; activating a capability expands jobs inline below that capability and draws only local realized-by job edges; the expanded capability, its jobs, and its visible realization edges share a deterministic branch accent color; activating a job expands process, actors, tools, evidence, metrics, and gaps inline and draws only that job's local context edges within the same capability map.
Actors mapWho or what handles the work?Render all actors as the default actor surface without a global actor-job mesh; activating an actor expands the jobs that actor performs and visually draws those handled-job edges from actor to job without switching back to the capability map.

States and rules

State or ruleMeaningRequired behavior
Canonical versus derivedHTML substrate and source registry own meaning; graph/canvas/app views are derived.Derived views must link back to source artifacts and must be rebuildable.
Partial knowledgeSome intake questions, capabilities, jobs, or evidence paths are incomplete.Show partial, unknown, and low-confidence states instead of hiding gaps.
Cross-lane synthesisOne capability or job may be supported by answers from many intake lanes and sources.Graph links should preserve every material contributing source and confidence note.
Actor neutralityPeople, vendors, software, agents, partners, and customers can all perform work.Actor type should be metadata, not a separate model that breaks capability/job comparison.
Process placementCapabilities name outcomes; jobs carry the recurring work and process detail.Capability views summarize handling but link into job specs for detailed processes.
Capability/job mappingCapabilities are not considered mapped unless their child jobs are visible as realization edges.Capability graph metadata must use realized-by from capability nodes to job nodes, and job graph metadata must use supports from job nodes to their owning capability.
Canvas navigationA graph can exceed the available viewport.The local canvas must support pointer drag panning from the canvas background, keep source links and node clicks usable, and preserve selection while the user moves around the graph.
Document navigationThe same HTML file may load the Foundation docs navigation.The graph page must keep a fixed canvas shell, avoid inherited document padding, and rely on the shared docs navigation control for collapsing or restoring the docs nav. The collapse control belongs in the far-left document sidebar so it can apply across generated specs, canvases, and target repos that load the shared nav scripts.
Expandable hierarchyClients need a calm overview before seeing process detail.The default capability map shows centered system nodes and capability nodes first; activating a capability expands or collapses its jobs, and activating a job expands or collapses that job's process, actors, tools, evidence, metrics, and gaps in the same map.
Visual edge policyRaw graph relationships are not all useful visual relationships at the same time.The default capability overview and actor overview draw no global edges. Expanded capability views draw capability-to-job realization edges only. Expanded job views draw job-to-process, job-to-actor, job-to-tool, job-to-evidence, job-to-metric, and job-to-gap edges only. supports, depends-on, contains outside capability-to-job realization, evaluates, and template/provenance relationships should be available in the selection details and source links rather than drawn as global canvas lines.
Branch colorUsers need to preserve orientation after expanding one or more capabilities.Each expanded capability should receive a stable accent color derived from its ID. Jobs revealed under that capability and the local realization edges connecting them should share the same accent. Job context edges may use the same branch accent at lower visual weight so the selected path remains recognizable.
Actor expansionClients need to see actor workload without leaving the graph model.The actors map shows actors first; activating an actor expands or collapses the jobs that actor handles.
Evidence preservationEvery graph claim should know why the system believes it.Claims without source, review, metric, or observation evidence show as unsupported or provisional.
Evolution over timeThe business graph changes as the business, sources, metrics, and review judgments change.Views should surface freshness, revision history, and changed-source impact before presenting stale certainty.
Foundation-wide compatibilityEvery Foundation HTML spec participates in the graph.Technical, eval, template, and index specs must expose support, evaluation, template, or index nodes rather than remaining invisible registry entries.

Open questions

  1. First pilot surface: Should V1 render a static HTML graph summary, a lightweight interactive local view, or a client portal behind login?
  2. Canonical metadata schema: Should graph metadata live in each spec's existing spec-metadata, a separate JSON script block, or generated JSONL sidecars?
  3. Intake promotion: Which answer fields from docs/fundamentals/business-intake.html should be promoted automatically, and which require review before updating durable substrate?
  4. Canvas persistence: Should manual layout and annotations be saved as view state, or should the graph remain fully auto-laid out from substrate?
  5. Client interaction model: Can clients edit graph nodes directly, or do edits create proposed substrate revisions for review?
  6. Health scoring: Which signals should influence capability health: confidence, evidence freshness, metric performance, actor overload, missing process detail, source conflicts, or strategic importance?
