Golden descriptive slice excerpt

A field coordinator needs to check into a scheduled visit from a phone, confirm they are at the right location, and leave the team with a reliable timestamp and readiness signal before work starts.

Intent contract

TopicIntentEvidence
Primary userAssigned field coordinator checking into a visit at the customer site.VisitCheckInPanel.tsx, route assignment guard.
Primary outcomeThe visit has a trustworthy arrival event and the next work action becomes visible.checkInVisit.ts, visit_events.sql.
Secondary userOperations supervisor reviewing manual overrides, late arrivals, and location warnings.operations dashboard status badges and override notes.
Architecture stanceDescriptive spec defines behavior, state, language, and recovery without requiring REST, RPC, server actions, or a specific database shape.Backfill architecture flexibility rule.

User flow

  1. Open visit: the coordinator opens today's visit and sees address, scheduled time, customer name, and check-in status.
  2. Confirm location: the coordinator taps Check in. The system compares current location with the visit address tolerance and shows a clear warning when the user is outside the expected radius.
  3. Submit check-in: the coordinator confirms arrival. The visit moves to Checked in, records timestamp and user, and exposes Start work as the next action.
  4. Recover: if location is unavailable, the coordinator can submit with a reason. The operations view flags the check-in for review.

This descriptive layer avoids framework, database, and cloud choices. It defines user-visible behavior, copy, states, recovery, and unresolved product decisions clearly enough for a build agent to implement the intended experience without copying the current UI framework.

Quality notes

  • The durable queue makes continuation mechanical: the next agent can resume from the slice ledger without reconstructing context from prose.
  • The descriptive excerpt is specific while preserving implementation flexibility: it defines intent, states, copy, recovery, and rendered experience without choosing framework architecture.
  • The technical excerpt preserves contracts and makes flexible implementation choices explicit: required behavior is separated from current evidence and replaceable implementation details.
  • The evaluator scorecard is strict enough to distinguish excellent, rebuild-useful specs from passable summaries.
  • A generated backfill that only says "field visits have check-in" should score around 30-45; a broad but useful summary should score around 65-75; an acceptable slice should score 96-100.
