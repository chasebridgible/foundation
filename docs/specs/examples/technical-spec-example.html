Kitchen queue technical contract

Every software task needs a technical spec. This example needs a deeper one because there are many plausible ways to move order state in real time. This spec chooses a server-authoritative status transition model with websocket fanout and stale-write protection.

Status: active
Parent: fup.kitchen-queue.descriptive
Updated: 2026-05-11

Non-goals

  • No peer-to-peer tablet sync. All writes go through the server.
  • No optimistic customer status beyond the last committed server state.
  • No kitchen batching algorithm in v1. Sorting uses promised time and created time only.

Internal sequence

  1. Client sends mutation: PATCH /api/orders/:orderId/items/:itemId/status with the requested next status and the card's last seen version.
  2. Server validates: staff PIN session, store access, item belongs to store, transition is legal, and client version is not stale.
  3. Server mutates: one database transaction updates the order item status, increments item version, recalculates aggregate order status, and writes an order status event row.
  4. Server publishes: OrderStatusBus emits kitchen.order.updated to staff tablets and customer.order.statusChanged to customer trackers.
  5. Client reconciles: kitchen queue applies the event by order ID and version. Older events are ignored.

API and websocket contracts

Status mutation request

{
  "nextStatus": "in-progress",
  "lastSeenVersion": 7
}

Kitchen queue event

{
  "event": "kitchen.order.updated",
  "orderId": "ord_123",
  "version": 8,
  "orderStatus": "in-progress",
  "items": [
    { "id": "item_1", "status": "in-progress", "version": 8 }
  ],
  "occurredAt": "2026-05-11T15:24:00.000Z"
}

Event names and payload field names are contract names. A code change that renames them must update this spec and the test spec in the same commit.

Timing, concurrency, and ordering

  • Stale write: if lastSeenVersion is older than the server item version, the server returns 409 with the latest item and order state.
  • Concurrent taps: item update uses a row-level lock. The first legal transition commits; the second receives the newer state and does not double-advance.
  • Event order: clients apply an event only if event.version > current.version for that order.
  • Normal propagation target: kitchen tablets and customer trackers should reflect committed status within 2 seconds under normal network conditions.
  • Reconnect catch-up: on reconnect, client requests events since its last seen order version, then falls back to full queue fetch if the gap cannot be filled.

Boundaries and abstraction level

  • Owns: kitchen status mutation route, status transition helper, websocket event payload, reconnect catch-up query, order item status migration.
  • References: kitchen page rendering from descriptive spec.
  • Does not own: Stripe webhook handling, menu CRUD, staff PIN auth internals, customer checkout.
