Let a user manage notification preferences

This job spec describes the user-facing and operator-facing body of work required for a signed-in user to view, change, save, and trust notification preferences.

Status: active example
Type: job
Updated: 2026-05-31

Capability supported

Users control communication preferences is the capability. The job is the named body of work that makes one user preference change possible and trustworthy.

Current handling to capture

A real version would name the existing settings surfaces, preference fields, defaults, communication providers, permissions, legal constraints, and current failure modes such as stale state, duplicate notifications, or hidden unsubscribe logic.

Proposed handling

If this job is being redesigned, the proposed handling names the changed interface, actor, provider contract, evidence rule, or recovery path. After the eval accepts it, the proposal becomes current handling and the old behavior moves into revision history.

Process inside the job

  1. Open settings. A signed-in user reaches notification preferences from account settings.
  2. See current state. The screen shows email, SMS, and product update preferences using labels that match actual system behavior.
  3. Change a preference. The user toggles a preference, sees a stable pending state, and cannot accidentally submit conflicting choices.
  4. Save. The system persists the change for the current account and shows a confirmation only after the server accepts it.
  5. Recover from failure. If the save fails, the screen restores the last accepted state and tells the user what happened.
  6. Prove the result. A refresh, audit event, and downstream provider state show the preference was applied.

Technical boundary

The job spec owns the intended behavior and process. A technical spec is still required before implementation because it must choose the data model, API contract, authorization rule, provider integration, migration, and failure-handling mechanics.
