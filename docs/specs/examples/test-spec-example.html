Kitchen queue verification contract

The kitchen queue is correct when a staff user can advance order items quickly and safely, customers see committed status changes, concurrent tablets cannot corrupt status, and reconnect recovery preserves the true queue.

Status: active
Validates: descriptive + technical
Updated: 2026-05-11

Acceptance conditions mapped to specs

IDConditionMaps toMust prove
KQ-A1Staff can advance an item from received to in progress to ready with one tap per transition.fup.kitchen-queue.descriptive#interface-journeyUI updates and customer tracker reflects committed status.
KQ-A2Staff cannot skip received directly to ready.fup.kitchen-queue.descriptive#states-and-rules, fup.kitchen-queue.technical#data-contractInvalid transition is rejected and current server state is shown.
KQ-A3Two tablets updating the same item cannot double-advance or lose the true state.fup.kitchen-queue.technical#concurrency-and-timingOne write wins; stale client receives latest state.
KQ-A4Reconnect catches up to committed queue state.fup.kitchen-queue.descriptive#edge-cases, fup.kitchen-queue.technical#failure-modesDisconnected client disables writes, reconnects, and renders latest events or full refetch.

Commands or evidence required

  • npm test -- order-card - proves card state labels, one-tap actions, pending disabled state, and accepted status text.
  • npm test -- kitchen-orders - proves transition guard, stale version handling, row-lock behavior, and event persistence.
  • npm test -- order-status - proves status transition table and event payload keys.
  • npm run test:e2e -- kitchen-queue - proves staff update and customer tracker synchronization across the stack.
  • evidence/kitchen-queue/tablet-density.png - manual screenshot showing active queue on 10-inch landscape viewport.

Co-evolution rule

Required: if fup.kitchen-queue.descriptive changes any state, label, journey step, or recovery path, this test spec must change in the same commit. If this test spec changes, code-level tests must be regenerated or updated.

Reviewers should reject a kitchen queue behavior diff when code paths change without corresponding updates to the descriptive spec, technical spec, and this test spec coverage table.
