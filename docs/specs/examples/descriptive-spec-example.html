Kitchen queue for live order prep

Kitchen staff need a tablet-friendly queue that shows active orders in the order they should be prepared, lets them advance each item with one tap, and keeps customer status screens current without requiring staff to think about synchronization.

Status: active
Owner: Foundation example
Updated: 2026-05-11
Confidence: medium

Product promise

During service, a cook can glance at a 10-inch landscape tablet and know what to make next, what is already in progress, and what can be marked ready. The screen favors speed and confidence over management detail.

Scope

This spec covers the kitchen-facing queue and the user-visible status effects of staff actions. Menu management, payment, manager reports, and delivery assignment are outside this feature.

Interface journey

  1. Staff opens Kitchen Queue. The page shows active orders sorted by promised pickup/delivery time, with the oldest urgent order first. Empty state says "No active orders".
  2. New order arrives. A new card appears without page reload. The card is highlighted for 2 seconds and includes order number, pickup/delivery label, promised time, item list, and current item states. KQ-A1
  3. Staff taps Start on an item. The tapped item changes from received to in progress. The button becomes disabled while the update is being saved, then changes to "Mark ready".
  4. Staff taps Ready. The item changes to ready. When every item in the order is ready, the order card moves to the Ready group and the customer tracker shows "Ready for pickup" or "Out for delivery" depending on fulfillment type.
  5. Order leaves the kitchen queue. Pickup orders leave when a staff user taps "Picked up". Delivery orders leave when a staff user taps "Out for delivery".

  • Staff cannot skip received directly to ready.
  • Order cards are grouped as Active, Ready, and Late. Late is a visual treatment, not a separate order state.
  • One tap must be enough to advance an item. No confirmation dialog appears for normal status advancement.
  • Buttons disabled by a pending save must preserve their width so the card does not shift during rush.

Edge cases and recovery paths

  • Websocket disconnects: a small banner says "Reconnecting - queue may be stale". Staff can keep viewing the last known queue, but status buttons are disabled until reconnect completes.
  • Status save fails: the item returns to its previous state and the card shows "Could not update. Try again." The action button re-enables.
  • Another tablet updated first: this tablet accepts the newer state and shows a short "Updated elsewhere" note on the card.
  • No active orders: the page shows a quiet empty state with current store name and "No active orders" rather than a blank board.
  • Order is canceled: the card leaves the queue and a small toast says "Order #123 canceled". It is not shown as ready or completed.
