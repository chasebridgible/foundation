Knowledge and Data Layer

The Knowledge and Data Layer turns business intake, existing organizational sources, and operating data into durable substrate that people and agents can use to understand, improve, and evaluate a business system.

Status: draft
Updated: 2026-06-01
Confidence: medium
Scope: high-level system spec

Intent

This spec defines the first durable shape for an Orderly Systems knowledge and data layer. The layer should capture how a business works, connect that understanding to the organization's existing sources, collect operating data where useful, and produce context, evidence, and evaluation for business improvement work.

The system has two cooperating loops. The knowledge loop uses the combined business intake workspace to ask business intake questions, read existing sources, mark answer status, and turn reviewed answers into canonical operating substrate. The data loop brings SaaS and existing business data into an Orderly-managed database so Reality answers can use current measurements instead of only interview memory or static documents. Evaluation then judges those answers against intent. The Visible Business Graph child spec defines how that substrate can become an inspectable graph of capabilities, jobs, actors, processes, tools, evidence, and gaps.

Out of scope

This draft does not choose a database, vector store, sync vendor, SaaS connector set, permission model, or UI. It defines the conceptual contract that those implementation decisions must satisfy later.

System model

The layer should preserve the distinction between raw sources, canonical operating substrate, structured data, and retrieved context. Search and databases are useful, but neither should become the only place where business meaning lives.

Knowledge and data layer
data -> reality; intake + sources -> substrate -> revision
The registry maps sources.

It says what exists, where it lives, what it is about, who owns it, how fresh it is, and which operating concepts it supports.

The substrate owns meaning.

It preserves the current best understanding of capabilities, jobs, processes, roles, metrics, decisions, specs, and evaluation rules.

The database stores reality.

It holds selected measurements from SaaS and operational data so Reality answers can include what is happening now.

Interface journey

  1. Initialize: choose a client or business unit, create a source namespace, and load docs/fundamentals/business-intake.html.
  2. Interview and observe: collect business intake answers from conversations, meetings, existing documents, and real work.
  3. Register sources: create source-registry rows for documents, files, tools, dashboards, spreadsheets, meeting notes, SaaS objects, and data exports.
  4. Classify meaning: link each source to the capabilities, jobs, processes, actors, tools, metrics, risks, decisions, or evidence it informs.
  5. Build substrate: turn the strongest answers into canonical artifacts such as capability maps, job specs with process sections, role specs, metric definitions, and eval specs.
  6. Ingest data: pull selected SaaS or existing data into an Orderly operating database when the data can produce useful measurements for Reality answers.
  7. Assemble context: build a bounded context pack for one question, job, capability, recommendation, automation, or implementation path.
  8. Act: produce a recommendation, process, software system, agent workflow, checklist, dashboard, or handoff.
  9. Evaluate: compare evidence against intent. Evidence may come from source documents, observed work, human review, or Reality answers produced from the operating database.
  10. Revise: update the relevant substrate, source-registry rows, data definitions, job specs, or eval specs so future work starts from better context.

States and rules

State or ruleMeaningRequired behavior
Source registry rowA durable record for a source the organization uses or has produced.Rows should carry source type, location, owner, access boundary, freshness, trust level, business domain, related capabilities/jobs/processes, and evidence value.
Canonical substrate artifactA reviewed artifact that states operating meaning.Artifacts should name intent, context, evidence, and revision rules for the layer where work happens.
Operating databaseA structured store of selected data from existing systems.Data should be ingested when it can produce measurements for Reality answers, dashboards, recommendations, automation, or context generation.
Metric definitionThe substrate artifact that says what a metric means and how to interpret it.Metric definitions should sit in the substrate even when metric values sit in the database.
Context packA bounded retrieval product for a specific job or question.It should include only relevant source refs, substrate artifacts, data evidence, definitions, exclusions, confidence, and open gaps.
EvidenceObservable signal from documents, data, users, operations, or review.Evidence should cite where it came from and which intent it answers.
EvaluationJudgment comparing evidence against intent.Evaluation should produce accept, revise, continue, stop, or explicit uncertainty, then route revisions to the smallest substrate artifact.
FreshnessWhether a source, answer, data pull, or substrate artifact still reflects reality.Stale sources should remain visible and flagged, not silently removed from retrieval.
Derived indexesSearch indexes, embeddings, graph views, dashboards, and caches.Derived structures should be rebuildable from source registry rows, canonical substrate, and operating data.

Edge cases and recovery paths

  • Conflicting sources: keep both sources linked, mark the conflict, choose a canonical answer only after review or evidence.
  • Untrusted data: ingest it with confidence and lineage fields, then prevent it from driving evaluation until a metric definition or review rule exists.
  • Private or restricted documents: register the source and its access boundary without leaking contents into broad context packs.
  • Stale SOP or dashboard: do not delete it from the registry. Flag freshness and route revision to the owning process, metric, or source row.
  • Interview answer without evidence: preserve it as an intake answer with confidence, then link it to follow-up evidence or mark it as unresolved.
  • Data without meaning: do not treat a table or dashboard as useful until the substrate defines what the fields mean and what decisions they support.

Hidden mechanics as operating contracts

MechanicOperating contractFailure promise
Source registrationThe system has seen the source before claiming to understand the business area it covers.Missing source coverage becomes an explicit gap in the context pack.
Substrate promotionDraft answers become durable only when they are clear enough to guide future human or agent action.Weak answers stay in intake or review state instead of becoming canonical operating memory.
Data ingestionOnly selected operational data is normalized into the Orderly database, and each data set should serve a known measurement or context purpose.Unmapped data is visible as a data gap or exploratory source, not used as an authoritative Reality answer.
RetrievalRetrieval assembles context from registry, substrate, and data; it does not decide truth by itself.When retrieval confidence is low, the context pack names missing sources, stale rows, conflicting evidence, or absent metric definitions.
EvaluationEval specs decide how evidence will be judged before the system claims improvement.Evidence without an eval rule becomes reporting, not learning.

Open questions

This spec is intentionally high level. These questions should be answered through pilots rather than abstract design alone.

  1. First useful vertical: Which lane should prove the system first: capabilities, processes, data/metrics, or systems/coordination?
  2. Registry name: Should the business-facing term be Organization Source Registry, Business Source Registry, or another name?
  3. Data path: Which operating measurements should feed Reality answers first, and which metric definitions are required before those measurements can be trusted?
  4. Canonical artifact set: Which artifacts are required in V1: capability map, job specs with embedded process sections, role specs, metric definitions, source registry, and context packs?
  5. Access model: How should the system represent private, client-owned, employee-owned, and vendor-owned sources?
  6. Freshness rules: What should force a source row, data pull, context pack, or substrate artifact back into review?
  7. Client delivery shape: Does Orderly host the registry and operating database, install them into the client's environment, or export portable artifacts?
