Shift Clock Eval

A shift-clocked run is acceptable only when the clock made the worker continue useful bounded work, not wait, poll, produce filler, or exit before the current-run clock expired.

Status: active
Updated: 2026-06-20
Canonical section: #verification-contract

Verification contract

The reviewer must be able to inspect the shift clock start, checks, work evidence between checks, final expired state, and handoff without reading chat history. The clock passes only when it caused productive continuation and preserved artifact quality.

Acceptance conditions mapped to specs

IDConditionMaps toMust prove
SC-1The current run started a shift clock before substantive work.foundation.shift-clock.technical#data-modelClock state and start receipt include run ID, start, deadline, duration, and mechanism.
SC-2The worker did not exit while the clock was open.foundation.shift-clock.capability#operating-contractAny expired:false check is followed by more bounded work or a concrete external blocker.
SC-3The worker did not watch the clock.foundation.shift-clock.technical#concurrency-and-timingClock checks are sparse and separated by substantive evidence-backed work.
SC-4The extra work preserved quality boundaries.foundation.shift-clock.capability#scoreboardNo weak filler rows or internal evidence polluted reviewer-facing artifacts.
SC-5The run closed with reviewable evidence.foundation.shift-clock.capability#evidence-and-evaluationFinal handoff names run ID, branch, final clock state, validations, artifacts, blocker or next action.

Co-evolution rule

Required: when the shift-clock capability changes, update the worker skill, supervisor skill, technical contract, and eval together. When a project changes clock commands or receipts, update the project spec and checker in the same change.

Do not accept a future shift-clocked run that passes only by waiting, checking repeatedly, adding filler, or claiming effort without durable work evidence.
