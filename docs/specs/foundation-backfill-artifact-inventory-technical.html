Foundation Artifact Inventory Technical Spec

The Artifact Inventory is a deterministic-plus-agentic indexing subsystem: commands create complete file rows, agents fill semantic meaning in bounded batches, and graph checks connect files to specs, capabilities, and verification.

Status: active
Updated: 2026-05-27
Parent: foundation.backfill-artifact-inventory.descriptive

This is a T5-level Foundation workflow because later backfill layers and steady-state build agents depend on its completeness. The implementation should prefer scripts for enumeration/checking, durable artifacts for graph state, and agent judgment for semantic row filling and quality evaluation.

Internal sequence

  1. Enumerate: generate docs/specs/backfill/file-manifest-YYYYMMDD-NN.json from repo-owned files. Default source is tracked files plus untracked non-ignored files. Git ignored files are outside the repo-owned set unless explicitly included.
  2. Initialize: create docs/specs/backfill/artifact-inventory-YYYYMMDD-NN.jsonl with one row for every manifest file, mechanical fields populated, semantic fields empty, and status pending.
  3. Seed or fill: use the Artifact Inventory fill loop or deterministic V1 fill helper to update bounded pending batches. The helper is a first-pass mapping assistant and eval can still route weak rows back to the agent loop.
  4. Select batch: choose pending rows by deterministic order plus priority hints: entry points, routes, schemas, services, jobs, tests, docs, then support files.
  5. Inspect: read a bounded batch and any immediately related files needed to understand each row.
  6. Fill: update semantic fields, status, confidence, related paths, and review flags.
  7. Validate: run the Artifact Inventory checker after batch updates. Batch mode may report remaining pending rows; handoff mode requires zero pending rows.
  8. Evaluate: run an Artifact Inventory quality eval on deterministic risk-based samples before advancing to Surface / Function Map.
  9. Calibrate: run Sandia Oil as the first tractable calibration repo before large-repo use and after schema or rubric changes. Compare to human-authored expected rows only after that fixture exists.
  10. Link graph: after the spec and capability layers exist, attach behavior-bearing rows to capability IDs, spec section IDs, and verification targets or explicit gaps.
  11. Resume: after context compaction, interruption, or a new agent session, reread target AGENTS.md, Foundation AGENTS.md, the active skill, report, Artifact Inventory, and run log, then continue from pending or stale rows.
  12. Refresh: during steady-state build work, recompute changed-file rows, check graph links, and route affected specs/tests before handoff.

Graph links

Artifact Inventory rows are the bridge from implementation evidence to durable intent. Links are allowed to mature over time: bootstrap rows may start with inferred domains and unresolved capability hints, but steady-state rows must resolve to the spec registry and Capability Map when those artifacts exist.

LinkRequired forValid targetFailure behavior
File to spec sectionBehavior-bearing files, behavior-defining docs, config with behavior impact, and tests.specId#sectionId present in the generated spec registry.Graph checker fails or records an unresolved bootstrap flag.
File to capabilityBehavior-bearing implementation, tests, workflows, jobs, and operator surfaces.Capability ID in the active backfill or steady-state Capability Map.Capability inference/eval routes the row to review.
File to verificationBehavior-bearing files after capability/spec coverage exists.Command, test path, eval ID, manual evidence, or explicit test gap mapped to the same spec/capability.Handoff reports a test-gap or verification-link finding.
Related filesEvery mapped row when useful evidence exists.Other manifest paths, external system labels, or generated/support references.Quality eval can route vague or missing relationships back to the fill loop.

The graph checker should support both bootstrap mode and strict mode. Bootstrap mode permits unresolved capability/spec/test links when the downstream artifact does not yet exist and requires review flags. Strict mode requires all behavior-bearing rows to resolve.

Timing, concurrency, and ordering

  • Single-agent baseline: correctness cannot depend on multiple agents. Parallelization may be an optimization only if rows are disjoint and merged through the same checker.
  • Batch size: choose batches small enough for the agent to inspect accurately and update rows before losing context.
  • Change handling: changed files invalidate affected rows and downstream surfaces/capabilities derived from them.
  • Graph drift: changed specs, capability rows, or verification targets can invalidate registry links even when file hashes are unchanged.

Boundaries and abstraction level

  • Owns: file manifest, Artifact Inventory schema, fill loop, graph-link model, inventory checker, inventory quality eval contract.
  • Feeds: Surface / Function Map, Capability Map, Author Specs, stale-doc review, cleanup planning.
  • Consumes: generated spec registry, Capability Map, test/eval metadata, and run-log/report state.
  • Does not own: capability clustering, final spec graph evaluation, or code/document cleanup.
