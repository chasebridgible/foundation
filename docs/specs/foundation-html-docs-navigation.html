Foundation HTML Docs Navigation

Every Foundation-based repo needs a shared left-side document browser for its durable HTML files. The browser's structure comes from that repo's actual folders and files. It complements, but does not replace, the spec registry that agents use to resolve specs by capabilities, jobs, processes, paths, and coverage.

Status: active
Updated: 2026-05-30
Canonical section: #product-intent

Product intent

The left side of every rendered HTML document in a Foundation-based repo should answer: "Where am I in this repo's documents, and what nearby documents can I open?" It should behave like a document tree for the repo, not like a curated product taxonomy and not like a spec-system index.

The desired model is simple: if durable HTML documents are arranged in folders in the repo, that arrangement appears in the navigation. In Foundation, a page under docs/specs/templates/ appears under that folder path and a page under docs/principles/ appears under that folder path. In a target repo, a page under that repo's own docs/, knowledge-base/, or configured HTML docs root appears under its own local path.

The feature has two consumers. Humans need the visible folder tree to navigate rendered documents. Agents need the spec registry to resolve behavior-bearing specs by ID, capability, job, process, path ownership, implementation reference, and coverage. Those two systems must coexist: the sidebar organizes documents by file location; the spec registry organizes specs by durable semantic metadata.

Out of scope

  • This spec does not replace or weaken the spec registry, spec metadata schema, spec authoring process, or code-path ownership semantics.
  • This spec does not define a marketing site, search engine, content recommendation system, or hand-authored table of contents.
  • This spec does not require implementation by a runtime app. A generated static JavaScript file is acceptable when its generated data preserves the folder tree.
Separation rule: spec pages are HTML documents for this feature and semantic contracts for the spec registry. Their spec IDs, types, parent/child graph, capability links, job links, process links, path ownership, and coverage remain critical for agents, but they must not decide which folder or category a document appears under in the visual document tree.

Interface journey

  1. Foundation setup: a repo is connected to Foundation through the target-repo setup flow. The setup creates or verifies the local assets, commands, and scaffolding needed for that repo's own durable HTML docs to render the shared navigation.
  2. Entry: a reader opens any durable HTML document in Foundation or a Foundation-based target repo that participates in the shared renderer.
  3. Orientation: the left sidebar shows a document-tree region and a page-local section region. The current document is visibly selected inside its actual folder path.
  4. Collapse: the reader can use the far-left sidebar control to collapse or restore the document navigation without leaving the document. The same control behavior applies to generated specs, graph canvases, and target-repo HTML docs that load the shared navigation scripts.
  5. Folder navigation: the reader expands or collapses folders. Opening a folder reveals child folders and HTML files in deterministic path order.
  6. Document navigation: selecting a document navigates to that document. On the destination page, the same document-tree model appears, with the destination document selected and ancestor folders open.
  7. Page-local navigation: section links for the current document remain separate from the repo document tree so the reader can distinguish "move within this page" from "open another page."
  8. Agent lookup: if the task is to find the spec behind a capability, job, process, implementation path, or coverage item, the agent uses the repo's spec registry. The sidebar is only visual document navigation and must not be treated as the semantic spec graph.
  9. Exit: the reader can stop on any document with enough sidebar context to know where the document lives in the repo.

Foundation-based repo setup

When a repo is connected to Foundation, HTML docs navigation becomes part of the repo setup contract. The target repo keeps its own product specs, project docs, knowledge, tests, ADRs, implementation paths, and generated navigation data. Foundation supplies the shared process, generator behavior, renderer behavior, validators, and setup guidance.

Edge cases

  • Root-level HTML files: show them directly under the docs root rather than inventing a topical category.
  • Nested folders: preserve nesting deeply enough that moving a file changes the visible path after regeneration.
  • Duplicate filenames or titles: rely on folder position for disambiguation and keep link targets exact.
  • Moved documents: a path move is a navigation event. Regenerating the map should remove the old link and add the new link.
  • Broken generated data: a malformed generated map should fail checks before handoff; rendered pages should degrade without hiding content.
  • Pages without shared chrome: either opt them into the shared navigation or exclude them from the generated tree with an explicit reason.
  • Target repo without docs yet: show a valid empty state and keep spec-registry setup checks separate.
  • Foundation revision changes: target repos using pinned Foundation behavior should have an explicit update path so docs navigation changes do not silently alter generated output.
  • Large document trees: collapsed folders, stable indentation, and selected-path expansion keep the sidebar scannable without flattening the tree.
