Foundation Backfill Surface / Function Map Test Spec

This test layer defines the deterministic and semantic gates that prove Foundation Backfill Surface / Function Map can hand off safely.

Status: active
Updated: 2026-05-28
Parent: foundation.backfill-surface-registry.descriptive

The tests treat the layer as a reproducible unit: scope is checked mechanically, structure is checked mechanically, semantic quality is judged through eval receipts, and the HTML summary is a derived review surface.

Acceptance mapping

IDConditionMust prove
AC-1Canonical artifacts exist.surface-registry-YYYYMMDD-NN.jsonl, surface-registry-check-YYYYMMDD-NN.json, and surface-registry-eval-YYYYMMDD-NN.jsonl exist and share the active run ID.
AC-2Upstream references resolve and stay in scope.Every row cites valid Surface / Function Map-eligible upstream IDs from Artifact Inventory or records a blocker; inert artifacts are skipped instead of marked.
AC-3State model is valid.Rows only use declared states: pending, mapped, needs-evidence, ready-for-capability.
AC-4Checker blocks structural drift.Missing rows, duplicate IDs, invalid enums, missing required fields, stale upstream references, and report drift fail.
AC-5Eval blocks weak semantics.Vague, over-broad, unsupported, missing or generic full-file-read evidence, or under-evidenced rows produce revision targets.
AC-6Handoff is explicit.all Surface / Function Map-eligible file rows are resolved to surfaces or explicit support/review classifications; skipped Artifact Inventory rows are counted outside scope; checker passes; eval receipt and HTML summary exist with zero revision targets; report names next layer as Capability Map only after revision targets are resolved.
AC-6AInert files do not enter the Surface / Function Map loop.assets, generated artifacts, test evidence, fixtures, lockfiles, archive/delete-candidate docs, screenshots, images, PPTX exports, spreadsheets, RTF/CSV samples, and support-only docs are not pending work for the Surface / Function Map and cannot be marked by the fill command.
AC-6BThe fill loop cannot be replaced by generated or batch rows.the fill command rejects generated surface-row files, all-file fill modes, and batch-size options; accepted rows must arrive inline for one file after the agent reads that complete file.
AC-7Changed upstream rows invalidate dependent surfaces.Refresh preserves unchanged surfaces and returns changed/new upstream file rows to pending for one-file read and mark revision.
AC-8Calibration is recorded without pretending gold status.Sandia Oil eval receipts name calibration-not-gold until human-authored expected rows exist.

Commands or evidence required

  • npm run foundation:surface-registry:init -- --repo /path/to/repo --run-id YYYYMMDD-NN
  • npm run foundation:surface-registry:fill -- --repo /path/to/repo --run-id YYYYMMDD-NN --next
  • npm run foundation:surface-registry:fill -- --repo /path/to/repo --run-id YYYYMMDD-NN --path repo/relative/file --surfaces-json '[...]'
  • npm run foundation:surface-registry:check -- --repo /path/to/repo --run-id YYYYMMDD-NN --phase batch|handoff
  • npm run foundation:surface-registry:eval -- --repo /path/to/repo --run-id YYYYMMDD-NN
  • npm run foundation:surface-registry:refresh -- --repo /path/to/repo --run-id YYYYMMDD-NN
  • npm run foundation:surface-registry:report -- --repo /path/to/repo --run-id YYYYMMDD-NN --report docs/specs/backfill/review-report-YYYYMMDD-NN.html

Eval contract

samples surface rows against Artifact Inventory rows to judge whether exposed and dependent surfaces were missed, over-merged, mislabeled, unsupported by a complete file read, too vague to feed Capability Map, or detached from evidence. The canonical output is JSONL with one receipt per evaluated row or finding. The HTML summary is generated from JSONL so humans can read counts, failures, revision targets, and examples.

  • Every blocking finding names the row ID, evidence source, failure category, and revision target.
  • Rows with severe uncertainty must remain non-terminal or carry a named human decision.
  • Calibration against Sandia Oil is required before using the layer as a large-repo gate; the receipt must mark it as calibration evidence rather than a gold benchmark until expected rows are human-authored.

Gaps and follow-ups

  • Implementation commands now exist in Foundation; future schema or rubric changes must update tests with fixtures before use on another pilot.
  • Sandia Oil starts as calibration evidence, not a perfect gold fixture.
  • Schema or rubric changes must update descriptive, technical, and test specs together.
