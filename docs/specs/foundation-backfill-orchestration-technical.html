Foundation Backfill Orchestration Technical Spec

The backfill orchestrator is a resumable workflow that delegates each layer of repo intent extraction to focused skills and only advances a slice after adequacy review and strict evaluation.

Status: active
Updated: 2026-05-21
Canonical section: #required-depth

Required depth

The backfill system extracts intended behavior from repo evidence. Code is the strongest available evidence, but the durable output is a spec graph that states what the system is meant to do for users, operators, and future implementers.

The orchestrated output is intent-rigid and architecture-flexible. Descriptive specs carry architecture-agnostic intent. Technical specs carry required contracts, current implementation evidence, justified architecture constraints, and implementation latitude.

Backfill completion depends on coverage, adequacy, and strict evaluation. A complete run has a durable queue where every relevant inventory item is mapped to an acceptable slice, intentionally covered by a parent section, blocked by a named human decision, or explicitly marked out of scope.

Internal sequence

  1. Connect: use Foundation setup and doctor to confirm the target repo can consume Foundation.
  2. Inventory: run backfill-repo-inventory to build or refresh the coverage ledger.
  3. Queue: create or refresh the embedded durable queue and choose the next slice awaiting owner specs, child behavior coverage, revision, or strict evaluation.
  4. Observe: append JSONL run-log events when phases start, finish, checkpoint, validate, evaluate, revise, block, or hand off.
  5. Extract flows: run backfill-user-flow-extraction for user-facing inventory items.
  6. Draft descriptive: run backfill-descriptive-spec-author and create or update draft descriptive specs.
  7. Render UX: run backfill-rendered-ux-spec when the descriptive slice has visual or UX-visible behavior.
  8. Draft technical: run backfill-technical-spec-author and create or update draft technical specs that classify required contracts, current evidence, architecture constraints, and implementation latitude.
  9. Review adequacy: run backfill-spec-adequacy-review, revise layers that need more detail, and mark the slice ready for evaluator scoring after it passes.
  10. Evaluate slice: run evaluate-backfill-specs, revise category gaps below threshold, and re-evaluate until the slice is acceptable or blocked by a named human decision.
  11. Validate: check the durable queue, check the execution run log, regenerate registry, and run spec checks.
  12. Continue: update the report and start the next queued slice until inventory coverage is complete.
  13. Evaluate graph: run evaluate-backfill-specs against the completed graph, golden example, and rubric before final handoff.

Timing, concurrency, and ordering

Backfill runs are allowed to span multiple sessions. Ordering is enforced by durable report state.

  • Resume: read the report, inventory, top-level spec, current slice specs, and relevant skill before continuing.
  • Observe: append run-log checkpoint, start, complete, validation, evaluation, and handoff events as phases progress.
  • Revision loop: adequacy review or strict evaluation may send the slice back to inventory, flow extraction, descriptive authoring, rendered UX, or technical authoring before the next slice begins.
  • Validation cadence: run spec registry/check after each acceptable slice or grouped set of tightly related child specs.

Boundaries and abstraction level

  • Orchestrator: sequencing, report state, skill selection, validation, and completion.
  • Inventory skill: evidence discovery and coverage ledger.
  • User-flow skill: user-centered journeys and flow granularity.
  • Authoring skills: descriptive intent, rendered UX, and technical contracts.
  • Review skill: adequacy gate and revision loop.

Migrations and ADR links

This change splits workflow instructions into skills and adds a technical orchestration contract. Future work may add deterministic report checkers or inventory parsers.
