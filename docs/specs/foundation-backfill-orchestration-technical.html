Foundation Backfill Orchestration Technical Spec

The backfill orchestrator is a resumable workflow that delegates each layer of repo intent extraction to focused skills and only advances a slice after adequacy review.

Status: active
Updated: 2026-05-20
Canonical section: #required-depth

Required depth

The backfill system extracts intended behavior from repo evidence. Code is the strongest available evidence, but the durable output is a spec graph that states what the system is meant to do for users, operators, and future implementers.

The orchestrated output is intent-rigid and architecture-flexible. Descriptive specs carry architecture-agnostic intent. Technical specs carry required contracts, current implementation evidence, justified architecture constraints, and implementation latitude.

Backfill completion depends on coverage and adequacy. A complete run has a coverage ledger where every relevant inventory item is mapped to an adequate spec, intentionally covered by a parent section, or explicitly marked out of scope.

Internal sequence

  1. Connect: use Foundation setup and doctor to confirm the target repo can consume Foundation.
  2. Inventory: run backfill-repo-inventory to build or refresh the coverage ledger.
  3. Queue: choose the next slice from inventory rows awaiting owner specs, child behavior coverage, or adequacy review.
  4. Extract flows: run backfill-user-flow-extraction for user-facing inventory items.
  5. Draft descriptive: run backfill-descriptive-spec-author and create or update draft descriptive specs.
  6. Render UX: run backfill-rendered-ux-spec when the descriptive slice has visual or UX-visible behavior.
  7. Draft technical: run backfill-technical-spec-author and create or update draft technical specs that classify required contracts, current evidence, architecture constraints, and implementation latitude.
  8. Review adequacy: run backfill-spec-adequacy-review, revise layers that need more detail, and mark the slice adequacy-reviewed after it passes.
  9. Validate: regenerate registry and run spec checks.
  10. Continue: update the report and start the next unreviewed slice until inventory coverage is complete.
  11. Evaluate: run evaluate-backfill-specs against the completed graph, golden example, and rubric before final handoff.

Timing, concurrency, and ordering

Backfill runs are allowed to span multiple sessions. Ordering is enforced by durable report state.

  • Resume: read the report, inventory, top-level spec, current slice specs, and relevant skill before continuing.
  • Revision loop: adequacy review may send the slice back to inventory, flow extraction, descriptive authoring, rendered UX, or technical authoring before the next slice begins.
  • Validation cadence: run spec registry/check after each drafted slice or grouped set of tightly related child specs.

Boundaries and abstraction level

  • Orchestrator: sequencing, report state, skill selection, validation, and completion.
  • Inventory skill: evidence discovery and coverage ledger.
  • User-flow skill: user-centered journeys and flow granularity.
  • Authoring skills: descriptive intent, rendered UX, and technical contracts.
  • Review skill: adequacy gate and revision loop.

Migrations and ADR links
