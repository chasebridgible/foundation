Foundation Backfill Orchestration Technical Spec

The backfill orchestrator is a resumable workflow that delegates each layer of repo intent extraction to focused skills and only advances a slice after Review Spec Adequacy and strict evaluation.

Status: active
Updated: 2026-05-29
Canonical section: #required-depth

Required depth

The backfill system extracts intended behavior from repo evidence. Code is the strongest available evidence, but the durable output is a spec graph that states what the system is meant to do for users, operators, and future implementers.

The orchestrated output is intent-rigid and architecture-flexible. Descriptive specs carry architecture-agnostic intent. Technical specs carry required contracts, current implementation evidence, justified architecture constraints, and implementation latitude.

Backfill completion depends on capability coverage, adequacy, and strict evaluation. A complete run has a Capability Map where every relevant behavior-bearing capability is acceptable, intentionally parent-owned with a precise reason, blocked by a named human decision, or explicitly marked out of scope. The Job / Spec Queue is the execution order for that matrix.

Internal sequence

  1. Connect: use Foundation setup and doctor to confirm the target repo can consume Foundation.
  2. Inventory Artifacts: create or refresh the manifest and Artifact Inventory with one row per repo-owned file. This V1 layer includes steady-state changed-file refresh, not only bootstrap backfill.
  3. Map Surfaces: convert eligible mapped file rows into evidence surfaces: routes, screens, APIs, commands, jobs, tables, workflows, infra resources, active behavior docs, and external dependencies.
  4. Map Capabilities: convert reviewed ready surfaces into capability rows using actor, intended outcome, domain object, actions, states, rules, experience, backing contracts, failure/recovery, and evidence.
  5. Define Spec Jobs: split broad capability rows before closure and create a Job / Spec Queue with exact slice scope, owner layer, exit criteria, status, score, blockers, evidence rows, and next action.
  6. Gather Context: gather only the files, surfaces, capabilities, tests, docs, schemas, and snippets needed for the current queued slice.
  7. Observe: append JSONL run-log events when phases start, finish, checkpoint, validate, evaluate, revise, block, or hand off.
  8. Map Processes: map user and operator processes, actions, state transitions, permissions, validations, empty/loading/error states, recovery behavior, and visible/operator behavior.
  9. Author Specs: create or update architecture-agnostic job specs from the queued slice, Context Pack, and process/action rows.
  10. Render UX: run backfill-rendered-ux-spec when the job slice has visual or UX-visible behavior.
  11. Author Technical Specs: create or update contract-first technical specs that classify required contracts, current evidence, architecture constraints, and implementation latitude.
  12. Evaluate Job Slices: run the hard loop for each queued slice: check, evaluate, revise category gaps, re-evaluate, and mark acceptable only after the gate passes or a named human decision blocks progress.
  13. Validate: check layer artifacts, check the Job / Spec Queue, check the execution run log, regenerate the spec registry, and run spec checks.
  14. Continue: update the report and start the next queued slice until every capability is acceptable, parent-owned with a precise reason, blocked by a named human decision, or out of scope.
  15. Evaluate System Coherence: evaluate the completed spec graph for rebuild-grade intent preservation before final handoff.
  16. Handoff: produce the final report with coverage, artifact paths, ready/draft specs, blockers, uncertainties, stale/conflicting evidence, and next actions.

Timing, concurrency, and ordering

Backfill runs are allowed to span multiple sessions. Ordering is enforced by durable report state.

  • Resume: read the report, inventory, top-level spec, current slice specs, and relevant skill before continuing.
  • Observe: append run-log checkpoint, start, complete, validation, evaluation, and handoff events as phases progress.
  • Revision loop: Review Spec Adequacy or strict evaluation may send the slice back to inventory, Process / Action Map, job authoring, rendered UX, or technical authoring before the next slice begins.
  • Validation cadence: run spec registry/check after each acceptable slice or grouped set of tightly related child specs.

Boundaries and abstraction level

  • Orchestrator: sequencing, report state, skill selection, validation, and completion.
  • Inventory skill: evidence discovery and coverage ledger.
  • Process / Action Map skill: actor-centered journeys, decisions, actions, and state granularity.
  • Authoring skills: job intent, rendered UX, and technical contracts.
  • Review skill: adequacy gate and revision loop.

Migrations and ADR links
