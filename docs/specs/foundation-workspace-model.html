Foundation Workspace Model

Foundation is the canonical development substrate for software work across local repositories. It must be easy for an agent to discover, safe for a project repo to consume, and reproducible in CI without copying shared rules into every repo by hand.

Status: active
Updated: 2026-05-20
Canonical section: #product-intent

Foundation workspace model

A user should be able to work in any repository under a local workspace and have Codex reliably apply the shared Foundation process while keeping project-specific truth inside the target repo.

Intent

Foundation should behave like the software-development operating system for the workspace: shared principles, spec workflow, skills, validators, and CI patterns live in one canonical repository, while each product repo keeps its own specs, knowledge, implementation paths, tests, and project-specific agent notes.

The model must prevent drift. Updating the Foundation substrate should improve future work across repos without requiring the user to manually copy common docs, skills, or rules into every project after each change.

Out of scope

This spec does not define the full technical implementation of the installer, sync command, global Codex configuration, or target-repo CI. Those need technical and test specs after this descriptive model is accepted.

This spec also does not define Tribal Innovations, MatchCoach, Sandia Oil, or any other product repo's feature specs. Those specs belong in the target repo and should reference Foundation only for shared process.

Interface journey

  1. Canonical setup: the workspace has a known Foundation repo path. The user's global Codex instructions point agents to that canonical location before repo-specific work begins.
  2. Project entry: when a user opens a product repo, Codex reads the global Foundation pointer, reads Foundation's canonical AGENTS.md, then reads the target repo's local AGENTS.md if present.
  3. Task routing: shared process questions route through Foundation skills. Product-specific questions route into the target repo's specs, knowledge base, tests, and code.
  4. Spec creation: if the user asks to create a user-facing spec, Codex uses Foundation's descriptive spec interview workflow, then writes the resulting project spec into the target repo unless the spec is itself about Foundation.
  5. Foundation improvement: if work reveals a reusable process improvement, Codex updates Foundation, not the product repo, and leaves the target repo with only the project-specific effect.
  6. Validation: before handoff, local checks run in the target repo and Foundation-backed checks verify the target repo's specs and substrate shape.

Edge cases and recovery paths

  • Agent starts inside a product repo: global Codex instructions must still identify the canonical Foundation path, then the local repo adapter supplies only project-specific additions.
  • Agent starts inside the workspace parent: Foundation is not discovered by sibling scanning alone; the global pointer or an explicit workspace config must name it.
  • Target repo has no adapter: Codex uses Foundation rules and asks where project specs, commands, and knowledge should live before creating durable artifacts.
  • Local adapter conflicts with Foundation: the conflict should be treated as a substrate issue. Shared-process changes belong in Foundation; local exceptions must be explicit and narrow.
  • Foundation changes while target work is active: the current task should finish against the loaded/pinned process unless the user asks to refresh Foundation mid-task.
  • CI cannot reach the local Foundation path: CI checks out the pinned Foundation revision named by the target repo workflow instead of relying on the developer's filesystem layout.

Hidden mechanics as UX contracts

Hidden mechanicUX contractFailure promise
AGENTS.md layeringGlobal Foundation instructions load before repo-local instructions; repo-local rules refine, not replace, the shared process.If the agent cannot confirm the load path, it states that and reads the relevant files manually before continuing.
Foundation versioningLocal work uses the canonical Foundation repo; CI checks out the pinned Foundation revision named by the target repo.If Foundation and CI disagree, the mismatch is reported as a version issue, not treated as a product bug.
Skill routingShared workflows are invoked by intent, not by copy-pasted prompts.If a Foundation skill is unavailable from a target repo, the agent reports the missing discovery path and falls back by reading the source skill.
Substrate ownershipEvery durable artifact has one home: Foundation for shared process, target repos for product truth.If an artifact could belong in either place, the agent asks or records an explicit assumption before writing it.
