Business Intake Fill Loop

Turn source artifacts into reviewed business graph substrate: source registry rows, intake answer rows, capability and job signals, promoted HTML specs, and a derived visible business canvas.

Status: draft
Updated: 2026-06-01
Canonical section: #job-intent

Job intent

The Business Intake Fill Loop gives agents a repeatable way to convert client artifacts into source-backed operating knowledge. The job starts with raw sources and ends with a target repo that can show a business as capabilities, jobs, actors, processes, tools, evidence, metrics, and gaps.

The actor may be a human operator, an agent, or a human-agent pair. The output is not a narrative summary; it is reviewed substrate that can be inspected by humans and parsed by graph tooling.

Capability supported

This job supports the Knowledge and Data Layer capability to turn intake, source files, and operating observations into durable substrate. It also supports the Visible Business Graph capability by making graph-compatible specs the promotion target.

Current handling

Foundation has a business intake workspace and visible graph tooling, but intake execution can still happen ad hoc: sources may be summarized without stable source IDs, capabilities may be inferred without answer status, and graph gaps may not be preserved as first-class outputs.

Proposed handling

Agents use `skills/business-intake-fill-loop/SKILL.md` as the repeatable workflow. Each source is registered, each material answer receives a status and source refs, signals are extracted into capabilities/jobs/processes/actors/tools/evidence/metrics/gaps, and reviewed signals are promoted into target-owned specs with `graph-metadata`.

Process

  1. Prepare target repo: connect or create the target repo, install Foundation adapter files, and confirm target docs/spec paths.
  2. Register sources: copy or reference each artifact and create stable source IDs with owner, date, trust, freshness, access, and related lanes.
  3. Extract answers: fill answer rows for business intake lanes and mark status as unanswered, partial, answered, conflict, stale, or reviewed.
  4. Extract signals: identify candidate capabilities, jobs, processes, actors, tools, evidence, metrics, risks, gaps, and decision rules.
  5. Promote substrate: author target system, capability, job, technical, and eval specs only from reviewed signals and explicit gaps.
  6. Build graph: run graph check/build/render commands against the target repo and treat graph/canvas files as derived artifacts.
  7. Evaluate: compare graph nodes, edges, source refs, and canvas usefulness against the source registry and expected fixture where present.
  8. Revise: route weak or unsupported claims back to answer rows, source registry, specs, or gaps.

Context, tools, and interfaces

  • docs/fundamentals/business-intake.html defines intake lanes, statuses, source registry fields, signal types, and promotion targets.
  • skills/business-intake-fill-loop/SKILL.md defines the procedural workflow agents follow.
  • Target repos own client facts, source copies, source registries, HTML specs, graph fixtures, and generated canvases.
  • Foundation graph commands validate, build, render, and evaluate the promoted substrate.

Evidence and evaluation

Acceptable evidence includes a target source registry, answer rows with statuses and source refs, graph-compatible target specs, passing graph check, generated graph JSON, generated canvas HTML, and an eval result with no unsupported required nodes or edges.

Revision

If a pilot exposes missing workflow steps, update this spec, the technical spec, the eval spec, and the skill together. Shared process improvements remain in Foundation; client-specific findings remain in the target repo.
