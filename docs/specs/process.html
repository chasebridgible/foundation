Spec maintenance process

The process keeps specs, evidence, evals, technical work, and implementation moving together. Plans are temporary reasoning unless they are converted into system, capability, job, technical, eval, ADR, test, or board/session artifacts.

Status: active
Updated: 2026-06-03
Parent: foundation.author-durable-specs.capability

Workflow contract

The durable loop is spec -> test -> prompt -> execute. The steps are the contract; the workflow skills that run inside them can change.

StepPurposeDurable output
SpecState what system, capability, or job should exist; who or what it serves; what process runs; what rules govern it; and where intent lives.System spec, capability spec, or job spec.
EvalConvert intent into a verification contract before implementation claims success.Eval spec plus code-level tests, business checks, manual evidence, or planned gaps.
PromptLoad only the relevant specs, tests, rules, skills, and code paths.Run prompt or board/session record, not a stale plan file.
ExecuteChange code, tests, and specs together, then produce evidence.Diff, passing commands, screenshots/traces where useful, passing npm run spec:check.
Recovery rule: when a step fails, correct the smallest upstream artifact, usually the spec, then regenerate downstream tests and prompts.

Reading order

Progressive disclosure is mandatory. An agent should not read every spec in the repo when one feature section is enough.

  1. Read the registry and identify candidate specs by ID, path, title, tags, or board card.
  2. Load the relevant job spec section first. Job intent outranks implementation convenience. If the work is broad, load the parent capability or system spec first.
  3. Load the eval spec acceptance mapping before editing code. The eval spec tells the agent what will count as done.
  4. Load the technical spec for every software, data, automation, infrastructure, or integration task. For small tasks, expect a compact internal contract; for complex tasks, expect deeper sequencing, data, timing, concurrency, failure, and boundary detail.
  5. Load direct owned paths from the lowest-level relevant spec. Avoid walking every path in parent specs.

Spec choice

Choose the spec type through Spec Selection before authoring begins. Do not create several specs when one job spec can carry the job, its SOP/checklist/process, and its evidence needs clearly.

Use this specWhen the intent isExample
System specThe whole operating system, product, repo, or durable subsystem needs a root contract."Agents choose the right spec type across Foundation."
Capability specThe system must reliably produce a class of outcome under real constraints."Spec requests are routed to the correct spec type."
Job specA named body of work is required for a capability to exist, operate, or improve."Classify one incoming request, select the template, and name required eval coverage."
Technical specSoftware, data, automation, infrastructure, or integration implementation details matter."Implement a scaffold command that creates the selected spec and metadata."
Eval specEvidence needs an explicit judgment contract before the system can accept the work."Given ten request examples, the agent chooses system, capability, job, technical, or eval correctly."

A capability usually has multiple jobs. A job usually has a process. A technical spec is added when implementation details matter. An eval spec is added when the work needs a durable definition of what evidence will prove it worked.

Routing rule: generic spec requests must produce a Spec Selection Result first. Then the selected type-specific authoring skill writes the spec; authoring skills do not choose their own spec type.

Spec authoring

When a human and agent create or revise a spec together, the HTML spec is the authored source. The generated registry is only the index.

  1. Search the registry first. For Foundation-owned work, load the Foundation capability map and name the capability plus owning job before adding a durable spec, skill, template, validator, or workflow.
  2. Before adding or changing a Foundation skill, answer: which Foundation capability does this improve, and is there already a job spec for this work? A skill must have an owning capability, owning job spec, technical spec when it defines scripts or data contracts, eval spec when quality judgment matters, and tests or checks when it mutates repo state.
  3. Update an existing spec when it already owns the behavior; create a new spec only when the behavior needs its own durable contract.
  4. Copy the closest HTML template: system for whole-system intent, capability for a reliable outcome, job for the work contract and its process, technical for implementation behavior, or eval for verification.
  5. Fill spec-metadata before relying on the prose. Use a stable dotted id, accurate title, valid type, status, lastUpdated, reviewCadence, and confidence. Do not add document-level owner metadata.
  6. Fill graph-metadata before handoff. Every spec must expose graph nodes and edges that let Foundation render systems, capabilities, jobs, processes, actors, tools, evidence, metrics, gaps, technical contracts, evals, templates, and indexes. Capability specs must connect child jobs with realized-by edges, and job specs must connect back to their owning capability with supports edges.
  7. Set graph fields deliberately: parent, children, and relatedSpecs. Prefer linking to child specs over copying their leaf paths into a parent.
  8. Map paths deliberately: ownedPaths for files this spec directly owns; implementationPaths for non-owning references that help navigation.
  9. Map verification deliberately: coverage entries should point to tests or evidence and use mapsTo section references for the behavior they verify. Planned or gap coverage is acceptable when it is explicit.
  10. Keep the cheap HTML tags aligned with metadata: spec:id, spec:type, spec:status, and spec:last-updated.
  11. Mark exactly one entry section with data-spec-canonical="true". The registry generator uses it for canonicalSection.
  12. Update visible links, path lists, and coverage tables so the human-readable prose agrees with the metadata.
  13. Run npm run spec:registry, then npm run spec:check, then the relevant npm run foundation:visible-business-graph:check -- --repo <repo>. If Foundation capability, job, or skill ownership changed, also run npm run foundation:self-map:check. A human review still has to confirm the mappings are semantically true.
Scaffold: use npm run spec:new -- --type job --id product.feature.job --title "Feature Job Spec" --out docs/specs/features/feature-job.html to create a valid HTML starter, then fill the prose and metadata deliberately.

Risk tiers

Spec depth scales with behavioral risk. The goal is to avoid both skipped rigor and low-value spec churn.

TierChange typeSpec obligationVerification obligation
T0Formatting, typo, comment, or navigation-only edits that do not change product, test, or implementation behavior.No spec update unless the edited doc is itself inaccurate.Run the narrowest relevant check; npm run spec:check when spec files or harness files change.
T1Spec metadata, graph metadata, path, registry, CI, or harness maintenance.Update canonical spec-metadata, graph-metadata, or process docs, then regenerate the registry when metadata changed.npm run spec:registry when metadata changed, then npm run spec:check and graph check when graph metadata changed.
T2Small bugfix or refactor with no intended user-visible behavior change.Compact technical spec note or update to the owning technical section; job spec usually unchanged.Targeted unit/static/manual evidence, plus explicit eval gap if no automated signal is worth adding.
T3User-visible or operator-visible behavior change inside an existing job boundary.Update job, technical, and eval spec sections that define the behavior and done signal.Mapped coverage entry or explicit gap in the eval spec; run the relevant evidence command or review.
T4New capability, new job, new flow, or meaningful job split.Create or update capability, job, technical, and eval specs. Prefer child specs over broad parent path ownership.Acceptance mapping plus planned, existing, or gap coverage for the primary journey and main technical or operating risks.
T5Architecture, data model, security, concurrency, migration, or cross-feature contract change.Deep technical spec section, related ADR when the decision is non-obvious, and all affected parent/child links.Integration/e2e/manual evidence appropriate to the risk, with explicit rollback or follow-up gaps.

If the tier is ambiguous, choose the higher tier until the spec or tests make the risk smaller.

Semantic review

npm run spec:check proves that metadata is valid and references resolve. It does not prove the metadata tells the truth.

FieldReview questionFailure smell
relatedSpecsDo these relationships explain the actual dependency, validation, parent, child, or sibling relationship?Broad links added only so an agent can find more context.
ownedPathsDoes this spec directly own each listed file?Parent specs owning every leaf file, or duplicate direct ownership across siblings.
implementationPathsAre these useful non-owning references, not hidden ownership claims?Reference paths that should really be direct ownership in a child spec.
coverage.mapsToDoes the evidence actually verify the mapped section?Coverage linked to a nearby section because the right section was missing or unclear.
Visible proseDoes the human-readable path list, coverage table, and behavior text agree with metadata?Registry/check passes while prose says something different.
graph-metadataDo graph nodes and edges accurately expose what this spec represents and how it connects?Canvas renders an orphan, wrong capability/job link, or unsupported certainty.
Risk tierDoes the depth of system, capability, job, technical, and eval work match the tier?Heavy spec edits for T0/T1 work, or shallow notes for T4/T5 work.

Final answers for spec work should say whether this semantic review found any unresolved mapping risk.

A diff that changes behavior without changing the relevant specs and eval specs is incomplete by default.

Blockers and exits

Blocker conditions

  • No job spec exists for the user-visible, operator-visible, or system work being changed.
  • The eval spec cannot name the evidence required to accept the change.
  • No technical spec exists for the work, or the technical spec is too shallow for the implementation risk.
  • Registry links point to missing specs, moved files, or deleted tests.
  • The agent cannot tell whether a path is directly owned by the current feature or merely referenced by a parent spec.

Exit criteria

  • Relevant specs state the current intended behavior.
  • Eval spec maps acceptance conditions to job and technical sections.
  • Code-level tests, manual evidence, or explicit gaps match the eval spec.
  • Risk tier is named or obvious from the diff, and spec depth matches that tier.
  • Semantic review confirms metadata mappings are truthful, not just valid.
  • npm run spec:check proves the generated registry is current and resolves related specs, owned paths, coverage paths, and section references.
  • CI runs npm run spec:check on pull requests and pushes to main; stale generated registry output blocks the change.
  • Final evidence names commands run and unresolved decisions.

What belongs in workflow skills

Workflows are procedures. Specs are source-of-truth contracts. Do not blur them.

Workflow skill fieldWhat it should defineWhat it must not become
TriggersWhen to invoke the workflow: new feature, bug, refactor, review, release.A duplicate product spec.
Required inputsSpec IDs, board card, repo path, constraints, environment access.A hidden source of intent not reflected in specs.
Reading orderWhich registry fields and spec sections to load first.A requirement to read everything.
Context filesAGENTS/CLAUDE files, package scripts, local conventions, relevant specs.A dumping ground for stale plans.
Update obligationsWhich specs, tests, registry entries, ADRs, or board records must change.A substitute for changing those artifacts.
Exit criteriaEvidence required to stop: tests, screenshots, traces, review notes.A self-certifying "looks done" statement.
Substrate updatesWhen to extract a new template, capability, skill, or convention.Spec content that should live in the spec system.
