Spec maintenance process

The process keeps specs, tests, and code moving together. Plans are temporary reasoning unless they are converted into specs, ADRs, tests, or board/session history.

Status: active
Updated: 2026-05-11
Parent: foundation.spec-system

Workflow contract

The durable loop is spec -> test -> prompt -> execute. The steps are the contract; the workflow skills that run inside them can change.

StepPurposeDurable output
SpecState what should exist, who it serves, what rules govern it, and where intent lives.Descriptive spec; technical spec scaled to task depth; related ADR if a decision is non-obvious.
TestConvert intent into a verification contract before implementation claims success.Test spec plus code-level tests or planned gaps.
PromptLoad only the relevant specs, tests, rules, skills, and code paths.Run prompt or board/session record, not a stale plan file.
ExecuteChange code, tests, and specs together, then produce evidence.Diff, passing commands, screenshots/traces where useful, passing npm run spec:check.
Recovery rule: when a step fails, correct the smallest upstream artifact, usually the spec, then regenerate downstream tests and prompts.

Reading order

Progressive disclosure is mandatory. An agent should not read every spec in the repo when one feature section is enough.

  1. Read the registry and identify candidate specs by ID, path, title, tags, or board card.
  2. Load the relevant descriptive spec section first. Product intent outranks implementation convenience.
  3. Load the test spec acceptance mapping before editing code. The test spec tells the agent what will count as done.
  4. Load the technical spec for every implementation task. For small tasks, expect a compact internal contract; for complex tasks, expect deeper sequencing, data, timing, concurrency, failure, and boundary detail.
  5. Load direct owned paths from the lowest-level relevant spec. Avoid walking every path in parent specs.

Spec authoring

When a human and agent create or revise a spec together, the HTML spec is the authored source. The generated registry is only the index.

  1. Search the registry first. Update an existing spec when it already owns the behavior; create a new spec only when the behavior needs its own durable contract.
  2. Copy the closest HTML template: descriptive for product intent, technical for internal behavior, or test for verification.
  3. Fill spec-metadata before relying on the prose. Use a stable dotted id, accurate title, valid type, status, lastUpdated, reviewCadence, and confidence. Do not add document-level owner metadata.
  4. Set graph fields deliberately: parent, children, and relatedSpecs. Prefer linking to child specs over copying their leaf paths into a parent.
  5. Map paths deliberately: ownedPaths for files this spec directly owns; implementationPaths for non-owning references that help navigation.
  6. Map verification deliberately: coverage entries should point to tests or evidence and use mapsTo section references for the behavior they verify. Planned or gap coverage is acceptable when it is explicit.
  7. Keep the cheap HTML tags aligned with metadata: spec:id, spec:type, spec:status, and spec:last-updated.
  8. Mark exactly one entry section with data-spec-canonical="true". The registry generator uses it for canonicalSection.
  9. Update visible links, path lists, and coverage tables so the human-readable prose agrees with the metadata.
  10. Run npm run spec:registry, then npm run spec:check. A human review still has to confirm the mappings are semantically true.

A diff that changes behavior without changing the relevant specs and test specs is incomplete by default.

Blockers and exits

Blocker conditions

  • No descriptive spec exists for the user-visible behavior being changed.
  • The test spec cannot name the evidence required to accept the change.
  • No technical spec exists for the work, or the technical spec is too shallow for the implementation risk.
  • Registry links point to missing specs, moved files, or deleted tests.
  • The agent cannot tell whether a path is directly owned by the current feature or merely referenced by a parent spec.

Exit criteria

  • Relevant specs state the current intended behavior.
  • Test spec maps acceptance conditions to descriptive and technical sections.
  • Code-level tests, manual evidence, or explicit gaps match the test spec.
  • npm run spec:check proves the generated registry is current and resolves related specs, owned paths, coverage paths, and section references.
  • CI runs npm run spec:check on pull requests and pushes to main; stale generated registry output blocks the change.
  • Final evidence names commands run and unresolved decisions.

What belongs in workflow skills

Workflows are procedures. Specs are source-of-truth contracts. Do not blur them.

Workflow skill fieldWhat it should defineWhat it must not become
TriggersWhen to invoke the workflow: new feature, bug, refactor, review, release.A duplicate product spec.
Required inputsSpec IDs, board card, repo path, constraints, environment access.A hidden source of intent not reflected in specs.
Reading orderWhich registry fields and spec sections to load first.A requirement to read everything.
Context filesAGENTS/CLAUDE files, package scripts, local conventions, relevant specs.A dumping ground for stale plans.
Update obligationsWhich specs, tests, registry entries, ADRs, or board records must change.A substitute for changing those artifacts.
Exit criteriaEvidence required to stop: tests, screenshots, traces, review notes.A self-certifying "looks done" statement.
Substrate updatesWhen to extract a new template, capability, skill, or convention.Spec content that should live in the spec system.
