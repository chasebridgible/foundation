[Job name] job spec

A job spec is the contract for a named body of work required for a capability to exist, operate, or improve. It is the source of truth for the job's intent, current handling, proposed handling, process, context, evidence, and revision history.

Template status: active
Updated: 2026-05-31
Copy rule: when instantiating this template, replace metadata, visible placeholders, related spec IDs, owned paths, coverage entries, and domain language before using it to drive work.

Intent

State what this job is meant to make true and why the capability needs it.

Trigger

Name when this job starts: user action, business event, scheduled run, incident, request, opportunity, handoff, or evidence signal.

Capability supported

Link the capability spec this job supports. If one job supports multiple capabilities, name the primary capability and explain why the job is not too broad.

Current handling

Describe how the job currently happens, including known gaps. In software, cite current interfaces, files, commands, tests, logs, or users. In business, cite roles, docs, SaaS records, meetings, dashboards, SOPs, customers, or owner knowledge.

  • Current actor: [person, agent, system, vendor, workflow, hybrid].
  • Current path: [plain-language summary].
  • Known gaps: [missing context, weak process, unclear evidence, owner dependency, implementation risk].

Proposed handling

Use this section when the job is being redesigned. Keep the proposal in the same job spec so current and future handling do not become competing documents.

  • Proposed actor: [person, agent, software, vendor, workflow, hybrid, or unchanged].
  • Proposed path: [plain-language summary of the changed handling].
  • Reason for change: [capability gap, evidence signal, cost, risk, quality, speed, reliability].
  • Decision state: proposed / accepted / rejected / superseded.

Process

Define the repeatable path that performs the job. SOPs, checklists, process maps, decision trees, and handoff steps belong here by default. If a long checklist, diagram, or operating aid is useful as a separate file, link it as a view of this job spec instead of making it a competing source of truth.

  1. Prepare: [context, permissions, inputs, source data, examples].
  2. Decide: [decision criteria, branches, escalation rules].
  3. Act: [actions taken in software, business, communication, data, or physical reality].
  4. Check: [quality gate, review, automated test, metric, checklist, acceptance signal].
  5. Handoff or close: [what becomes true and where it is recorded].
View rule: reusable or regulated process detail can be rendered as a linked SOP, checklist, diagram, workflow file, or skill. The job spec still names the job intent, owning capability, current process, evidence, and revision rules.

Revision

Name what changes when this job misses: the job spec, technical spec, eval spec, context pack, source registry, decision rule, actor assignment, tool, or capability spec. When a proposed handling is accepted, promote it to current handling and move the old handling into revision history. Rejected proposals should remain as brief decision notes, not living parallel docs.

  • Revision target: [artifact or system layer].
  • Update trigger: [failed eval, new source, changed process, repeated exception, user/customer outcome].
  • Promotion rule: [what evidence or decision makes proposed handling become current handling].
