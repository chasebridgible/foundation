Backfill File Registry

Before Foundation can infer surfaces, capabilities, or specs from an existing repo, it needs a durable map of every repo-owned file and what that file contributes.

Status: draft
Updated: 2026-05-22
Parent: foundation.backfill-specs.descriptive

Intent

The file registry fill loop turns raw repository files into a complete, structured evidence index. The agent should not rely on memory to understand the repo. It should generate a file manifest, create a registry row for every repo-owned file, inspect bounded batches, update rows, run checks, and resume from the registry until all files are mapped.

This layer is the base of the backfill tree. Surface registries, capability matrices, and specs are derived from it. The target is rebuild-grade evidence: downstream agents should be able to derive surfaces and capabilities from the registry and verify that derivation without rediscovering the whole repo.

Out of scope

This spec does not define capability clustering, descriptive spec authoring, or code cleanup. Those happen after the file registry passes its checks and evals.

Interface journey

  1. Manifest: a deterministic command enumerates repo-owned files from the manifest rule and records path, size, hash, extension, detected language, and source status.
  2. Skeleton: the registry is initialized with one row for every manifest file. Every row starts pending.
  3. Batch selection: the agent reads the registry, picks the next bounded pending batch, and records a run-log start event.
  4. File inspection: the agent reads the batch, identifies each file's role, kind, domain, evidence value, responsibilities, important symbols, related files, and review flags.
  5. Registry update: the agent updates rows to mapped. Review uncertainty is recorded in row fields, not in a separate status.
  6. Check: the deterministic checker proves every manifest file has exactly one current registry row, no registry row exists outside the manifest, mapped rows satisfy the schema, and final handoff has zero pending rows.
  7. Eval: the evaluation layer samples mapped rows against source files and scores accuracy, specificity, behavior-bearing detection, navigability, and large-file handling.
  8. Gold calibration: before using the layer for large-repo backfill or after schema/rubric changes, run the file-registry loop against a small known-answer repo and compare it to a manually authored gold registry.
  9. Resume: after context compaction, interruption, or a new agent session, the agent rereads target AGENTS.md, Foundation AGENTS.md, the active skill, report, registry, and run log, then continues from pending rows.
  10. Handoff: when the registry has zero pending rows, no blocking review flags, passing check/eval evidence, and current gold calibration, the report records the registry path, checker result, eval result, latest run-log sequence, pending count, and next layer. The surface registry layer starts from that recorded state.

Edge cases and recovery paths

  • Committed generated or vendor file: map it as generated/supporting and record why it matters or that it supports tooling.
  • External dependency, cache, or build output: keep it outside the manifest only by deterministic manifest rule.
  • Huge file: include responsibilities or sections so later agents can find the behavior inside it.
  • Binary or media asset: map path, kind, usage evidence if known, and related files that reference it.
  • Stale or misleading document: map the file and flag it as stale/suspicious evidence instead of deleting or ignoring it.
  • Context compaction or handoff: reread the instruction chain, active report, registry, and run log; continue from pending rows without restarting discovery from memory.
  • File changed after mapping: checker marks the row stale and routes it back to the fill loop.
