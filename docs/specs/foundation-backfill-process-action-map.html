Foundation Backfill Process / Action Map

Process / Action Map turns Context Packs into concrete actor processes, actions, states, permissions, rules, empty/loading/error states, recovery behavior, and visible or operator behavior. This layer answers: What behavior, states, rules, and edge cases must specs preserve?

Status: active
Updated: 2026-06-03
Upstream: Context Pack
Next: Author Specs

This layer is a reproducible unit in the backfill substrate. It initializes from the previous layer, fills exactly one Context Pack row at a time, checks deterministic structure, writes row-targeted JSONL eval receipts, derives a readable HTML summary, revises the same row until outstanding, and records handoff state before the next layer begins.

GBrain's source-of-record lesson applies directly: canonical append-friendly artifacts own state; readable HTML summaries and report tables are derived views. Later agents cite row IDs and evidence links instead of relying on inherited conversation memory.

Small-loop journey

  1. Initialize: run foundation:process-action-map:init to create the canonical Process / Action Map artifact from Context Pack handoff output.
  2. Select one row: run foundation:process-action-map:fill -- --next to name the next Context Pack row without writing. The named row is the only valid fill target.
  3. Read one pack: read exactly one Context Pack row and only the evidence needed for that row. Do not inspect many packs to build a classifier or reusable template.
  4. Fill one process row: run foundation:process-action-map:fill with exactly one --pack-id or one --slice-id and exactly one reviewed inline process JSON object for the current --next target. Multi-ID, multi-row, out-of-order target, --all, --batch-size, generated process-file, driver-script, shell-loop, regex-classifier, and template-synthesis shortcuts are rejected.
  5. Check: run foundation:process-action-map:check -- --phase batch for the current row loop and repair structural failures.
  6. Evaluate one row: run foundation:process-action-map:eval with that same --pack-id, --slice-id, or --process-map-id to write a current row receipt. Row-targeted eval requires a current passing check artifact for the latest Process / Action Map fingerprint.
  7. Revise to outstanding: revise that same row until deterministic check passes, row eval has no blocking findings, row eval has no warnings, no revision targets remain, and Author Specs can proceed without rediscovering source for core behavior.
  8. Continue: return to --next only after the current non-pending row has a current outstanding row receipt; run foundation:process-action-map:refresh after Context Pack rows change.
  9. Handoff: after all rows are outstanding or explicitly blocked, run handoff check, full eval, and foundation:process-action-map:report; advance only when the report state names Author Specs or named human decisions block the layer.

Checker and eval contract

MechanismContract
Checkervalidates every user-facing or operator-facing queued slice has process/action rows or explicit non-UI/operator classification; validates process/action rows link back to Context Packs and capability IDs; supports batch-phase checks during the one-row loop and handoff-phase checks before Author Specs. Row eval may not substitute for this check.
Eval receiptscores row-targeted process/action receipts against Context Packs to detect missing states, hidden permissions, unmodeled errors, architecture leakage, missing spec targets, and under-described visible behavior. Eval output is canonical JSONL; HTML is a derived summary for human review.
Sandia Oil calibrationSandia Oil is the intended first real calibration target after this substrate layer is ready to run. Calibration is not required for this layer's Foundation readiness gate.

Handoff gate

all active Context Pack rows have outstanding Process / Action Map rows or explicit blocker detail; every non-pending row has a current outstanding row-level eval receipt; the checker artifact passes for the exact current Process / Action Map fingerprint; eval receipt and HTML summary exist for the exact current fingerprint with zero revision targets and zero warnings; report names next layer as Author Specs.

  • Canonical artifacts exist and share a run ID.
  • All rows are in valid terminal or explicitly blocked states.
  • Checker passes for structure, references, currentness, and required fields.
  • Eval JSONL and HTML summary exist for the current Process / Action Map artifact, with current outstanding row receipts, zero revision targets, zero blocking findings, and zero warnings.
  • Active report records counts, artifact paths, checker/eval status, run-log sequence, and next layer.
