Shift Clock

A shift clock is a reusable process primitive for committed work time: it names the current run, the deadline, the evidence expected during the shift, and the natural exit checks that close the run.

Status: active
Updated: 2026-06-20
Canonical section: #capability-intent

Capability intent

Foundation should make shift clocks available as a clear process primitive for long-running agent work. A shift clock turns committed time into inspectable work: the agent knows the current run, works in bounded units, records evidence, and checks the clock at natural exit or handoff points.

This is a child capability under Long-Running Agent Work. It is intentionally narrower than the full long-running worker job: the clock governs time commitment and evidence, while the owning job or skill governs the domain work.

Current handling

  • Current actors: long-running Codex workers and watchful supervisors.
  • Current path: Foundation currently names the behavior in Agent Principles, the Research specs, and long-running worker/supervisor skills. Some project repos still have legacy timer-named commands and receipts.
  • Known gaps: the concept has been called "timer" or "time gate" in multiple places, which hides the actual operating model: a work shift with accountability, sparse exit checks, and evidence-backed continuation.

Proposed handling

  • Proposed actor: a long-running agent worker, optionally observed by a watchful supervisor.
  • Proposed path: use "shift clock" as the Foundation term; project commands may retain legacy names such as timer:start while their specs describe the behavior as a shift clock.
  • Reason for change: agents treated timers as something to watch or wait through. The shift-clock model makes the intended behavior explicit: work until the shift can legitimately close.
  • Decision state: accepted as Foundation terminology for shift-clocked long-running agent work.

Jobs

No child jobs live under this primitive. Long-running worker and supervisor skills are owned by foundation.long-running-agent-worker.job. Shift Clock supplies the time-bound evidence rule that those skills and project jobs can opt into.

Accountability rule: the durable judge is evidence. The agent earns credit through useful artifacts, current-run receipts, checks, and clear handoff state.

Operating contract

  1. Start shift: create or confirm a goal, run ID, branch/worktree, clock start, deadline, duration, expected artifacts, and current-run receipt path.
  2. Declare artifact contract: name reviewer-facing outputs, internal evidence, rejections, routes, coverage records, and handoff notes so extra work cannot pollute the wrong surface.
  3. Work bounded unit: select one source, row, file, queue item, target, test failure, or focused search family.
  4. Record evidence: write a durable update such as a finding, rejection, no-result, source status, patch, validation receipt, or handoff note.
  5. Score progress: give credit for useful new bounded work, current receipts, artifact quality, and validation evidence.
  6. Check at exit points: when the agent believes it is done, blocked, or ready to hand off, check the shift clock.
  7. Continue on open clock: if the clock is still open, record the exit attempt and continue with another useful bounded unit, adjacent source/search family, audit, validation, or cleanup task.
  8. Close on expired clock: if the clock is expired, finish the current bounded unit, regenerate reports, run checks, and hand off with final clock evidence.

Revision

If shift-clocked work lacks current-run evidence, sparse exit checks, useful work between checks, artifact quality, or resumable handoff state, revise the smallest owning layer: project command, receipt schema, worker skill, supervisor skill, eval, or this capability spec.
