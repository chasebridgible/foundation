Author System Spec Job

This job authors or revises root system specs after Spec Selection classifies the work as whole-system intent.

Status: active
Updated: 2026-06-06
Canonical section: #job-intent

Job intent

This job turns a Spec Selection Result for system-level intent into an HTML-native system spec with one root goal, boundaries, parent capability families, operating loop, evidence model, and revision model.

Capability supported

Supports foundation.author-durable-specs.capability by ensuring whole-system intent is captured before child capabilities, jobs, technical contracts, and evals depend on it.

Current handling

Before this lane, generic spec requests could start in a job or capability template even when the user was changing the root system goal. That let downstream maps drift because the system goal was not settled first.

Proposed handling

The lane starts only from a Spec Selection Result whose primary classification is system. If the request is a feature, workflow, artifact, interface, or implementation detail, the lane rejects it and returns to Spec Selection.

Process

  1. Read the Spec Selection Result, existing system spec if present, Foundation definitions, and system template.
  2. Run the readiness gate: ask only questions that materially change root goal, boundary, capability families, evidence, or revision model; otherwise continue with labeled assumptions.
  3. Confirm there is one root goal and that broader philosophical framing belongs outside the operating goal when needed.
  4. Define system boundaries, actors, parent capability families, operating loop, evidence model, revision model, and open decisions.
  5. Author or revise HTML prose, spec-metadata, graph-metadata, coverage, visible links, and owned paths together.
  6. Run the system-spec eval, revise blocking findings, then run registry and graph checks.

Context, tools, and interfaces

  • skills/author-system-spec/SKILL.md is the agent procedure.
  • docs/specs/templates/system-spec-template.html is the authoring shape.
  • foundation.author-system-spec.eval is the quality gate.

Readiness questions: What is the single root operating goal? What is inside and outside the system boundary? What parent capability families must this system make reliable? How should evidence revise the system over time?

Evidence and evaluation

The authored system spec is acceptable only when it has one root operating goal, coherent capability families, explicit boundaries, evidence and revision rules, graph-compatible metadata, and no unresolved classification mismatch.

Revision

Revise this job when agents create system specs for non-system work, omit the root goal, duplicate child-spec detail into the system spec, or fail to connect the system goal to capabilities and evaluation.
