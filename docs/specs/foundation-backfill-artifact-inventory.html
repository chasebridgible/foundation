Foundation Artifact Inventory

Foundation needs a durable map of every repo-owned file, what that file contributes, which capabilities it supports, which specs define its intent, and what verification protects it.

Status: active
Updated: 2026-05-27
Parent: foundation.backfill-specs.job

Intent

The Artifact Inventory fill loop turns raw repository files into a complete, structured evidence index. The agent should not rely on memory to understand the repo. It should generate a file manifest, create an inventory row for every repo-owned file, inspect bounded batches, update rows, run checks, and resume from the Artifact Inventory until all files are mapped.

The Artifact Inventory is a core Foundation substrate, not only a one-time backfill artifact. V1 includes both bootstrap backfill and steady-state changed-file refresh from the start. Backfill uses it as the bootstrap path when a target repo does not yet have inventory state. Normal build work uses it as the implementation index that keeps files, specs, capabilities, and verification targets connected as the repo changes.

The target is rebuild-grade evidence: downstream agents should be able to ask which files implement a capability, which spec sections define intended behavior, which tests or evals protect the behavior, and which inventory rows are stale or low-confidence without rediscovering the whole repo.

Out of scope

This spec does not define capability clustering, job Author Specs, or code cleanup. It defines the file-level substrate those workflows consume. Backfill, spec workflow, and future build agents may call the same artifact-inventory commands for different lifecycle phases.

Relationship to the Foundation graph

The Artifact Inventory is the implementation-facing companion to the spec registry. The spec registry says where durable intent lives. The Artifact Inventory says where implementation, docs, configs, assets, fixtures, tests, and generated artifacts live, what they do, and which durable intent they support.

Foundation layerQuestion it answersArtifact Inventory relationship
Spec registryWhat behavior is intended, and which spec section owns that intent?File rows link to spec IDs and section IDs when a file implements, documents, configures, verifies, or materially affects that intent.
Capability MapWhat actor outcome does the repo support?Behavior-bearing file rows link to capability IDs after capability discovery exists. During bootstrap, rows may carry unresolved capability hints and review flags.

BackfillHow does an existing repo get adopted into Foundation?Backfill calls the artifact-inventory commands first, then uses mapped rows as the evidence base for surfaces, capabilities, specs, and evals.
Normal build loopHow does ongoing work stay aligned?Changed files invalidate or refresh inventory rows, and the Artifact Inventory routes agents to affected specs, capabilities, and tests before editing continues.
Core invariant: after a repo has both an Artifact Inventory and capability/spec coverage, every behavior-bearing mapped row should link to at least one capability, at least one owning spec section, and at least one verification target or explicit test gap.

Interface journey

  1. Manifest: a deterministic command enumerates repo-owned files from the manifest rule and records path, size, algorithm-prefixed content identity, extension, detected language, and source status.
  2. Skeleton: the Artifact Inventory is initialized with one row for every manifest file. Every row starts pending.
  3. Batch selection: the agent reads the Artifact Inventory, picks the next bounded pending batch, and records a run-log start event.
  4. File inspection: the agent reads the batch, identifies each file's role, kind, domain, evidence value, responsibilities, important symbols, related files, and review flags.
  5. Inventory update: the agent updates rows to mapped. Review uncertainty is recorded in row fields, not in a separate status.
  6. Check: the deterministic checker proves every manifest file has exactly one current inventory row, no inventory row exists outside the manifest, mapped rows satisfy the schema, and final handoff has zero pending rows.
  7. Eval: the evaluation layer samples mapped rows against source files and scores accuracy, specificity, behavior-bearing detection, navigability, and large-file handling.
  8. Sandia Oil calibration: before using the layer for large-repo backfill or after schema/rubric changes, run the Artifact Inventory loop against Sandia Oil while the repo is small enough to inspect. Treat this as calibration evidence until a human-authored expected inventory exists.
  9. Resume: after context compaction, interruption, or a new agent session, the agent rereads target AGENTS.md, Foundation AGENTS.md, the active skill, report, Artifact Inventory, and run log, then continues from pending rows.
  10. Graph linking: once the spec and capability layers exist, behavior-bearing mapped rows are linked to capability IDs, spec section IDs, and verification targets or explicit gaps.
  11. Handoff: when the bootstrap Artifact Inventory has zero pending rows, no blocking review flags, passing check/eval evidence, and current Sandia Oil calibration evidence or a human-authored expected inventory when available, the report records the inventory path, checker result, eval result, latest run-log sequence, pending count, and next layer. The surface/capability layers start from that recorded state.
  12. Steady-state refresh: after bootstrap, normal build workflow refreshes or invalidates inventory rows for changed files and checks graph links before handoff.

Edge cases and recovery paths

  • Committed generated or vendor file: map it as generated/supporting and record why it matters or that it supports tooling.
  • External dependency, cache, or build output: keep it outside the manifest only by deterministic manifest rule.
  • Huge file: include responsibilities or sections so later agents can find the behavior inside it.
  • Binary or media asset: map path, kind, usage evidence if known, and related files that reference it.
  • Stale or misleading document: map the file and flag it as stale/suspicious evidence instead of deleting or ignoring it.
  • Context compaction or handoff: reread the instruction chain, active report, registry, and run log; continue from pending rows without restarting discovery from memory.
  • File changed after mapping: checker marks the row stale and routes it back to the fill loop.
  • Spec or capability changed: graph-link checks identify inventory rows whose spec section, capability ID, or verification target no longer resolves.
  • Current-run registry artifacts: generated manifest, registry, and evaluation files are excluded from the input snapshot that creates them unless a future implementation defines a stable self-entry convention. Prior committed registry artifacts can be mapped as generated/support files in later runs.
