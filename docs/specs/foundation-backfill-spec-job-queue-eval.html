Foundation Backfill Define Spec Jobs Eval Spec

This test layer defines the deterministic and semantic gates that prove Foundation Backfill Define Spec Jobs can hand off safely.

Status: active
Updated: 2026-06-07
Parent: foundation.backfill-spec-job-queue.job

The tests treat the layer as a reproducible unit: structure is checked mechanically, semantic quality is judged through eval receipts, and the HTML summary is a derived review surface.

Eval contract

samples slices against capability rows to judge whether the slice is small enough, aligned to capability identity and splitCriteria, exit criteria are objective, and no hidden behavior is parked inside a parent summary. The canonical output is JSONL with one receipt per evaluated row or finding plus the queue fingerprint that was evaluated. The HTML summary is generated from JSONL so humans can read counts, failures, revision targets, and examples.

  • Every blocking finding names the row ID, evidence source, failure category, and revision target.
  • Semantic alignment failures are revision targets when a child-slice name, scope, included behavior, or rationale points at an unrelated domain or incidental source path.
  • A child/sole capability that appears to need multiple queue slices is a Capability Map split problem, not a queue batching opportunity.
  • A passing eval is not reusable after the queue artifact changes; the checker must require regeneration against the current queue fingerprint.
  • Rows with severe uncertainty must remain non-terminal or carry a named human decision.
  • Calibration against Sandia Oil is required before using the layer as a large-repo gate.

Gaps and follow-ups

  • Implementation commands exist in Foundation; Sandia Oil can now calibrate the layer from Capability Map handoff.
  • Sandia Oil starts as calibration evidence, not a perfect gold fixture.
  • Schema or rubric changes must update job, technical, and eval specs together.
