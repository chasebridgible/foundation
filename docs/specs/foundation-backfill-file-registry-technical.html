Backfill File Registry Technical Spec

The file registry layer is a deterministic-plus-agentic indexing system: commands create complete file rows, and the agent fills semantic meaning in bounded, resumable batches.

Status: draft
Updated: 2026-05-22
Parent: foundation.backfill-file-registry.descriptive

This is a T5-level Foundation workflow because every later backfill layer depends on its completeness. The implementation should prefer scripts for enumeration/checking and agent judgment for semantic row filling and quality evaluation.

Internal sequence

  1. Enumerate: generate docs/specs/backfill/file-manifest-YYYYMMDD-NN.json from repo-owned files. Default source is tracked files plus untracked non-ignored files. Git ignored files are outside the repo-owned set unless explicitly included.
  2. Initialize: create docs/specs/backfill/file-registry-YYYYMMDD-NN.jsonl with one row for every manifest file, mechanical fields populated, semantic fields empty, and status pending.
  3. Select batch: choose pending rows by deterministic order plus priority hints: entry points, routes, schemas, services, jobs, tests, docs, then support files.
  4. Inspect: read a bounded batch and any immediately related files needed to understand each row.
  5. Fill: update semantic fields, status, confidence, related paths, and review flags.
  6. Validate: run the registry checker after batch updates. Batch mode may report remaining pending rows; handoff mode requires zero pending rows.
  7. Evaluate: run a file-registry quality eval on deterministic risk-based samples before advancing to surface registry.
  8. Calibrate: run a small known-answer gold comparison before large-repo use and after schema or rubric changes.
  9. Resume: after context compaction, interruption, or a new agent session, reread target AGENTS.md, Foundation AGENTS.md, the active skill, report, registry, and run log, then continue from pending or stale rows.

Structured meaningMapped rows use structured fields over prose-only summaries; unknown classifications are pending-only.Surface/capability extraction needs queryable evidence.
Batch resumabilityPending rows and batch checkpoints define the next action.Long runs survive compaction, interruption, and handoff.
Handoff gateSurface registry work begins only after zero pending rows, checker pass, quality eval pass, no blocking review flags, and current gold calibration.Bad file maps should not become capability gaps.

Timing, concurrency, and ordering

  • Single-agent baseline: correctness cannot depend on multiple agents. Parallelization may be an optimization only if rows are disjoint and merged through the same checker.
  • Batch size: choose batches small enough for the agent to inspect accurately and update rows before losing context.
  • Change handling: changed files invalidate affected rows and downstream surfaces/capabilities derived from them.

Boundaries and abstraction level

  • Owns: file manifest, file registry schema, fill loop, registry checker, registry quality eval contract.
  • Feeds: surface registry, capability matrix, spec authoring, stale-doc review, cleanup planning.
  • Does not own: capability clustering, final spec graph evaluation, or code/document cleanup.
