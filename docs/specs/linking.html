Spec linking model

The linking model makes each HTML spec both the human-readable contract and the machine-readable source of truth.

Status: active
Updated: 2026-05-11
Registry: index.html

HTML-native metadata

Each spec has machine-readable layers. Agents should read them in this order before reading prose.

LayerExampleUse
Stable file pathfoundation/docs/specs/examples/test-spec-example.htmlFilesystem discovery and durable linking.
HTML meta tags<meta name="spec:id" content="fup.kitchen-queue.test">Cheap parse without loading body content.
Embedded JSON<script type="application/json" id="spec-metadata">Canonical metadata: links, paths, coverage, status, cadence, confidence.
Section attributes<section id="states-and-rules" data-spec-section="states-and-rules">Precise section loading and acceptance mapping.
Canonical section markerdata-spec-canonical="true"One section per spec can define the generated registry entry point.
Link attributes<a data-spec-link="fup.kitchen-queue.test" href="./test-spec-example.html">Human links that still resolve by stable ID.

Human-facing links can use friendly labels. If a page uses wiki-style aliases or natural prose links, it still needs data-spec-link or canonical spec-metadata so agents can resolve them without visual inference.

Multi-level specs

Specs form a graph, not a flat pile. The goal is to avoid duplicating leaf paths in high-level specs.

High-level product spec
Owns product intent, boundaries, and child spec IDs. It should rarely own leaf implementation files.
Low-level feature spec
Owns concrete code, design, and test paths directly touched by the feature.
Test spec
Owns verification paths and maps acceptance conditions back to section IDs in descriptive and technical specs.

A parent should say "kitchen queue behavior is owned by fup.kitchen-queue.descriptive" rather than copying every React component and API route. A low-level spec should name the files it directly owns.

Resolution algorithm

  1. Parse foundation/docs/specs/index.html#spec-registry, the generated index built from each HTML spec's canonical spec-metadata.
  2. If starting from a code path, match exact path in ownedPaths, then fallback to implementationPaths.
  3. If multiple specs match, prefer direct ownership over reference ownership, then prefer lower-level child specs over parents.
  4. Load the descriptive spec for intent, the test spec for done conditions, and the technical spec for the internal contract. The technical spec is always required; its depth scales with the task.
  5. Use section IDs for focused loading. Do not load whole sibling specs unless the target section is insufficient.
  6. If no spec resolves, create or update a spec before changing durable behavior.
Board cards: a card should carry spec IDs when known. If a card only names a feature, the agent searches registry titles/tags, confirms the candidate, and records the board card reference in metadata if the work proceeds.

Path update loop

Moving or deleting a file is a spec event when a spec references that file.

ChangeAgent obligationReview check
File movedUpdate every matching ownedPaths.path, visible path list, and coverage path, then regenerate the registry.Search old path across specs before final answer.
File deletedRemove or replace path references, and explain whether spec ownership moved to a child spec.No generated registry entry points at a missing file unless status is retired and notes explain why.
Test renamedUpdate test spec coverage and regenerate the registry.Acceptance IDs still map to the right section IDs.
