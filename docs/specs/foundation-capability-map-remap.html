Foundation Capability Map Remap

This job remaps Foundation capability specs when the root goal, vocabulary, or parent capability model changes.

Status: active
Updated: 2026-06-20
Canonical section: #job-intent

Job intent

When Foundation's capability language changes, an agent must revise the capability map without turning artifact names, workflow phases, backfill layers, or implementation details into capabilities. This job creates a controlled remap plan first, evaluates that plan, then applies graph and metadata changes in small passes.

FieldContract
TriggerThe root Foundation goal, capability/job definitions, parent capability families, or spec-selection rules change.
ActorHuman-guided agent or Foundation maintainer.
OutputA remap plan plus updated system, capability, job, graph, validator, and skill routing artifacts where needed.
CompletionEvery affected current capability/job/spec is classified, actioned or explicitly deferred, and the remap eval plus structural checks pass.

Capability supported

This job supports foundation.author-durable-specs.capability by keeping capability specs outcome-shaped, graph-compatible, and aligned with the current Foundation system goal. It also uses foundation.spec-selection.job and foundation.author-capability-spec.job as routing and authoring inputs.

Current handling

The Foundation system spec now contains graph-backed parent capability families and current child capability specs. The remaining pressure is semantic enforcement: visible titles, reliable outcomes, graph links, validators, and skills must keep capability language outcome-shaped instead of drifting back to artifact names, workflow phases, or implementation layers. Backfill mapping rules are intentionally deferred to a later propagation pass.

Proposed handling

The remap proceeds from a parent capability draft into a graph-backed map. Each current spec gets one row before implementation changes: current classification, proposed parent capability, proposed child capability role, action needed, and blockers.

Process

  1. Read the root system goal, definitions, spec-selection job, capability authoring job, and current capability map.
  2. Generate or refresh the current spec inventory from the registry: system, capability, job, technical, eval, skill ownership, validator ownership, and backfill ownership.
  3. For exactly one current capability or proposed parent capability at a time, run the standing-ability, parent-capability, child-capability, triggered-work, and decomposition tests.
  4. Write one remap row with current classification, proposed classification, parent, child role, jobs/evidence affected, action needed, and blocker if any.
  5. Evaluate the row against foundation.capability-map-remap.eval. Revise until the row preserves intent and rejects artifact/phase/job-shaped names.
  6. After every row is acceptable, apply remap edits in passes: system map, parent capability specs, child capability specs, job parent/support links, graph metadata, validators, AGENTS/skills, then backfill/mapping rules.
  7. After each pass, run the smallest relevant checks. After all passes, run registry, spec, graph, self-map, and targeted remap checks.
No bulk closure: a remap is not complete because the parent family names look right. Every current capability, job owner, skill owner, validator assumption, and backfill boundary affected by the map must either be updated, explicitly deferred with evidence, or marked blocked.

Revision

If the remap eval finds a vague parent, job-shaped child capability, missing job owner, stale validator assumption, or omitted backfill consequence, revise the smallest wrong artifact first. Prefer revising the remap row before editing spec metadata. Prefer revising the affected child capability before changing parent graph links. Only update validators after the intended graph relationship is explicit in specs.
