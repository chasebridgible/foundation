Backfill evaluation process

Foundation evaluates backfill results with an LLM judge workflow that compares generated artifacts to a golden example, scores semantic quality on a strict 100-point scale, updates the Job / Spec Queue, and routes revisions before the work is called acceptable.

Status: active
Updated: 2026-05-21
Golden example: foundation.backfill-golden-example

Mechanical checks prove the substrate is wired. Evaluation proves the produced specs are useful: capability-complete, specific, traceable, architecture-flexible, and strong enough for a future build agent to understand intended behavior from specs alone. Passing quality is intentionally hard: broad summary specs are useful evidence, not acceptable backfill output.

Evaluator journey

  1. Calibrate: read the golden example and this process spec.
  2. Load artifacts: read target AGENTS.md, the dated backfill report, Capability Map, Artifact Inventory, Job / Spec Queue, top-level app spec, and current slice specs.
  3. Select scope: evaluate the current slice or repo-wide graph and record its queue status before scoring.
  4. Spec-only pass: review generated specs first and judge what a future build agent could understand from the specs alone, including which source files would still be needed for core behavior.
  5. Evidence audit: sample original evidence for high-risk claims, UX-visible behavior, permissions, data contracts, architecture constraints, integrations, and unresolved questions.
  6. Score: assign 0-10 category scores with citations to report rows, spec sections, and evidence paths.
  7. Report: create or update docs/specs/backfill/evaluation-report-YYYYMMDD-NN.html using the same run ID as the backfill report.
  8. Route revisions: append the evaluation result to the backfill report, update the Job / Spec Queue, and route each revision item to the owning backfill skill layer.

Scoring rubric

Each category is scored from 0 to 10, for a total of 100. The scale is intentionally strict: 6-7 is useful but incomplete, 8 is a strong draft, 9 is near-acceptable, and 10 is excellent. A merely passable backfill should score around 65-75, not pass.

Evaluation report contract

  • Header: run ID, target repo, evaluator date, evaluated report path, spec graph summary, and overall result.
  • Scorecard: category scores from 0 to 10, total score out of 100, citations, and category-level revision status.
  • Capability findings: Capability Map rows still unmapped, too broad, missing descriptive sections, missing technical sections, missing verification targets, or needing evaluation.
  • Slice findings: findings grouped by slice and routed to Artifact Inventory, Process / Action Map, descriptive, rendered UX, technical, or Review Spec Adequacy skills.
  • Architecture flexibility: findings on required contracts, current evidence, architecture constraints, and implementation latitude.
  • Spec-only rebuild probe: what a future build agent could build or would still need to know from specs alone, including source files still required for core behavior.
  • Revision queue: concrete next actions with owner skill, affected artifacts, blocking gap, and exit criterion.
  • Job / Spec Queue update: slice status, score, owner skill, next action, and whether the slice is needs-revision, revision-ready, acceptable, or blocked by human decision.

Acceptability gate

A backfill result is acceptable when the total score is at least 96, every category scores at least 9, spec-only rebuild readiness scores 10, evaluated capability rows are acceptable, no evaluated row needs split, and all remaining human questions are isolated as non-blocking decisions. A score below that threshold routes work back to the owning backfill layer and the slice is re-evaluated before it can be marked acceptable.
