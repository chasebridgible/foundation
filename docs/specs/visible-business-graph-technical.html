Visible Business Graph Technical Spec

The implementation must make graph compatibility enforceable across Foundation specs and renderable as local, derived HTML views.

Status: draft
Updated: 2026-06-01
Parent: foundation.visible-business-graph.system

This layer owns the graph metadata schema, validation rules, graph extraction, JSON build output, static canvas rendering, and graph eval plumbing. It must not make the canvas the source of truth; canonical meaning remains in HTML specs and source registry style metadata.

Internal sequence

  1. Discover HTML specs under docs/specs, excluding generated backfill summaries.
  2. Parse spec-metadata, sections, canonical section, and graph-metadata.
  3. Validate graph metadata for schema, node types, edge types, node sources, edge sources, section resolution, duplicate IDs, dangling edges, orphan support specs, capability-to-job realization edges, and job-to-capability support edges.
  4. Build deterministic graph JSON from the validated metadata.
  5. Render a standalone local HTML canvas from graph JSON.
  6. Evaluate graph and canvas outputs against expected graph fixtures when a fixture exists.

Contracts

ContractRuleReason
Graph metadata blockEvery spec carries graph-metadata with schema, owner spec ID, nodes, and edges.Agents and canvas builders need a stable graph source in the HTML artifact.
Source resolutionEvery node and edge names a source spec and section that resolve through the current repo.Canvas claims must link back to readable evidence.
Supported typesNodes and edges use the enums in the system spec.Derived views can group, filter, and style nodes predictably.
Capability/job mappingA capability child job must be connected by a realized-by edge from the capability node to the job node. A job with an owning capability must expose a supports edge from the job node back to the capability node.The capability map must be able to expand from outcomes into the jobs that make them real, and job inspection must preserve the owning capability.
Authoring integrationspec:new and templates scaffold graph metadata for all spec types.New specs should be graph-compatible by default.
Canvas outputCanvas HTML embeds graph data and local JS/CSS only.The user can open the canvas on a laptop without a server.
Canvas layoutThe default capability map lays nodes top-down, centers the system layer above one horizontal capability band, does not wrap capability cards into multiple rows, and keeps jobs and job context visible only after inline expansion. When the stage is wider than the visible canvas, the initial pan centers the stage inside the available viewport instead of pinning the far-left edge.The first view should communicate business shape before exposing every job/process detail.
Canvas edge policyThe renderer must treat visual edges as selected-path affordances, not an exhaustive drawing of graph metadata. The default capability and actor overviews draw no global edges; expanded capabilities draw only local capability-to-job realization edges; expanded jobs draw only local job context edges; expanded actors draw only handled-job edges, oriented visually from actor to job.The map must stay readable as an inspection surface while preserving complete metadata in the graph JSON and details panel.
Canvas branch colorExpanded capabilities, their revealed jobs, and their local realization edges must share a deterministic accent color derived from the capability ID. Expanded job context may inherit the same accent so the selected branch remains visually coherent.Color should reinforce hierarchy and selection without becoming canonical data.
Canvas expansionCapability cards expand jobs inline; job cards expand process, actor, tool, evidence, metric, and gap nodes inline; actor cards in the actors map expand handled jobs. Expandable cards update aria-expanded.Expansion must be real interactive behavior, not a static marker in generated HTML or a hidden tab switch.
Canvas tabsThe generated canvas exposes only Capability map and Actors tabs.Secondary detail views should not fragment the core business exploration path.
Canvas navigationThe rendered stage supports pointer drag panning from the canvas background while keeping node selection and source links usable.Large business graphs must be navigable without relying on browser scrollbars alone.
Document navigationGenerated canvases may load the shared HTML docs navigation, but the canvas shell must account for the injected sidebar, avoid inherited document padding, and respond when the shared far-left document-navigation toggle collapses or restores that sidebar.The graph must remain usable as a fixed canvas while retaining source-document navigation across specs, canvases, and target repos.
