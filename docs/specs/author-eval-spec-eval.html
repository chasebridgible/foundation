Author Eval Spec Eval

This eval judges whether an authored eval spec can reject bad output and route the smallest useful revision.

Status: active
Updated: 2026-06-06
Canonical section: #verification-contract

Verification contract

An eval spec passes only when it defines what evidence is collected, how the evidence is judged, what fails, what revisions are routed, and which gaps or blockers remain.

Acceptance mapping

IDConditionMust prove
EVAL-1Judgment target is clear.The eval names the spec, workflow, artifact, or behavior being judged.
EVAL-2Evidence is concrete.Tests, metrics, screenshots, traces, audits, fixtures, or model judgments are named.
EVAL-3Pass/fail is explicit.Acceptance and rejection conditions are specific enough to block weak output.
EVAL-4Revision routing is precise.Failures name the smallest wrong artifact, row, spec, skill, validator, or process.
EVAL-5Readiness questions are handled.Material ambiguity about evidence, pass/fail/block conditions, judge method, revision routing, or anti-shortcut cases is answered, blocked, or captured as a labeled assumption.
EVAL-6Anti-shortcut cases exist.Known agent shortcuts, generic compression, stale state, and self-certification risks are covered when relevant.

Coverage plan

Manual semantic review is required for judge adequacy. Structural evidence comes from registry and graph checks.

Commands and evidence

  • npm run spec:check
  • npm run foundation:visible-business-graph:check -- --repo .

Gaps and follow-ups

Future fixtures should cover vague "looks good" evals, missing revision targets, tests mistaken for evals, and evals that cannot reject generic agent output.
