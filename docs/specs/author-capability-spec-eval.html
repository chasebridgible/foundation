Author Capability Spec Eval

This eval judges whether an authored capability spec names a reliable outcome instead of a job, artifact, phase, or implementation detail.

Status: active
Updated: 2026-06-06
Canonical section: #verification-contract

Verification contract

A capability spec passes only when it names a standing ability or durable condition, has the right parent/child altitude, links to jobs and evidence, and avoids artifact or process naming.

Acceptance mapping

IDConditionMust prove
CAP-1Outcome-shaped name.The capability survives "the system can reliably..." or "this condition is reliably available..."
CAP-2Not triggered work.Verb phrases like create, run, record, review, map, author, deploy, or revise are routed to jobs unless rewritten as outcomes.
CAP-3Parent/child relation is outcome-based.A parent is a broader outcome family; a child is a narrower reliable outcome.
CAP-4Jobs are explicit.Capability specs expose realizing jobs with registry links and realized-by graph edges.
CAP-5Readiness questions are handled.Material ambiguity about outcome, parent/child altitude, realizing jobs, evidence, or constraints is answered, blocked, or captured as a labeled assumption.
CAP-6Evidence is visible.Evidence, evals, gaps, and revision targets are named.
CAP-7Deterministic language check passes.npm run foundation:capability-language:check rejects job-shaped child titles, artifact-only titles, and missing or weak reliable-outcome rows.

Coverage plan

Manual semantic review uses the capability/job tests in foundation.spec-selection.job. Structural evidence comes from registry, graph, and self-map checks.

Commands and evidence

  • npm run foundation:capability-language:check
  • npm run spec:check
  • npm run foundation:self-map:check

Gaps and follow-ups

Future fixtures should cover jobs disguised as capabilities, artifact names, vague parent capabilities, unnecessary child capabilities, and missing job links.
