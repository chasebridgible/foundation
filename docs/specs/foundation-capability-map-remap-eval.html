Foundation Capability Map Remap Eval

This eval judges whether a Foundation capability remap preserves intent, uses the settled vocabulary, and avoids turning jobs, phases, artifacts, or implementation details into capabilities.

Status: active
Updated: 2026-06-06
Canonical section: #verification-contract

Verification contract

A capability map remap passes only when every affected current capability, proposed parent capability, job owner, skill owner, validator assumption, and backfill boundary is classified and actioned or explicitly deferred. The eval should fail broad remaps that only rename headings, preserve old phase-shaped capabilities, or skip graph and validator consequences.

Acceptance mapping

IDConditionMust prove
REM-1Root goal preserved.The remap continues to serve the canonical Foundation system goal rather than a narrower backfill-only or spec-only objective.
REM-2Vocabulary applied.Capability, parent capability, child capability, job, process, action, artifact, interface, evidence, and eval are used according to docs/definitions/definitions.html and foundation.spec-selection.job.
REM-3Every current capability classified.Each current capability-type spec has a row naming current classification, proposed parent, proposed role, and action needed.
REM-4Parent/child links decompose outcomes.Parent to child relationships are broader reliable outcome to narrower reliable outcome, not phase to phase or artifact to artifact.
REM-5Job-shaped names are caught.Verb-led or phase-led candidates are rewritten as standing abilities or routed to jobs/process sections.
REM-6Subsystem-shaped specs are identified.Specs whose IDs or titles say system while metadata says capability are retained only with explicit compatibility notes or a later rename plan.
REM-7Graph and validator consequences are named.The plan names metadata, graph edge, self-map, visible graph, AGENTS/skill, and backfill/mapping consequences before edits are closed.
REM-8Gaps remain visible.Parent capabilities without current child specs, such as real-world operation if still unmapped, are explicit gaps and not silently omitted.
REM-9Smallest wrong artifact is revised first.Eval findings point to the exact row, spec, metadata edge, validator rule, or skill instruction that should change next.
REM-10Parent/child metadata is reciprocal.If a capability lists a child capability, that child must declare the same capability as its parent.

Coverage plan

The first coverage layer is row-level manual review of the remap table. Structural checks come after rows are acceptable because registry and graph checks cannot judge whether a capability name is actually outcome-shaped.

  • Manual review checks each remap row against the standing-ability, parent-capability, child-capability, triggered-work, and decomposition tests.
  • npm run foundation:capability-map-remap:check fails when any current capability spec lacks a remap row, any required parent family is missing, a remap row has an empty required cell, or parent/child capability metadata is stale.
  • npm run spec:check checks registry, graph metadata, and self-map structure after edits.
  • npm run foundation:self-map:check specifically checks Foundation capability/job/skill ownership assumptions.

Commands and evidence

  • npm run spec:registry after spec metadata changes.
  • npm run foundation:capability-map-remap:check after editing the remap table.
  • npm run spec:check before handoff.
  • npm run foundation:self-map:check after capability, job, or skill ownership changes.
  • npm run foundation:visible-business-graph:check -- --repo . after graph metadata changes.

Gaps and follow-ups

The deterministic checker validates table coverage and row shape. It does not replace semantic review because capability/job boundaries require judgment.
