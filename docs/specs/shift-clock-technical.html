Shift Clock Technical

A shift clock implementation persists a current-run work-shift deadline, records sparse check receipts, and connects clock evidence to bounded work records so an evaluator can distinguish productive continuation from clock watching.

Status: active
Updated: 2026-06-20
Canonical section: #required-depth

Required depth

This is a reusable technical contract for shift-clocked long-running agent work. It does not prescribe one command name or storage format. It constrains the state fields, receipt semantics, timing behavior, and score evidence needed for a reliable shift clock.

Internal sequence

  1. Start: persist clock state for the current run ID before substantive work begins.
  2. Work: write bounded unit evidence and validation receipts as the run progresses.
  3. Attempt exit: check the clock only when the worker believes it is done, blocked, or ready to hand off.
  4. Continue: when expired:false, append a blocked-exit receipt and select more useful work.
  5. Close: when expired:true, finish the current unit, run final checks, write final report/handoff, and mark the goal complete when available.

Timing, concurrency, and ordering

  • Exit-only checks: do not check before every bounded unit. Check after substantive work when ready to exit, blocked, or ready to hand off.
  • Sparse check target: a healthy run normally has long gaps or substantial evidence between checks. Repeated checks without work are failures.
  • No idle waits: sleep, long waits, guard sessions, or holding until the deadline are forbidden unless an external blocker makes all useful in-scope work impossible.
  • Single writer: only one process should mutate the clock state for a run. Supervisors read receipts and durable artifacts rather than racing the worker.
