Author System Spec Eval

This eval judges whether an authored system spec preserves root system intent and can govern downstream capability/job work.

Status: active
Updated: 2026-06-06
Canonical section: #verification-contract

Verification contract

A system spec passes only when it names one canonical root goal, the system boundary, parent capability families, operating loop, evidence model, revision model, and graph relationships clearly enough that downstream specs can align to it.

Acceptance mapping

IDConditionMust prove
SYS-1One root goal.The spec does not contain competing operating goals.
SYS-2Clear boundary.In-scope and out-of-scope system responsibilities are visible.
SYS-3Capability families.Parent capabilities decompose the root goal into understandable outcome families.
SYS-4Evidence and revision loop.The system says how reality updates intent over time.
SYS-5Readiness questions are handled.Material ambiguity about root goal, boundary, capability families, evidence, or revision is answered, blocked, or captured as a labeled assumption.
SYS-6Graph compatibility.Metadata and graph edges expose the system and its child specs.

Coverage plan

Manual semantic review is required until a system-spec fixture harness exists. Structural evidence comes from npm run spec:check and graph checks.

Commands and evidence

  • npm run spec:check
  • npm run foundation:visible-business-graph:check -- --repo .

Gaps and follow-ups

Future fixtures should cover root-goal rewrites, subsystem boundaries, and philosophical text that should not become the canonical operating goal.
