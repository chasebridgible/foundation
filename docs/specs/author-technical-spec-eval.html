Author Technical Spec Eval

This eval judges whether an authored technical spec defines implementation contracts without overwriting upstream intent.

Status: active
Updated: 2026-06-06
Canonical section: #verification-contract

Verification contract

A technical spec passes only when constrained behavior, implementation latitude, data/API/command contracts, failure and recovery behavior, operational constraints, and verification needs are explicit.

Acceptance mapping

IDConditionMust prove
TECH-1Attached to intent.The technical spec supports a job or capability and does not replace it.
TECH-2Contracts are explicit.Data, API, command, permission, sequencing, migration, or operational contracts are named as applicable.
TECH-3Latitude is preserved.Constrained behavior is separated from flexible implementation choices.
TECH-4Failure/recovery is covered.Important error, rollback, retry, stale-state, and recovery paths are visible.
TECH-5Readiness questions are handled.Material ambiguity about constrained behavior, flexible latitude, contracts, failure paths, or verification is answered, blocked, or captured as a labeled assumption.
TECH-6Verification is mapped.Tests, checks, manual evidence, or gaps point back to the technical contract.

Coverage plan

Manual semantic review is required for architecture and contract adequacy. Structural evidence comes from registry and graph checks.

Gaps and follow-ups
