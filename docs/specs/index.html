The spec system

Specs are durable source-of-truth contracts for what should exist. They are not traditional docs, stale plans, or workflow procedures. This folder defines the first HTML-native spec system for productized-agency software development substrate.

Status: active
Updated: 2026-05-12
Metadata: per-spec #spec-metadata is canonical

Registry contract

Each HTML spec owns its canonical spec-metadata. The registry is the generated aggregate index agents read before loading a full spec. It maps stable spec IDs to HTML files, statuses, parent/child links, directly owned implementation paths, and test coverage links.

Machine entry point: parse <script type="application/json" id="spec-registry"> in this file. When spec metadata changes, run npm run spec:registry, then npm run spec:check to prove the generated registry matches each spec's canonical metadata and spec-registry.schema.json. CI runs the same check on pull requests and pushes to main.

How an agent enters

From a code path
Search the registry for ownedPaths.path or implementationPaths.path. Load the owning low-level spec first, then walk to parent intent and sibling test specs.
From a feature name
Search registry titles, tags, and section IDs. Start with the descriptive spec, then load the technical spec at the depth appropriate to the implementation risk.
From a board card
Read the card's explicit spec IDs if present. If absent, search by feature terms, confirm the candidate spec, and add the board card reference during substrate update.

The default reading order for implementation work is: descriptive spec section relevant to the user outcome, test spec acceptance mapping, technical spec at the depth the task requires, then directly owned paths. When creating or revising a spec, follow the spec authoring contract before depending on the generated registry.

Templates

Descriptive spec template
Product intent, user model, interface journey, rules, edge cases, UX contracts, design placeholders, path ownership, and test mapping.
Technical spec template
Required for every software task. Defines internals, contracts, timing, boundaries, migrations, ADRs, and non-goals at a depth proportional to the task.
Test spec template
Plain-English verification contract with acceptance mapping, signal choice, commands, evidence, gaps, and co-evolution rules.

Concrete examples

The examples use the Fired Up Pizza kitchen queue as a low-level feature spec set. They are examples of shape, not canonical product direction.
