The spec system

Specs are durable source-of-truth contracts for what should exist, what work must happen, and how reality will be judged. The shared spec types are system, capability, job, technical, and eval. Legacy job specs remain valid during migration; the new name for that intent contract at work-scope is job spec.

Metadata: per-spec #spec-metadata is canonical

Registry contract

Each HTML spec owns its canonical spec-metadata. The registry is the generated aggregate index agents read before loading a full spec. It maps stable spec IDs to HTML files, statuses, parent/child links, directly owned implementation paths, and evidence or coverage links.

Machine entry point: parse <script type="application/json" id="spec-registry"> in this file. When spec metadata changes, run npm run spec:registry, then npm run spec:check to prove the generated registry matches each spec's canonical metadata and spec-registry.schema.json. CI runs the same check on pull requests and pushes to main.

How an agent enters

From a code path
Search the registry for ownedPaths.path or implementationPaths.path. Load the owning low-level spec first, then walk to parent intent and sibling eval specs.
From a feature name
Search registry titles, tags, and section IDs. Start with the job spec that owns the intended work, then load technical and eval specs as needed.
From a board card
Read the card's explicit spec IDs if present. If absent, search by feature terms, confirm the candidate spec, and add the board card reference during substrate update.

The default reading order for implementation work is: job spec section relevant to the intended work, eval spec acceptance mapping, technical spec at the depth the task requires, then directly owned paths. For broad orientation, start with the system spec and capability spec first. When creating or revising a spec, follow the spec authoring contract before depending on the generated registry.

Templates

System spec template
Root system intent, capability map, operating loop, spec graph, evidence, and revision model.
Capability spec template
Outcome contract, jobs, interfaces, sources, evidence, evaluation, and revision for one capability.
Job spec template
The work contract for a named body of work, with its process contained inside by default.
Legacy job spec template
Compatibility template for existing software feature specs. New work should usually start from a job spec.
Technical spec template
Required for every software task. Defines internals, contracts, timing, boundaries, migrations, ADRs, and non-goals at a depth proportional to the task.
Eval spec template
Plain-English verification contract with acceptance mapping, signal choice, commands, evidence, gaps, and co-evolution rules.
