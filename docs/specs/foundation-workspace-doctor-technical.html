Foundation Workspace Doctor Technical Spec

The Foundation doctor is an on-demand diagnostic command that tells a user whether the canonical Foundation repo, global Codex pointer, target repo adapter, target spec namespace, and target CI pinning are shaped correctly for the workspace model.

Status: active
Updated: 2026-05-29
Canonical section: #required-depth

Required depth

The doctor is a setup and maintenance diagnostic, not a long-running sync system. It should give deterministic local evidence that a machine and optional target repo are connected to canonical Foundation without requiring a network call or mutating the target repo.

The command must be cheap enough to run during setup, before handoff, and when a target repo feels misconfigured. It reports failures for broken core assumptions and warnings for missing optional maturity such as target CI that has not been configured yet.

Internal sequence

  1. Parse CLI flags for Foundation path, optional target repo path, global Codex AGENTS path, JSON output, and spec-check skipping.
  2. Validate the Foundation repo exists, is a Git repo, has AGENTS.md, and contains the required Foundation skills.
  3. Validate global Codex instructions exist and explicitly point to the canonical Foundation path and Foundation AGENTS.md.
  4. Run Foundation spec registry/check unless the caller deliberately skips it.
  5. When a target repo is provided, validate repo shape, local AGENTS.md adapter behavior, target spec namespace, and GitHub workflow Foundation pinning.
  6. Render a stable text or JSON report and exit nonzero when any failure exists.

Failure modes

  • Broken machine setup: the user should see the exact missing path or missing skill and a failed summary.
  • Unclear global pointer: the user should see which global AGENTS file was checked and which Foundation path was expected.
  • Target repo copies Foundation rules: the report should name duplicated rule text so the adapter can be reduced to local context.
  • Product specs under Foundation namespace: the report should name each file and spec ID that must move into the target namespace.
  • CI not ready: the report should warn without failing while the repo is still local-only or before GitHub workflows are configured.
