Research Corpus Refresh Technical

A research refresh implementation needs deterministic state transitions around agent judgment: source targets, findings, rejections, exports, checks, branch publication, and notifications.

Status: active
Updated: 2026-06-14
Canonical section: #required-depth

Required depth

This is a reusable technical contract for project-specific research systems. It does not prescribe one implementation language. It constrains the durable artifacts, command semantics, validation boundaries, workbook behavior, and scheduled publish path that make recurring research trustworthy.

Non-goals

This spec does not define a universal web crawler, a shared database, or a direct-to-main automation. Project repos keep their domain schema, source registry, and checkers.

Internal sequence

  1. Initialize: read project AGENTS, specs, source registry, current corpus, and workbook; create run manifest.
  2. Select: return the next bounded source target or work unit with enough context for one agent pass.
  3. Record: merge source-backed findings, rejections, route/contact records, coverage updates, and evidence references idempotently.
  4. Report: regenerate canonical JSON state if needed, workbook sheets, dated brief, source snapshot summary, and merge/notification templates.
  5. Check: validate schema, dedupe keys, source URLs, workbook readback, off-scope exclusions, terminal coverage, and project-specific acceptance rules.
  6. Publish: create or update a review PR, write merge and notification receipts, and leave explicit blockers when GitHub or checks are unavailable.

Timing, concurrency, and ordering

  • Single writer: one process writes the canonical corpus and workbook at a time. Parallel research workers can emit replayable inputs, but consolidation happens through one writer.
  • Timer-gated runs: a timer is an exit gate, not a work strategy. The worker keeps doing substantive bounded work until it checks an expired timer at a natural exit point. Reports attach timer status only when the persisted timer belongs to the current run ID; stale timers stay in historical receipts, not current briefs.
  • Scheduled cadence: monthly is the default for stable research corpora; project specs can select biweekly or ad hoc when source churn justifies it.
  • Branch ordering: automation creates a dated branch before corpus writes and publishes a review PR after validation.

Boundaries and abstraction level

  • Owns: generic refresh loop, command expectations, artifact classes, workbook derivation rule, timing rule, and review-first automation contract.
  • References: domain-specific source registries, schemas, scripts, workbook column choices, and notifications owned by project repos.
