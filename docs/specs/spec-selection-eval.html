Spec Selection Eval Spec

This eval judges whether a Spec Selection Result routed intent to the right layer before authoring begins.

Status: active
Updated: 2026-06-06
Canonical section: #verification-contract

Verification contract

The eval accepts a Spec Selection Result only when it preserves the user's material intent, asks or records the clarification needed for classification, chooses the correct primary classification, names required companion specs, rejects plausible wrong classifications, and hands off to the correct next skill or non-spec owner.

The eval fails any result that lets an authoring skill classify itself, starts authoring before routing, skips a material altitude question, asks unfocused questions that do not affect routing, accepts a job-shaped phrase as a capability, accepts an artifact/interface/process/action as a capability, or omits a required technical/eval companion spec.

Acceptance mapping

IDConditionMust prove
SS-1Intent is restated without losing the user's goal.The result separates intent from implementation guesses.
SS-2Existing ownership is searched before creating new specs.The result names an existing owner, no owner found, or the evidence for uncertainty.
SS-3Clarification is proportional.Material altitude ambiguity is answered, blocked, or captured as a labeled assumption; unnecessary broad interviewing is avoided.
SS-4Primary classification is correct.Classification matches system, parent capability, child capability, job, technical, eval, or non-spec boundary according to the job spec tests.
SS-5Capability/job boundary is preserved.Standing abilities remain capabilities; triggered work remains jobs.
SS-6Parent/child decomposition is outcome-based.Parent to child means broader outcome to narrower outcome, not phase to phase or artifact to artifact.
SS-7Companion specs are named when needed.Technical and eval companions are explicit or explicitly blocked/gapped.
SS-8Rejected classifications are specific.The result names plausible wrong answers and why they were rejected.
SS-9Handoff is enforceable.The next step is one authoring skill, one non-spec owner, or a blocker.

Coverage plan

The fixture set is the initial calibration surface. The deterministic command validates fixture shape and minimum coverage; future extensions may score a structured Spec Selection Result against the fixture fields, but the durable acceptance contract lives in this eval.

Gaps and follow-ups

  • The first implementation may be a manual semantic eval using this contract and fixtures.
  • A deterministic harness can validate output shape and fixture coverage, but semantic judgment still needs this eval.
  • When repeated misses appear, add the failed prompt and correct classification to the fixture set.
