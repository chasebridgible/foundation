Author Capability Spec Job

This job authors or revises parent and child capability specs after Spec Selection classifies the work as a reliable outcome.

Status: active
Updated: 2026-06-06
Canonical section: #job-intent

Job intent

This job turns a Spec Selection Result for a parent or child capability into an HTML-native capability spec that names a standing ability, not the work needed to create it.

Capability supported

Supports foundation.author-durable-specs.capability by making capability specs outcome-shaped, graph-compatible, and tied to the jobs and evals that realize them.

Current handling

Foundation currently has capability language that can be too phase-like or artifact-like. This lane prevents a remap from turning workflow layers, queue rows, file names, or commands into fake capabilities.

Proposed handling

The lane starts only from a Spec Selection Result whose primary classification is parent capability or child capability. If the phrase starts as triggered work, it must be rewritten as a standing ability or routed to the job lane.

Process

  1. Read the Spec Selection Result, definitions, capability/job tests, current parent capability if any, and capability template.
  2. Run the readiness gate: ask only questions that materially change capability outcome, parent/child altitude, realizing jobs, evidence, or constraints; otherwise continue with labeled assumptions.
  3. Confirm the candidate passes the standing-ability test.
  4. For a parent capability, name one outcome family and its child outcome boundaries.
  5. For a child capability, name the narrower reliable outcome, jobs, rules, evidence, evals, and parent relation. Keep the visible title and graph label outcome-shaped even when the compatibility ID is older.
  6. Author or revise HTML prose, metadata, graph edges, jobs table, coverage, and visible links.
  7. Run the capability-spec eval and npm run foundation:capability-language:check, revise blocking findings, then run registry, graph, and self-map checks.

Context, tools, and interfaces

  • skills/author-capability-spec/SKILL.md is the agent procedure.
  • docs/specs/templates/capability-spec-template.html is the authoring shape.
  • foundation.author-capability-spec.eval is the quality gate.

Readiness questions: What reliable outcome or standing ability should exist? Is this a broad parent outcome family or a narrower child outcome? What jobs make this capability real? What evidence would show the capability is reliable?

Evidence and evaluation

The authored capability spec is acceptable only when it names a reliable outcome, exposes parent/child links when needed, links jobs with realized-by, names evidence and evals, passes capability-language checking, and rejects artifact or phase naming.

Revision

Revise this job when agents misclassify jobs as capabilities, create unnecessary child capabilities, use artifact/layer names as capabilities, or fail to tie capabilities to jobs and evidence.
