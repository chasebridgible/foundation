Research Workflow Technical

A research workflow implementation needs deterministic state transitions around agent judgment: source targets, findings, rejections, checks, merge cleanup, publication, and final access surfaces.

Status: active
Updated: 2026-06-14
Canonical section: #required-depth

Required depth

This is a reusable technical contract for project-specific research systems. It does not prescribe one implementation language. It constrains the durable artifacts, command semantics, validation boundaries, merge/cleanup path, access-surface behavior, and scheduled publish path that make recurring research trustworthy.

Non-goals

This spec does not define a universal web crawler, a shared database, or a direct-to-main automation. Project repos keep their domain schema, source registry, and checkers.

Internal sequence

  1. Initialize: read project AGENTS, specs, source registry, current storage, and access-surface rules; create run manifest.
  2. Select: return the next bounded source target or work unit with enough context for one agent pass, following the project source list before broader discovery.
  3. Record: merge source-backed findings, rejections, route/contact records, coverage updates, and evidence references idempotently.
  4. Report: regenerate storage exports if needed, access surfaces, dated brief, source snapshot summary, and merge/notification templates.
  5. Check: validate schema, dedupe keys, source URLs, access-surface readback, off-scope exclusions, terminal coverage, and project-specific acceptance rules.
  6. Merge and cleanup: review changed records and generated artifacts, create or update a review PR, merge when policy allows, write merge/notification receipts, and leave explicit blockers when GitHub or checks are unavailable.
  7. Access: expose the final dashboard, workbook, brief, HTML view, app view, or other access surface from the clean published state.

Timing, concurrency, and ordering

  • Single writer: one process writes durable storage and generated access surfaces at a time. Parallel research workers can emit replayable inputs, but consolidation happens through one writer.
  • Shift-clocked runs: a shift clock is an exit gate, not a work strategy. The worker keeps doing substantive bounded work until it checks an expired shift clock at a natural exit point. Reports attach shift-clock status only when the persisted clock belongs to the current run ID; stale receipts stay in historical evidence, not current briefs.
  • Scheduled cadence: monthly is the default for stable research corpora; project specs can select biweekly or ad hoc when source churn justifies it.
  • Branch ordering: automation creates a dated branch before storage writes, validates changed artifacts, follows the PR/merge path, and only then treats final access links as ready to share.

Boundaries and abstraction level

  • Owns: generic research loop, command expectations, artifact classes, merge/cleanup path, access-surface derivation rule, timing rule, and review-first automation contract.
  • References: domain-specific source registries, schemas, scripts, dashboard/sheet/view choices, and notifications owned by project repos.
