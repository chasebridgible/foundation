Foundation Backfill Specs

Backfill Specs is the Foundation workflow for meeting an existing repository where it is, reading the built system as evidence, and creating a complete draft job and technical spec graph that becomes the repo's intended contract from here forward while preserving implementation flexibility.

Status: active
Updated: 2026-06-03
Canonical section: #product-intent

Backfill Specs

A user should be able to point Foundation at a mature repo with messy or missing docs and receive a durable draft spec system for the whole repo, while keeping code intact and treating stale documentation as supporting evidence.

Intent

The backfill workflow converts existing code, schemas, tests, and supporting docs into an outstanding rebuild-grade set of draft job and technical specs backed by durable evidence layers. It gives the user a reviewable spec graph, a top-level system/app contract, capability-level specs, canonical JSONL layer artifacts, derived HTML summaries, a dated HTML report with an embedded queue, and a JSONL execution run log that makes the work observable across long sessions and context resets.

Backfilled specs are intent-rigid and architecture-flexible. Job specs define user and operator intent at an architecture-agnostic level. Technical specs preserve required contracts and classify current implementation evidence, architecture constraints, and implementation latitude.

Backfill completeness is capability-based. A capability combines actor, intended outcome, domain object, actions, state model, permissions/rules, user or operator-visible experience, backing contracts, failure/recovery behavior, and evidence. Routes, screens, endpoints, tables, jobs, workflows, and infrastructure resources are evidence for capabilities, not proof of coverage on their own.

The workflow is designed for large repos and monorepos. It operates in loops: create a capability-backed slice, evaluate the slice, revise weak categories, mark the slice outstanding, then continue. It loads the evidence needed for the current slice and keeps moving through the Job / Spec Queue until every relevant capability is outstanding, blocked by a named human decision, parent-owned with an explicit reason, or marked out of scope.

This spec is the umbrella workflow contract: the whole backfill system produces outstanding rebuild-grade specs from existing repos. Each durable layer job spec owns the exact agent behavior, artifact contract, row or slice loop, and correctness gate for that layer.

The sequence is a no-skip contract. Inventory Artifacts proves the repo was seen; Map Surfaces, Map Capabilities, Define Spec Jobs, Gather Context, Map Processes, Author Specs, Evaluate Job Slices, and Evaluate System Coherence are the required process steps for proving the system was understood. A later layer cannot claim completion from chat memory, broad repo search, or a direct jump from artifact rows to specs.

The durable artifact names are Artifact Inventory, Surface / Function Map, Capability Map, Job / Spec Queue, Context Pack, Process / Action Map, Specs, Job / Capability Eval, and System Eval. Spec file names, script file names, spec IDs, artifact names, and command names use canonical implementation keys such as artifact-inventory, surface-function-map, capability-map, spec-job-queue, context-pack, and process-action-map.

Workflow boundary

Each durable backfill layer has job, technical, and eval specs when it owns an artifact, state model, or correctness gate. Implementation tests and product acceptance tests can still be backfilled later, but the substrate's own check/eval contracts are part of this workflow.

Backfill Specs creates spec artifacts and review reports. Code cleanup, documentation deletion, architecture rewrites, and old-plan cleanup belong to separate cleanup or implementation workflows after the specs are reviewed.

Interface journey

  1. Setup: the target repo is connected with the Foundation setup skill, target package aliases are synced from Foundation when package.json exists, then npm run foundation:doctor -- --repo <target-repo> confirms the shared substrate is discoverable.
  2. Backfill start: the agent reads target AGENTS.md, Foundation AGENTS.md, the Backfill Specs skill, any existing dated report, and the top-level system/app spec if present.
  3. Run identity: the agent creates or continues a dated report such as docs/specs/backfill/review-report-20260521-01.html and matching run log such as docs/specs/backfill/run-log-20260521-01.jsonl. If a target repo adapter names the current report, that file is canonical for the run.
  4. Inventory Artifacts: the agent creates or refreshes the manifest and artifact inventory so every repo-owned file has exactly one current row. Canonical files and commands use artifact-inventory.
  5. Map Surfaces: eligible mapped artifact rows become evidence surfaces such as routes, screens, APIs, commands, jobs, tables, workflows, infra resources, active behavior docs, and dependencies; inert rows remain Artifact Inventory evidence.
  6. Map Capabilities: ready surfaces become capability rows using the full formula: actor, intended outcome, domain object, actions, states, rules, experience, backing contracts, failure/recovery, and evidence.
  7. Define Spec Jobs: broad capability rows are split before closure, and a durable Job / Spec Queue names exact slices, owner layer, exit criteria, status, blockers, and next action.
  8. Gather Context: each queued slice receives a Context Pack with only the artifacts, surfaces, capabilities, tests, docs, schemas, and snippets needed for that slice.
  9. Map Processes: user and operator processes/actions, states, rules, permissions, empty/loading/error states, recovery paths, and visible behavior are mapped before specs are authored.
  10. Author Specs: the agent consumes one outstanding Process / Action Map row at a time, writes or updates one architecture-agnostic job spec and one contract-first technical spec, checks them, evaluates that row to outstanding, and only then moves to the next target.
  11. Evaluate Job Slices: each slice runs the hard loop: create, evaluate, revise, re-evaluate, mark outstanding, then continue.
  12. Evaluate System Coherence: after slice coverage is complete, the agent evaluates whether the spec graph preserves system intent well enough that a future agent can rebuild behavior from specs without rediscovering source.
  13. Handoff: the agent produces the final report: what was covered, which specs are draft or ready, what remains uncertain, what is blocked by human decisions, and what should happen next.

Backfill terminology flow

  1. Artifact Inventory
    Maps every repo-owned file into a canonical Artifact Inventory row.
    Output: file manifest, Artifact Inventory JSONL, Artifact Inventory eval JSONL, and Artifact Inventory eval summary.
  2. Surface / Function Map
    Converts files into evidence surfaces: routes, screens, APIs, commands, jobs, tables, workflows, infra resources, docs, tests, generated artifacts, implementation functions, and support classifications.
    This answers: "What things does this repo expose, depend on, or implement?"
    Needs job, technical, and eval specs.
  3. Capability Map
    Converts surfaces into capability rows using the formula: actor + intended outcome + domain object + actions + state model + permissions/rules + visible or operator experience + backing contracts + failure/recovery + evidence.
    This answers: "What is this system supposed to let someone or something do?"
    Needs job, technical, and eval specs.
  4. Define Spec Jobs / Job / Spec Queue
    Decides whether capabilities are too broad and creates the durable Job / Spec Queue.
    This answers: "What exact slice should the agent work on next, and when is it complete?"
    Needs job, technical, and eval specs because this is where agents previously escaped by closing coarse slices.
  5. Context Pack
    For each queued slice, gathers only the files, surfaces, capabilities, tests, docs, schemas, and snippets needed for that slice.
    This protects context and makes the work resumable.
    Needs job, technical, and eval specs.
  6. Process / Action Map
    Extracts user/operator flows, states, rules, permissions, edge cases, empty/loading/error states, recovery paths, and visible/operator behavior.
    This becomes the raw material for architecture-agnostic job specs and rendered UX specs.
    Needs job, technical, and eval specs.
  7. Author Specs - Job Specs and Rendered UX
    Creates architecture-agnostic intent specs, including rendered and interactive UX when user-facing.
    Uses the Backfill Author Specs one-target loop plus strict adequacy review.
  8. Author Specs - Technical Specs
    Creates contract-first technical specs: required contracts, current implementation evidence, constraints, latitude, operational behavior, and failure modes.
    Uses the Backfill Author Specs one-target loop plus strict adequacy review.
  9. Job Slice Evaluation
    Evaluates each slice, revises it, re-evaluates it, then marks it outstanding.
    This is a hard loop: create -> evaluate -> revise -> re-evaluate -> outstanding -> continue.
  10. System Coherence Evaluation
    Evaluates the whole spec graph after all slices are outstanding, parent-owned with a precise reason, blocked by a named human decision, or out of scope.
    This asks the hard question: "Could a future agent rebuild the system from these specs without reading source?"
    Needs job, technical, and eval specs.
  11. Backfill Handoff
    Produces the final review report: what was covered, what remains uncertain, what needs human decisions, which specs are draft vs ready, latest run-log sequence, validation status, and remaining review queue.

Edge cases and recovery paths

  • Huge monorepo: start with packages, apps, routes, data domains, and service boundaries. Create the first report and Job / Spec Queue early, then load evidence slice by slice.
  • Stale documentation: record the stale doc only when it creates a material question about intent. Code and schema remain stronger evidence.
  • Behavior exists but intent is unclear: draft the observable behavior with low confidence and add the intent question to the report review queue.
  • Old docs should be deleted eventually: leave them alone during backfill and flag cleanup as a later pass after specs have been reviewed.
  • Existing specs already exist: revise them in place when they clearly own the behavior. Create new specs only when ownership is missing or the existing spec is the wrong altitude.
  • Context reset: the next agent reads the report, top-level spec, and relevant slice specs, then continues from the remaining Job / Spec Queue.
  • Large review queue: continue drafting the spec graph and keep material issues in the report so the user can review them after the backfill has full coverage.
  • Broad slice compression: when a slice contains several capabilities, split it before authoring or mark the parent row as a map with child capability rows.
  • Spec-only rebuild gap: if a future build agent would still need source files to understand core behavior, route the slice back through the owning skill instead of treating the missing behavior as a review note.

Rendered experience and design evidence

Backfilled job specs should show the intended experience inside the HTML spec whenever the slice has user-facing UI or UX-visible behavior. The rendered section is a spec-native prototype: enough HTML, CSS, and lightweight JavaScript to inspect states and interactions directly.

Job spec rendering

  • Visual surface: include a rendered layout using believable product data and copy.
  • State behavior: show important empty, loading, ready, error, permission, and completed states.
  • Interaction: include lightweight controls for stateful behaviors such as comments, tabs, filters, menus, inline editing, retries, approvals, and selections.
  • Evidence: link source screenshots, Figma frames, browser captures, or existing UI paths when they help explain the intended design.

Backfill report rendering

The dated HTML review report should be scannable, durable, and useful when opened directly in a browser.

  • Report header: run ID, date, target repo, Foundation revision if known, top-level spec ID, and current status.
  • Capability Map: visible table plus embedded JSON of capability rows, surfaces, backing contracts, spec owners, verification targets, status, and split requirements.
  • Job / Spec Queue: visible table plus embedded JSON of proposed slices with status, owner skill, spec IDs, exit criterion, blocking gaps, and next action.
  • Architecture classification: table or section showing required contracts, current implementation evidence, architecture constraints, and implementation latitude for each technical slice.
  • Evaluation status: job-slice and system-coherence eval report paths, scorecard result, and revision queue owner for generated graph quality.
  • Run log status: JSONL run log path, last event sequence, last completed phase, and latest validation result.
  • Review queue: material discrepancies, stale-doc warnings, fallback behavior decisions, and human activation needs.
  • Run state: remaining Job / Spec Queue, current status, review queue, and commands/checks last run.
