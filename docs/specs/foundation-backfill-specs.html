Foundation Backfill Specs

Backfill Specs is the Foundation workflow for meeting an existing repository where it is, reading the built system as evidence, and creating a complete draft descriptive and technical spec graph that becomes the repo's intended contract from here forward while preserving implementation flexibility.

Status: active
Updated: 2026-05-21
Canonical section: #product-intent

Backfill Specs

A user should be able to point Foundation at a mature repo with messy or missing docs and receive a durable draft spec system for the whole repo, while keeping code intact and treating stale documentation as supporting evidence.

Intent

The backfill workflow converts existing code, schemas, tests, and supporting docs into a complete set of draft descriptive and technical specs. It gives the user a reviewable spec graph, a top-level system/app contract, slice-level specs, and a dated HTML report with an embedded durable queue that makes the work resumable across context resets.

Backfilled specs are intent-rigid and architecture-flexible. Descriptive specs define user and operator intent at an architecture-agnostic level. Technical specs preserve required contracts and classify current implementation evidence, architecture constraints, and implementation latitude.

The workflow is designed for large repos and monorepos. It operates in loops: create a slice, evaluate the slice, revise weak categories, mark the slice acceptable, then continue. It loads the evidence needed for the current slice and keeps moving through the durable queue until every relevant slice is acceptable, blocked by a named human decision, parent-covered with an explicit reason, or marked out of scope.

Workflow boundary

Test specs and acceptance mapping belong to a later Backfill Tests skill when the user includes test backfill in the run.

Backfill Specs creates spec artifacts and review reports. Code cleanup, documentation deletion, architecture rewrites, and old-plan cleanup belong to separate cleanup or implementation workflows after the specs are reviewed.

Interface journey

  1. Setup: the target repo is connected with the Foundation setup skill, then npm run foundation:doctor -- --repo <target-repo> confirms the shared substrate is discoverable.
  2. Backfill start: the agent reads target AGENTS.md, Foundation AGENTS.md, the Backfill Specs skill, any existing dated report, and the top-level system/app spec if present.
  3. Run identity: the agent creates or continues a dated report such as docs/specs/backfill/review-report-20260521-01.html. If a target repo adapter names the current report, that file is canonical for the run.
  4. Repo map: the agent creates a bounded repo map from file structure, package boundaries, routes, schemas, tests, and docs using progressive disclosure.
  5. Spec graph: the agent proposes a top-level system/app spec and child spec slices. The top-level spec is created early as the spine for product promise, vocabulary, boundaries, and child specs.
  6. Slice loop: for each slice, the agent loads only relevant evidence, writes or updates draft descriptive and technical specs, classifies architecture constraints and implementation latitude, adds rendered UX where required, runs adequacy review, scores the slice, revises weak categories, and marks the slice acceptable only after it passes the strict evaluator gate.
  7. Graph evaluation: after slice coverage is complete, the agent runs Evaluate Backfill Specs against the whole graph and writes a dated evaluation report using the same run ID.
  8. Completion: the agent hands off when the proposed spec graph has acceptable descriptive and technical specs for every relevant slice, plus an evaluation report and review queue for material contradictions, fallback decisions, stale docs, and unresolved intent.

Edge cases and recovery paths

  • Huge monorepo: start with packages, apps, routes, data domains, and service boundaries. Create the first report and slice queue early, then load evidence slice by slice.
  • Stale documentation: record the stale doc only when it creates a material question about intent. Code and schema remain stronger evidence.
  • Behavior exists but intent is unclear: draft the observable behavior with low confidence and add the intent question to the report review queue.
  • Old docs should be deleted eventually: leave them alone during backfill and flag cleanup as a later pass after specs have been reviewed.
  • Existing specs already exist: revise them in place when they clearly own the behavior. Create new specs only when ownership is missing or the existing spec is the wrong altitude.
  • Context reset: the next agent reads the report, top-level spec, and relevant slice specs, then continues from the remaining slice queue.
  • Large review queue: continue drafting the spec graph and keep material issues in the report so the user can review them after the backfill has full coverage.

Rendered experience and design evidence

Backfilled descriptive specs should show the intended experience inside the HTML spec whenever the slice has user-facing UI or UX-visible behavior. The rendered section is a spec-native prototype: enough HTML, CSS, and lightweight JavaScript to inspect states and interactions directly.

Descriptive spec rendering

  • Visual surface: include a rendered layout using believable product data and copy.
  • State behavior: show important empty, loading, ready, error, permission, and completed states.
  • Interaction: include lightweight controls for stateful behaviors such as comments, tabs, filters, menus, inline editing, retries, approvals, and selections.
  • Evidence: link source screenshots, Figma frames, browser captures, or existing UI paths when they help explain the intended design.

Backfill report rendering

The dated HTML review report should be scannable, durable, and useful when opened directly in a browser.

  • Report header: run ID, date, target repo, Foundation revision if known, top-level spec ID, and current status.
  • Slice queue: visible table plus embedded JSON of proposed slices with status, owner skill, score, evidence loaded, spec IDs, exit criterion, blocking gaps, and next action.
  • Architecture classification: table or section showing required contracts, current implementation evidence, architecture constraints, and implementation latitude for each technical slice.
  • Evaluation status: slice and graph evaluation report paths, scorecard result, and revision queue owner for generated graph quality.
  • Review queue: material discrepancies, stale-doc warnings, fallback behavior decisions, and human activation needs.
  • Run state: remaining slice queue, current status, review queue, and commands/checks last run.
