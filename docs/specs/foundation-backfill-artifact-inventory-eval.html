Foundation Artifact Inventory Eval Spec

The test layer is the eval system for the Foundation Artifact Inventory: deterministic checks prove complete current coverage, graph checks prove links resolve, and quality evals prove rows are accurate enough to support agents during both backfill and steady-state build work.

Status: active
Updated: 2026-05-27
Parent: foundation.backfill-artifact-inventory.job

If this evidence passes, a reviewer can trust that every repo-owned file has a current inventory row, sampled rows describe source files accurately enough for downstream surface/capability inference, and behavior-bearing rows connect to the intended specs, capabilities, and verification evidence once those artifacts exist.

Commands or evidence required

  • npm run foundation:artifact-inventory:init -- --repo /path/to/repo --run-id YYYYMMDD-NN --mode backfill - command that creates a manifest and skeleton Artifact Inventory. New aliases may call this command but should not duplicate its logic.
  • npm run foundation:artifact-inventory:fill -- --repo /path/to/repo --run-id YYYYMMDD-NN --batch-size 25 - deterministic V1 fill helper for bounded pending batches; agents still revise rows when eval finds weak evidence.
  • npm run foundation:artifact-inventory:check -- --repo /path/to/repo --run-id YYYYMMDD-NN --mode bootstrap|strict --phase batch|handoff - command that verifies deterministic invariants, freshness, graph links, and handoff gate conditions.
  • npm run foundation:artifact-inventory:eval -- --repo /path/to/repo --run-id YYYYMMDD-NN - command that writes canonical JSONL row-quality receipts and a derived HTML summary.
  • npm run foundation:artifact-inventory:graph-check -- --repo /path/to/repo --run-id YYYYMMDD-NN --mode bootstrap|strict - command that resolves spec section IDs, capability IDs, verification targets, and explicit test gaps.
  • npm run foundation:artifact-inventory:refresh -- --repo /path/to/repo --run-id YYYYMMDD-NN - steady-state command that refreshes changed rows and reports affected files before remap.
  • npm run foundation:artifact-inventory:report -- --repo /path/to/repo --run-id YYYYMMDD-NN --report docs/specs/backfill/review-report-YYYYMMDD-NN.html - report integration command that records inventory path, pending count, checker result, eval result, latest run-log sequence, and next layer.
  • Calibration report - Sandia Oil run output, manual review notes, and optional human-authored expected file rows when a fixture is created.

Artifact Inventory eval rubric

CategoryPointsExcellent means
Completeness and freshness20Manifest and registry match exactly; content identities and sizes are current.
Kind/domain/evidence classification15Rows classify file purpose and evidence value accurately.
Role and responsibilities20Rows explain what the file does with enough detail for later discovery.
Behavior-bearing detection15Routes, services, models, jobs, tests, docs, and infra with behavior implications are not downplayed.
Related-file usefulness10Related paths help future agents navigate evidence clusters.
Large-file handling10Large or multi-role files include responsibilities or sections.
Resume and review state10Pending rows, review flags, and next actions are explicit.

Sampling contract

  • Repos with 100 or fewer manifest files evaluate every mapped row.
  • Large repos evaluate every row with a blocking review flag, every large or multi-role file, every entry point, and every behavior-bearing file identified by static signals.
  • Large repos also include a deterministic stratified sample across kind, domain, evidenceValue, source status, docs, tests, config, infra, generated files, assets, and support files.
  • When a sampled row fails, the next eval expands that stratum until the failure pattern is resolved.

Sandia Oil calibration gate

  • Sandia Oil is the first real calibration target because it is small enough for manual inspection while still carrying production-shaped behavior.
  • Calibration must identify obvious missed files, under-described behavior-bearing files, stale classifications, and downstream surface/capability hints before larger repo use.
  • When human-authored expected rows exist, manifest recall and critical behavior-bearing recall must be 100; row-quality score must be at least 96, with no normalized category below 9/10.

Acceptable means deterministic checks pass, total eval score is at least 96, no category is below 9/10 when normalized, no pending rows remain, no blocking review flags remain, current Sandia Oil calibration evidence exists, and no critical behavior-bearing file is materially misclassified. When a human-authored expected Artifact Inventory exists, it becomes an additional hard gate.

Strict-mode acceptable additionally means behavior-bearing rows resolve to capability IDs, spec section IDs, and verification targets or explicit test gaps. Bootstrap-mode acceptable may carry unresolved graph links only when downstream artifacts do not exist yet and review flags make the gap visible.

Co-evolution rule

Required: changes to Artifact Inventory schema, status model, manifest definition, graph-link model, lifecycle modes, checker behavior, or eval rubric must update the job, technical, and eval specs together.

When implementation exists, fixture outputs, Sandia Oil calibration evidence, and any human-authored expected Artifact Inventory must be updated in the same change as schema or rubric changes.
