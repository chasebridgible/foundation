Spec Selection Technical Spec

This technical spec defines the skill interface and durable handoff contract for routing spec work before authoring begins.

Status: active
Updated: 2026-06-06
Canonical section: #required-depth

Required depth

Spec Selection is a workflow interface, not an implementation-heavy feature. The technical contract must define what the skill reads, the exact output it produces, how fixtures are represented, how a future command can check the output, and how handoff reaches the type-specific authoring skills.

Internal sequence

  1. Load the spec-selection job spec, definitions, registry, and current system/capability owner when relevant.
  2. Restate the user intent without choosing a spec type.
  3. Search existing specs and paths for ownership.
  4. Run the standing-ability, triggered-work, parent capability, child capability, decomposition, and necessity tests.
  5. Emit one Spec Selection Result.
  6. Evaluate the result against the classification fixture set and the eval spec.
  7. Route to exactly one next authoring skill or name a blocker/non-spec owner.

Contracts

ContractRequired behaviorWhy
Skill entryskills/spec-selection/SKILL.md is mandatory before ambiguous spec work, spec splitting, remapping, or type routing.Agents must not jump from fuzzy intent to authoring.
Output shapeEvery run returns intent, existing owner, clarifying questions/answers/assumptions, primary classification, companion specs, parent/child/job relationship, rejected classifications, evidence, open questions, and next skill/process.Downstream authoring skills need a stable handoff instead of chat memory.
Question gateSpec Selection asks only questions that could change classification, owner, companion specs, or non-spec routing. If ambiguity is minor, it records a labeled assumption.Clarification should improve routing without turning every request into an interview.
Fixture shapeEach JSONL fixture has id, prompt, expectedClassification, reason, and commonWrongAnswer.Classification examples become reusable eval evidence.
Authoring handoffThe selected lane must be one of author-system-spec, author-capability-spec, author-job-spec, author-technical-spec, or author-eval-spec.No authoring skill decides its own spec type.
Non-spec resultArtifact, interface, process, action, evidence, and implementation-detail results must name the owning spec instead of creating a new spec file.Prevents artifact names and process labels from becoming fake capabilities.

Failure modes

FailureBlocked byRevision target
Job-shaped phrase accepted as a capability.Standing-ability and triggered-work tests.Spec Selection Result classification and rejected classifications.
Artifact, route, screen, or layer treated as a capability.Non-spec boundary rule.Attach to owning job, technical, or eval spec.
Vague parent capability hides unrelated children.Parent-capability and decomposition tests.Rewrite parent as one outcome family or split.
Authoring starts before routing is proven.AGENTS rule, spec-selection skill, and eval gate.Return to spec selection before editing specs.
Companion technical or eval spec is omitted.Companion-spec check in the output contract.Add companion spec or record explicit gap.
