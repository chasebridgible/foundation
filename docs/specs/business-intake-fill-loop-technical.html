Business Intake Fill Loop Technical Spec

Define the technical sequence and artifact contracts for converting intake sources into graph-compatible target specs and derived canvas outputs.

Status: draft
Updated: 2026-06-01
Canonical section: #required-depth

Required depth

This is a T4 workflow because it creates a new repeatable job and target-repo artifact set. It requires explicit source labels, stable IDs, graph-compatible specs, and evaluation evidence, but does not require a database, server, authentication layer, or automated extraction engine in v1.

Internal sequence

  1. Normalize input source paths and copy or reference them in the target repo.
  2. Create source registry rows before claim extraction.
  3. Extract answer rows with statuses, source refs, confidence, freshness, and suggested signals.
  4. Create target-owned specs with stable IDs and `graph-metadata` script blocks.
  5. Build graph JSON from target specs and render static canvas HTML from graph JSON.
  6. Optionally evaluate against an expected graph fixture with required nodes, edges, node-type counts, and canvas text.

Contracts

ContractRequirement
Source registryEvery material source has stable ID, source type, path or URL, date captured, owner/origin, trust level, freshness, access boundary, and related lanes.
Answer rowEvery extracted answer has lane/question ID, status, extracted answer, source refs, confidence, freshness, suggested signals, gaps, and promotion target.
Signal rowEvery promoted signal identifies type, stable ID, label, source refs, owning capability or job, confidence, and review state.
Target specEvery target spec has HTML-readable content, `spec-metadata`, `graph-metadata`, stable section IDs, and source references in prose or attributes.
Derived graphGraph JSON and canvas HTML are built from target specs and can be deleted/rebuilt without losing meaning.

Failure modes

  • Unsupported certainty: a claim appears in a capability or job without a source ref or explicit gap.
  • Hidden incompleteness: unanswered lanes are omitted rather than represented as gaps.
  • Graph drift: visible prose and `graph-metadata` disagree.
  • Source leakage: private source contents are copied into broad artifacts without respecting access boundaries.
  • Canvas as truth: derived graph/canvas files are manually edited instead of rebuilding from specs.

Paths and coverage

The skill is the implementation surface. Validation uses Foundation `spec:check` and visible business graph commands for Foundation, and graph check/build/render/eval commands for each target repo.
