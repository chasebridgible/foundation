Author Job Spec Job

This job authors or revises job specs after Spec Selection classifies the work as triggered work an actor performs.

Status: active
Updated: 2026-06-06
Canonical section: #job-intent

Job intent

This job turns a Spec Selection Result for triggered work into an HTML-native job spec with trigger, actor, process, context, states, rules, outputs, evidence, and revision target.

Capability supported

Supports foundation.author-durable-specs.capability by making work contracts concrete enough for humans and agents to execute without rediscovering intent.

Current handling

The retired generic interview model mixed spec-type routing with job authoring. This lane keeps job authoring narrow and starts only after routing is complete.

Proposed handling

The lane starts only from a Spec Selection Result whose primary classification is job. If the request names a durable ability, system goal, implementation contract, or eval contract, the lane returns to Spec Selection.

Process

  1. Read the Spec Selection Result, owning capability, job template, and any related technical/eval specs.
  2. Run the readiness gate: ask only questions that materially change trigger, actor, output, process, state, rule, evidence, or revision target; otherwise continue with labeled assumptions.
  3. Capture trigger, actor candidates, desired outcome, process, context, tools, interfaces, states, rules, edge cases, outputs, and exit criteria.
  4. Name technical and eval companions when implementation or durable judgment matters.
  5. Author or revise HTML prose, metadata, graph edges, coverage, and visible links.
  6. Run the job-spec eval, revise blocking findings, then run registry and graph checks.

Context, tools, and interfaces

  • skills/author-job-spec/SKILL.md is the agent procedure.
  • docs/specs/templates/job-spec-template.html is the authoring shape.
  • foundation.author-job-spec.eval is the quality gate.

Readiness questions: What triggers this job? Who or what performs it? What output or state proves it is complete? What states, rules, permissions, edge cases, or recovery paths matter?

Evidence and evaluation

The authored job spec is acceptable only when a downstream agent can perform the work from the spec, understand when it starts and ends, know what context and interfaces matter, and see how evidence will be judged.

Revision

Revise this job when agents write generic prose, omit trigger/state/rule/evidence details, classify a capability inside the job lane, or hide required technical/eval companions.
