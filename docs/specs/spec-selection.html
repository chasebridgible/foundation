Spec Selection Job

This job gives humans and agents a repeatable way to decide whether intent belongs in a system, parent capability, child capability, job, technical, or eval spec before authoring begins.

Status: active
Updated: 2026-06-06
Parent: foundation.author-durable-specs.capability

Intent

Foundation needs a stable discernment layer before it remaps capabilities or authors durable specs. The job is to classify the level of intent, choose the smallest durable spec set that preserves that intent, and reject artifact names, process labels, and implementation details as capabilities unless they are rewritten as reliable outcomes.

Trigger

This job starts when a human or agent is about to create, revise, split, merge, or remap a system goal, capability, job, technical contract, eval, skill, workflow, validator, backfill row, process/action map row, or durable spec artifact.

Classification contract

The classifier must answer one primary question first: what layer of intent is being named? It should then choose the primary spec type and any companion specs required to make the work executable and verifiable.

The hardest boundary is capability versus job. A capability is a durable ability or standing condition that remains available after work is done. A job is triggered work an actor performs to create, operate, or improve that ability.

QuestionClassification signalPrimary output
Is this the root contract for a whole system, product, repo, or durable subsystem?It names the overall goal, boundaries, capability families, operating loop, evidence model, and revision model.System spec.
Is this a broad standing ability or durable outcome family that child outcomes share?It describes one larger reliable result that remains possible across jobs, actors, and runs.Parent capability spec.
Is this a narrower standing ability or durable outcome inside a parent capability?It names a specific reliable condition or possibility that can have jobs, rules, evidence, and evals.Child capability spec.
Is this named work an actor performs to create, operate, or improve a capability?It has a trigger, actor candidates, required context, process, states, outputs, and exit criteria.Job spec.
Does implementation behavior matter?Software, data, automation, infrastructure, integration, permissions, sequencing, failure handling, or contracts must be specified.Technical spec, usually attached to a job.
Does the acceptance judgment need to be durable?The system must know what evidence proves success, what fails, and what gets revised.Eval spec.

The result can be one spec or a small spec set. A broad request may need a system spec plus capability specs. A build request usually needs a job spec, technical spec, and eval spec. A narrow process improvement may only need a job-spec revision if technical and eval details already exist.

Clarification gate

Spec Selection must ask before classification only when the answer could change the spec type, owner, companion specs, or non-spec routing. The agent should ask the smallest useful set of questions, no more than three at a time. If the ambiguity is minor, the agent may continue with a labeled assumption and include that assumption in the Spec Selection Result.

Question typeUse whenExample question
System altitudeThe request might change a whole-system goal or subsystem boundary.Are we defining the root system goal or a narrower part of the system?
Capability altitudeThe request might name what must become reliably possible.What durable ability or outcome should exist after the work?
Job altitudeThe request might name work an actor performs.What trigger starts this work, and what output closes it?
Technical altitudeThe request might be implementation-contract work.Which implementation contract or failure mode must be constrained?
Eval altitudeThe request might define evidence judgment.What evidence should be judged, and what would fail?
Non-spec boundaryThe request might be an artifact, interface, process, action, evidence item, or implementation detail inside an existing spec.Which existing spec owns this file, interface, process step, or evidence item?

Capability and job tests

Before anything is accepted as a parent or child capability, it must pass the standing-ability test. Before anything is accepted as a job, it must pass the triggered-work test. These tests are deliberately simple because future agents must apply them under ambiguity.

TestPasses whenFails when
Standing-ability testThe phrase survives: "the system, business, user, or agent can reliably..." or "this condition is reliably available..."The phrase mainly describes work being performed, a transformation being done, or a phase being completed.
Triggered-work testThe phrase survives: "an actor performs this work when..." and has a trigger, process, output, and completion condition.The phrase mainly describes a durable ability, stable condition, or ongoing possibility.
Parent-capability testThe capability is broad enough to contain multiple narrower standing abilities that share one larger purpose.The parent is just a folder, artifact group, lifecycle phase, org department, or process chain.
Child-capability testThe child is a narrower standing ability inside the parent and can have its own jobs, rules, evidence, or evals.The child is a step, verb phrase, queue row, screen, route, command, checklist, or report.
Decomposition testParent to child means broader outcome to narrower outcome.The relationship is phase to phase, artifact to artifact, interface to route, process to step, or job to action.
Necessity testA child capability exists because the parent is too broad to govern jobs, evidence, or rules directly.The parent can already own the jobs and evidence clearly without a child layer.
Classifier rule: if a candidate capability starts with verbs such as create, record, convert, run, review, update, implement, gather, map, author, evaluate, deploy, or revise, treat it as job-shaped until it can be rewritten as a standing ability or durable condition.

Decision process

  1. Restate the intent. Rewrite the request as the outcome or work the user wants, without choosing the spec type yet.
  2. Clarify only material ambiguity. Ask up to three altitude questions when the answer could change classification, owner, companion specs, or non-spec routing; otherwise proceed with a labeled assumption.
  3. Find existing ownership. Search the registry, system spec, capability map, and relevant job specs before creating anything new.
  4. Run the capability/job tests. Decide whether the request names a standing ability, a durable condition, or triggered work. Do this before choosing parent capability, child capability, or job.
  5. Choose the altitude. Decide whether the request is whole-system intent, parent capability, child capability, work job, implementation behavior, verification judgment, or a non-spec artifact/interface/process/action.
  6. Name the primary spec. Pick the spec type that owns the main intent. Prefer revising an existing owner over creating a new file.
  7. Add companion specs only when needed. Add technical specs when implementation contracts matter. Add eval specs when acceptance judgment must persist beyond the current conversation.
  8. Reject overloaded names. If the name is a layer, artifact, queue, route, screen, command, or phase, rewrite it as an outcome or classify it as an artifact/interface/process/action inside the owning job.
  9. Record the routing decision. The spec or later skill should explain why the selected level is right and what adjacent levels were intentionally not created.

Spec type rules

System spec

Use a system spec when the work needs a root contract for a whole operating system, product, repo, client system, or durable subsystem. A system spec owns the main goal, boundaries, parent capability families, operating loop, evidence model, and revision model. It should orient future work; it should not own every leaf file or process step.

Parent capability spec

Use a parent capability spec when the system needs to make a broad standing ability or durable outcome family reliable and that outcome family naturally contains multiple child capabilities. A parent capability should still be understandable as one larger thing the system can reliably make possible.

Child capability spec

Use a child capability spec when the outcome is narrower than a parent capability but still names a standing ability or durable condition. A child capability can own jobs, rules, evidence, and evals. It should not be a process step, artifact name, interface name, implementation layer, queue item, or verb phrase for work being performed.

Job spec

Use a job spec when an actor must perform named work so a capability can exist, operate, or improve. A job spec carries trigger, actor candidates, process, required context, tools, interfaces, states, rules, edge cases, outputs, evidence, and revision target. SOPs, checklists, process maps, handoff notes, and job records live inside the job spec by default.

Technical spec

Use a technical spec when software, data, automation, infrastructure, integration, security, permissions, sequencing, concurrency, migration, failure handling, or operational contracts matter. A technical spec explains how implementation should behave. It should attach to the job or capability whose intent it supports.

Eval spec

Use an eval spec when evidence needs a durable judgment contract. An eval spec names the evidence, passing condition, failure condition, review method, and revision target. Tests, metrics, screenshots, traces, audits, customer outcomes, and model judgments can produce evidence; the eval decides whether the evidence is good enough.

Non-spec boundaries

Not every important thing becomes its own spec. Many things are sections, views, evidence, or implementation details inside the owning spec.

ThingWhat it isWhere it usually belongs
ArtifactA durable object used, produced, owned, or referenced by the system.Owned paths, implementation paths, evidence tables, or context sections.
InterfaceThe boundary where an actor uses, observes, or depends on the system.Job context/interface section, technical contract, or evidence source.
ProcessThe repeatable path that performs a job.Process section inside the job spec, or a linked skill/SOP view.
ActionA concrete thing an actor does or decides.Process/action map, job process section, or technical sequence.
EvidenceObservable source-backed reality.Eval spec, coverage entry, review report, test output, or evidence table.
Implementation detailA code, data, infrastructure, or tool detail.Technical spec if durable; otherwise the implementation diff or note.

Eval path

The skill must not self-certify classification. It must compare the routing decision against the eval contract and fixture examples before handing off. The first eval is universal for spec selection; type-specific evals run after the selected spec type is known.

Eval layerPurposeWhen used
Spec-selection evalJudge whether the request was classified at the right layer: system, parent capability, child capability, job, technical, eval, artifact, interface, process, action, or evidence.Before creating or revising durable specs from fuzzy intent.
Capability/job boundary evalStress-test standing ability versus triggered work, parent versus child decomposition, and whether child capabilities are actually needed.Whenever a capability or job is created, split, merged, renamed, or remapped.
System spec evalCheck whether a system spec has one root goal, boundaries, capability families, operating loop, evidence model, and revision model.After classifying work as system-level.
Capability spec evalCheck whether a capability is an outcome statement with jobs, evidence, constraints, and parent/child links that decompose outcomes.After classifying work as parent or child capability.
Job spec evalCheck whether a job has trigger, actor candidates, process, context, states, rules, outputs, evidence, and revision targets.After classifying work as a job.
Technical spec evalCheck whether implementation contracts, sequencing, data, permissions, failure modes, and operational constraints are explicit enough to build.After classifying work as technical or technical companion work.
Eval spec evalCheck whether evidence, acceptance conditions, failure modes, judge method, and revision targets are clear enough to reject bad output.After classifying work as eval or eval companion work.

The universal spec-selection eval should run first because the wrong spec type makes every downstream adequacy eval less useful. Once classification is correct, the type-specific eval can judge whether the selected spec is strong enough.

Revision

This job spec changes when agents misroute spec work, when examples expose a boundary gap, or when templates/evals add new spec-type rules.

  • Revision target: this job spec first, then skills/spec-selection/SKILL.md, the type-specific authoring skills, templates, and validators.
  • Update trigger: ambiguous classifications, broad capability names, jobs misclassified as capabilities, or artifact/interface names passing as outcomes.
  • Promotion rule: mark this spec active only after the language, skill handoff, and classification eval are accepted or explicitly scoped with remaining gaps.
