Foundation File Registry Test Spec

The test layer is the eval system for the Foundation file registry: deterministic checks prove complete current coverage, graph checks prove links resolve, and quality evals prove rows are accurate enough to support agents during both backfill and steady-state build work.

Status: draft
Updated: 2026-05-26
Parent: foundation.backfill-file-registry.descriptive

If this evidence passes, a reviewer can trust that every repo-owned file has a current registry row, sampled rows describe source files accurately enough for downstream surface/capability inference, and behavior-bearing rows connect to the intended specs, capabilities, and verification evidence once those artifacts exist.

Commands or evidence required

  • npm run foundation:file-registry:init -- --repo /path/to/repo --run-id YYYYMMDD-NN --mode backfill - planned command that creates a manifest and skeleton registry. Backfill aliases may call this command but should not duplicate its logic.
  • npm run foundation:file-registry:check -- --repo /path/to/repo --run-id YYYYMMDD-NN --mode bootstrap|strict - planned command that verifies deterministic invariants, freshness, graph links, and handoff gate conditions.
  • npm run foundation:file-registry:eval -- --repo /path/to/repo --run-id YYYYMMDD-NN - planned command or skill wrapper that runs semantic row-quality evaluation.
  • npm run foundation:file-registry:graph-check -- --repo /path/to/repo --run-id YYYYMMDD-NN - planned command that resolves spec section IDs, capability IDs, verification targets, and explicit test gaps.
  • npm run foundation:file-registry:refresh -- --repo /path/to/repo --changed-only - planned steady-state command that refreshes changed rows and reports affected specs, capabilities, and tests.
  • Calibration report - Sandia Oil run output, manual review notes, and optional human-authored expected file rows when a fixture is created.

Behavior-bearing detection15Routes, services, models, jobs, tests, docs, and infra with behavior implications are not downplayed.

Sampling contract

  • Repos with 100 or fewer manifest files evaluate every mapped row.
  • Large repos evaluate every row with a blocking review flag, every large or multi-role file, every entry point, and every behavior-bearing file identified by static signals.
  • Large repos also include a deterministic stratified sample across kind, domain, evidenceValue, source status, docs, tests, config, infra, generated files, assets, and support files.
  • When a sampled row fails, the next eval expands that stratum until the failure pattern is resolved.

Sandia Oil calibration gate

  • Sandia Oil is the first real calibration target because it is small enough for manual inspection while still carrying production-shaped behavior.
  • Calibration must identify obvious missed files, under-described behavior-bearing files, stale classifications, and downstream surface/capability hints before larger repo use.
  • When human-authored expected rows exist, manifest recall and critical behavior-bearing recall must be 100; row-quality score must be at least 96, with no normalized category below 9/10.

Acceptable means deterministic checks pass, total eval score is at least 96, no category is below 9/10 when normalized, no pending rows remain, no blocking review flags remain, current Sandia Oil calibration evidence exists, and no critical behavior-bearing file is materially misclassified. When a human-authored expected registry exists, it becomes an additional hard gate.

Strict-mode acceptable additionally means behavior-bearing rows resolve to capability IDs, spec section IDs, and verification targets or explicit test gaps. Bootstrap-mode acceptable may carry unresolved graph links only when downstream artifacts do not exist yet and review flags make the gap visible.

Co-evolution rule

Required: changes to file registry schema, status model, manifest definition, graph-link model, lifecycle modes, checker behavior, or eval rubric must update the descriptive, technical, and test specs together.

When implementation exists, fixture outputs, Sandia Oil calibration evidence, and any human-authored expected registry must be updated in the same change as schema or rubric changes.
