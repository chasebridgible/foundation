Business Intake Fill Loop Evaluation

Evaluate whether a business intake run turns provided sources into a source-backed, graph-compatible, human-inspectable client substrate without hiding uncertainty.

Status: draft
Updated: 2026-06-01
Canonical section: #verification-contract

Verification contract

An intake run is acceptable when every material source is registered, promoted capabilities and jobs are backed by source refs or explicit gaps, target specs pass graph validation, the canvas renders from derived graph JSON, and the handoff identifies remaining unresolved decisions.

Acceptance mapping

Intake honestyjob#processAnswer rows show answered, partial, conflict, stale, and unanswered states rather than implied completeness.

Revision pathjob#revisionRemaining gaps name the source, capability/job affected, and next review action.

Coverage plan

Use a target repo pilot as the primary evaluation surface. For deterministic pilots, include `expected-graph.json` with required nodes, required edges, minimum node-type counts, and canvas text that must appear.

Commands and evidence

  • npm run spec:check after Foundation spec or skill metadata changes.
  • npm run foundation:visible-business-graph:check -- --repo . after Foundation graph metadata changes.
  • npm run foundation:visible-business-graph:check -- --repo <target-repo> after target graph metadata changes.
  • npm run foundation:visible-business-graph:eval -- --repo <target-repo> --graph ... --canvas ... --expected ... when an expected graph fixture exists.

Gaps and follow-ups

V1 is manual-agent-driven. Future revisions may add deterministic source-registry and answer-row validators, but those should follow pilot evidence rather than precede it.
