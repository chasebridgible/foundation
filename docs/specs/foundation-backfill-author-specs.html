Foundation Backfill Author Specs

Author Specs creates architecture-agnostic job specs and contract-first technical specs from queued slices, Context Packs, and process/action rows. This layer answers: What job and technical specs make this slice rebuild-grade?

Status: active
Updated: 2026-06-04
Upstream: Process / Action Map
Next: Evaluate Job Slices

This layer is a reproducible unit in the backfill substrate. It initializes from the previous layer, selects one Author Specs target with --next, authors one target-owned job/descriptive spec and one contract-first technical spec, checks deterministic structure, writes row-level JSONL eval receipts, revises that same target until outstanding, and repeats until --next returns no target before recording handoff state for the next layer.

Each job/technical spec pair must preserve the material content of its current Process / Action Map row: actor or role, trigger, intended outcome, domain object, actions, states and transitions, permissions, rules, visible/operator behavior, edge cases, recovery paths, evidence refs, explicit gaps, blockers, and human decisions. Broad prose that compresses named upstream particulars into generic categories is not outstanding.

GBrain's source-of-record lesson applies directly: canonical append-friendly artifacts own state; readable HTML summaries and report tables are derived views. Later agents cite row IDs and evidence links instead of relying on inherited conversation memory.

Small-loop journey

  1. Initialize: run foundation:author-specs:init only after Process / Action Map check, eval, summary, and report state prove the next layer is Author Specs.
  2. Select target: run foundation:author-specs:fill -- --next and use only the returned Author Specs target.
  3. Author: read the target Process / Action Map row and its bounded Context Pack evidence, then write or revise exactly one job/descriptive spec and one technical spec for that target.
  4. Record: run foundation:author-specs:fill -- --process-map-id <id> --job-spec <path> --technical-spec <path> for that target only.
  5. Check: run foundation:author-specs:check -- --phase batch; repair structural failures before eval.
  6. Evaluate: run row-targeted foundation:author-specs:eval; warnings, missing row receipts, omitted row material, generic compression, weak rebuild readiness, or revision targets are not outstanding.
  7. Revise: keep the same target until the row-level receipt is current, score 100, and outstanding with no warnings, blockers, or revision targets.
  8. Continue: a passing row receipt means select the next target; it does not mean the layer can hand off.
  9. Handoff: run handoff check, all-row eval, and report only after --next returns no target.

Checker and eval contract

MechanismContract
Checkervalidates Author Specs JSONL shape, upstream Process / Action Map freshness, one row per active process row, target-owned job and technical spec files, spec metadata, graph metadata, owned paths, canonical sections, required job language, required technical contract language, and report state.
Eval receiptscores each Author Specs row against its upstream Process / Action Map row and authored specs for upstream traceability, job intent coverage, technical contract coverage, graph traceability, adequacy readiness, material row fidelity, spec-only rebuild readiness, and generic-compression rejection. Eval output is canonical JSONL; HTML is a derived summary for human review.
Sandia Oil calibrationThe first real calibration target is Sandia Oil. Its output is not a gold benchmark unless a human-reviewed expected Artifact Inventory is created; it is used to catch obvious misses while context is tractable.

Handoff gate

Each active Process / Action Map row has one Author Specs row with target-owned job and technical specs or a named blocker; checker passes; every non-pending row has a current outstanding row-level eval receipt; --next returns no target; the all-row eval summary is outstanding; report names next layer as Evaluate Job Slices.

  • Canonical artifacts exist and share a run ID.
  • All rows are ready-for-slice-eval or explicitly blocked with named blocker detail.
  • Checker passes for structure, references, currentness, spec metadata, graph metadata, and report state.
  • Eval JSONL and HTML summary exist; warnings and revision targets are resolved rather than carried forward.
  • Active report records counts, artifact paths, checker/eval status, run-log sequence, and next layer.
