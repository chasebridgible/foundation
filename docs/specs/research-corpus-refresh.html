Research Corpus Refresh

Recurring research work becomes reliable when every run has a bounded source universe, durable machine-readable records, a readable share artifact, validation, and a reviewed handoff.

Status: active
Updated: 2026-06-14
Canonical section: #job-intent

Job intent

The Research Corpus Refresh job turns one-time or scheduled research into a repeatable agent workflow. It exists for research domains where people need a human-readable artifact, future agents need structured state, and every change should remain source-backed and reviewable.

Trigger

The job starts when a user asks to refresh a research corpus, when a schedule fires, or when a downstream repo needs to promote ad hoc research into durable JSON/JSONL, workbook, brief, and review artifacts.

Current handling

  • Current actor: human-directed Codex agents, often in project repos.
  • Current path: project-specific systems such as the Foundation Agent Capability Scout and Tribal Conference Research use useful pieces of the pattern, but the generic workflow is not yet named as reusable Foundation substrate.
  • Known gaps: ad hoc workbooks can live outside repos, agents can confuse dated events with ongoing routes, and scheduled refresh behavior is easy to re-prompt inconsistently.

Proposed handling

  • Proposed actor: a single research worker agent or scheduled Codex automation, with optional reviewer/evaluator agent only when the project spec calls for it.
  • Proposed path: the project defines a source universe, corpus schema, workbook view, checker, and review-first publish path; the worker processes one bounded target at a time until the run reaches terminal coverage.
  • Reason for change: strong research should not depend on remembered prompts, temporary files, or manual workbook cleanup.
  • Decision state: accepted as the reusable Foundation pattern for Tribal research systems and future similar corpora.

Process

  1. Define corpus: name the project repo, research goal, in-scope categories, out-of-scope categories, source families, schema, workbook views, and required checks.
  2. Start run: create or reuse a clean branch, generate a run ID, snapshot source registry state, and record trigger, deadline, cadence, and expected artifacts.
  3. Select one target: choose one source, region, facility, query group, or other bounded work unit from durable state.
  4. Research target: use source-backed public evidence, preserve exact URLs and search terms when search drove the discovery, and classify each candidate before adding it.
  5. Record state: merge findings, rejections, source statuses, coverage notes, and route or contact records into JSON/JSONL without relying on workbook formatting as source of truth.
  6. Export view: regenerate the human-readable workbook and dated brief from the canonical corpus, keeping internal next actions out of share-first sheets unless the project spec explicitly includes them.
  7. Check and revise: run deterministic validators, workbook readback, scope checks, source URL checks, dedupe checks, and project-specific evals; revise until no blocking findings remain.
  8. Publish for review: commit, push, open or update a PR, and notify the owner with changed rows, blockers, workbook path, run status, and requested action.
  9. Schedule next run: enable or update automation only when the project spec or operator-approved gate allows it, using review-first PR behavior by default.

Revision

If a refresh run produces messy workbook rows, loses source coverage, misclassifies scope, or stops without durable handoff evidence, revise the smallest owning layer: project source registry, project schema, workbook exporter, checker, eval, skill, or this reusable Foundation job. Do not patch only the next prompt when the failure will recur.
