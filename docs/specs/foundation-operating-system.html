System intent

Root system goal

Foundation helps humans and agents accomplish goals by turning intent, knowledge, evidence, and changing reality into a durable operating system of capabilities, jobs, context, actions, evaluation, and revision loops, so the right work can be understood, executed, improved, and reused over time.

What the system exists to make possible

Foundation is the shared operating substrate for agent-assisted business and software work. Its job is not to collect documents. Its job is to make goals, knowledge, repo state, business reality, and agent work inspectable and improvable by preserving intent, evidence, work contracts, implementation latitude, checks, and learning loops in durable HTML specs and executable skills.

A useful Foundation change compounds when a future human or agent can discover the goal, understand the capability that must exist, load the right job, use bounded context, run the relevant check, revise the smallest wrong artifact, and improve the substrate without relying on conversation memory.

Strategic direction

Strategic goal: use Foundation to become the best system for building agent-powered workflows, software, and operating knowledge that help businesses grow.

The root goal is broader than one business use case. The first strategic arena is agent systems for business growth because that is where durable goals, knowledge, repo evidence, jobs, evals, and revision loops can create compounding leverage.

Operating principles

  • Stay current. Foundation must improve as tools, models, evidence, and business reality change.
  • Separate possibility from work. Capabilities name what must become reliably possible. Jobs name the work actors do to create, operate, or improve those capabilities.
  • Preserve intent through evidence. Facts, guesses, uncertainty, source evidence, and human decisions must stay distinguishable.
  • Make context durable and bounded. Agents should get the right context for the current job without rediscovering the whole source.
  • Improve through revision loops. When work misses the goal, revise the smallest wrong artifact so the next run starts from a better substrate.

Capability map

Foundation capabilities should be named as reliable outcomes that a human, agent, software system, or business can understand. The parent capability map below is the durable intent structure. The implemented capability specs that follow are the current graph-backed child capability contracts, job-backed capability contracts, or explicitly scoped capability gaps under that parent structure.

Vocabulary source: capability, parent capability, child capability, job, process, action, artifact, interface, evidence, and eval terms are defined in Definitions - System philosophy concepts. Those definitions govern future capability remapping until a more specific spec-selection contract replaces them.

Operating loop

  1. Capture intent and sources without pretending the intake is complete.
  2. Inventory artifacts and map surfaces so agents know what exists and what behavior is visible.
  3. Extract capability, job, actor, process, tool, evidence, metric, and gap signals.
  4. Promote reviewed signals into capability, job, technical, and eval specs.
  5. Run structural checks, semantic evals, and visible graph rendering.
  6. Fold accepted lessons back into specs, skills, templates, and validators.

Spec graph

The graph must show one Foundation system node, the core capability nodes, and child jobs with their skills, scripts, tests, evals, technical contracts, source artifacts, and known gaps. System nodes use contains edges for capabilities, capability nodes use realized-by edges for jobs, and job nodes use supports edges back to their owning capability. The canvas is derived output; the source of truth remains HTML specs plus embedded metadata.

Authoring rule: before creating a spec, skill, validator, or durable workflow, identify which Foundation capability it improves and whether an owning job spec already exists. A job is not mapped until both the registry relationship and graph metadata expose the capability/job relationship.
