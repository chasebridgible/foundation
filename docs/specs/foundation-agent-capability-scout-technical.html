Agent Capability Scout Technical Spec

This is a T4/T5 architecture contract because the scout is scheduled, stateful, source-facing, git-mutating, notification-bearing, and intended to update Foundation memory and principles over time. The first implementation should be small, but the contracts must prevent unbounded research, unmerged memory, noisy findings, and unsupported doctrine edits.

Non-goals

The first scout is not a general web monitor, not an autonomous production upgrader, not an experiment planner, and not a replacement for owner judgment.

Internal sequence

  1. Trigger: Codex automation starts a run every other day at 5am with a run ID, source registry version, grading rubric version, principle-candidate rubric version, and notification target.
  2. Prepare: create or enter a clean Foundation branch/worktree and verify durable artifact paths are writable before fetching; if a clean Codex worktree is detached, create a dated scout branch and treat the run as publishable after branch creation.
  3. Validate: load source registry and reject disabled or out-of-scope sources.
  4. Fetch: retrieve source documents, pages, feeds, or changelog entries and store snapshots or content hashes with retrieval metadata.
  5. Diff: compare against prior successful run state and record observed additions, removals, or changed summaries.
  6. Normalize: write candidate finding records with evidence path, confidence, source family, topic, and 1-10 interest grade.
  7. Rank: sort findings by interest grade; low-interest findings remain queryable but do not require owner attention.
  8. Promote: for high-durability findings, write principle-candidate records and patch the relevant principles document when the candidate can be stated as a durable rule.
  9. Publish: complete a write-through artifact checkpoint before checks: append JSONL run, source-snapshot, finding, and principle-candidate artifacts for the same run ID, write a dated brief, keep any principles-doc patch tied to a candidate row, run Foundation checks, and record commit/PR/merge state. Evidence snapshots without matching JSONL rows and a brief are not publishable run state.
  10. Observe: log run metrics such as sources checked, failures, findings, principle candidates, top grade, duration, merge status, and notification status.

Boundaries and abstraction level

  • Owns: scout source registry contract, JSON/JSONL run artifact contract, finding and interest-grade contract, principle-candidate contract, merge receipt contract, and GitHub notification receipt contract.
  • References: Codex automations as the preferred wake-up surface, GitHub protected flow as the memory closeout and review-comment surface, and email as an optional fallback notification channel.
  • Does not own: follow-up experiment branches, production agent-harness implementation, or external source content.
