Foundation Backfill Process / Action Map Technical Spec

This technical layer makes Foundation Backfill Process / Action Map deterministic enough for scripts and structured enough for agent judgment.

Status: active
Updated: 2026-06-07
Parent: foundation.backfill-process-action-map.job

Implementation must preserve the source-of-record split: JSONL is canonical for layer rows and eval receipts, while HTML summaries are generated for human reading. Each command must be resumable from artifacts and must not rely on chat memory.

Contracts

ContractRuleReason
Upstream completenessThe layer starts only from a passing Context Pack handoff: current checker artifact, current passing eval receipt, zero revision targets, HTML summary, and optional report state naming Process / Action Map. If a report path is supplied and the embedded Context Pack state is stale or missing, init fails with the exact foundation:context-pack:report refresh command.Later synthesis must not hide earlier discovery gaps, and agents should receive deterministic recovery when canonical artifacts and report state drift apart.
Canonical JSONLLayer rows and eval receipts are JSONL; HTML summaries are derived.Agents need durable machine state and users need readable review surfaces.
Row-atomic fill loopfill accepts exactly one --pack-id or exactly one --slice-id, rejects anything other than the current --next target, rejects multiple IDs, rejects multi-row --processes-json, rejects --all, rejects --batch-size, rejects generated process files, and rejects filling another Context Pack row while any current non-pending row lacks an outstanding row eval receipt.Long-running work survives context compaction and prevents coarse closure or moving past an under-reviewed row.
No driver synthesisAgents must not create driver scripts, shell loops, generated payload files, regex classifiers, path-kind maps, or reusable templates that create, fill, check, evaluate, or revise more than the current --next row.The unit of review is the selected Context Pack row, not a bulk generator that happens to submit one payload at a time.
Deterministic checkervalidates one current row per active Context Pack, valid enums, current upstream fingerprints, required ready-row fields, trace evidence, zero duplicate row IDs, and optional report-state drift. Row-targeted eval requires a current passing check artifact for the latest Process / Action Map fingerprint.Structural drift should fail mechanically before semantic evaluation.
Semantic evalsupports row-targeted eval by pack ID, slice ID, or process row ID; a row receipt is outstanding only with score 100, no blocking findings, no warnings, no revision targets, current row fingerprint, resolved or named uncertainty, and enough behavior for Author Specs without reopening source for core behavior.Correctness includes meaning, specificity, evidence quality, and downstream usability.
Handoff gateall active Context Pack rows have ready-for-specs rows or explicit blocker detail; every process row preserves child/sole capability refs and rejects parent, needs-split, blocked, or non-queueable refs as active work; checker passes; eval receipt and HTML summary are current; every non-pending row has a current outstanding row eval receipt; eval has zero revision targets, zero blocking findings, and zero warnings; report names next layer as Author Specs.Each layer is a reproducible outstanding unit before the next layer consumes it.
Report self-checkfoundation:process-action-map:report writes or updates the report, validates the embedded Process / Action Map state, writes a current check artifact with the report path, rebuilds report state from that check, and rewrites the report.The report/check ordering should not require manual bootstrap knowledge or leave stale report state after handoff recording.

Checker and fill loop

  1. Read active report, run log, upstream artifacts, and current layer artifact.
  2. Select the next row with foundation:process-action-map:fill -- --next; the selected row is the only valid fill target.
  3. Inspect exactly one Context Pack row and only the evidence needed for that row; do not create or run a generator across many packs.
  4. Update canonical JSONL with foundation:process-action-map:fill using one pack or slice ID and one process JSON object.
  5. Run foundation:process-action-map:check -- --phase batch; repair structural failures before semantic eval.
  6. Run row-targeted foundation:process-action-map:eval; generate or refresh canonical row receipts and the HTML summary from JSONL.
  7. Revise the same row until it is outstanding, then select the next row.
  8. After every current row is outstanding or explicitly blocked, run handoff check, full eval, and foundation:process-action-map:report; the report command refreshes the check artifact against the embedded report state.
