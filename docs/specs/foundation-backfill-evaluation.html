Backfill evaluation process

Foundation evaluates backfill results with an LLM judge workflow that compares generated artifacts to a golden example, scores semantic quality, and produces a concrete revision queue before the work is called good enough.

Status: active
Updated: 2026-05-20
Golden example: foundation.backfill-golden-example

Mechanical checks prove the substrate is wired. Evaluation proves the produced specs are useful: specific, traceable, architecture-flexible, and strong enough for a future build agent to understand intended behavior from specs alone.

Evaluator journey

  1. Calibrate: read the golden example and this process spec.
  2. Load artifacts: read target AGENTS.md, the dated backfill report, inventory ledger, top-level app spec, and draft slice specs.
  3. Spec-only pass: review generated specs first and judge what a future build agent could understand from the specs alone.
  4. Evidence audit: sample original evidence for high-risk claims, UX-visible behavior, architecture constraints, and unresolved questions.
  5. Score: assign rubric scores with citations to report rows, spec sections, and evidence paths.
  6. Report: create or update docs/specs/backfill/evaluation-report-YYYYMMDD-NN.html using the same run ID as the backfill report.
  7. Route revisions: append the evaluation result to the backfill report and route each revision item to the owning backfill skill layer.

Scoring rubric

Each category is scored from 0 to 3: absent, shallow, usable, or good enough.

Evaluation report contract

  • Header: run ID, target repo, evaluator date, evaluated report path, spec graph summary, and overall result.
  • Scorecard: category scores from 0 to 3 with citations.
  • Slice findings: findings grouped by slice and routed to inventory, user-flow, descriptive, rendered UX, technical, or adequacy skills.
  • Architecture flexibility: findings on required contracts, current evidence, architecture constraints, and implementation latitude.
  • Spec-only rebuild probe: what a future build agent could build or would still need to know from specs alone.
  • Revision queue: concrete next actions with owner skill and affected artifacts.

Good enough gate

A backfill result is good enough for human review when every rubric category scores at least 2, spec-only rebuild readiness scores at least 2, and all remaining revision items are isolated in the evaluation report. A score below 2 routes work back to the owning backfill layer before final handoff.
