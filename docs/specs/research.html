Research Workflow

A research agent works best when it knows what to search for, where to look, where to save what it finds, how to check its work, and how people will read the final result.

Status: active
Updated: 2026-06-20
Main section: #job-intent

What this is for

This job makes research repeatable. Instead of leaving facts in chat, the agent saves what it finds, checks the work, and shows the results in a view people can use.

Trigger

This starts when a person asks for research, when a schedule starts a run, or when old chat research needs to become saved project data.

How this works now

  • Who does it now: people ask Codex agents to do research, often inside project repos.
  • How it happens now: some projects already use parts of this pattern, but the simple shared version was not written down clearly enough.
  • What goes wrong: research can end up in chat, temporary files, messy spreadsheets, or confusing rows unless the process is explicit.

A better way

  • Who does it: one research agent does the work. A reviewer can help when the project needs one.
  • Path: answer the setup questions, make a search plan, save each finding, check the work, publish it cleanly, and then show people the final view.
  • Reason for change: good research should not depend on memory, temporary files, or manual spreadsheet cleanup.
  • Decision: use this as the shared Foundation pattern for Tribal research systems and similar work.

Process

Questions to answer before the run
1
What are we trying to find?

Say what counts as a good answer.

2
Who will use it?

Name who reads the result and what decision they need to make.

3
Where should it look?

List the websites, regions, source types, and search words to try.

4
What should it skip?

Name things that are off topic, duplicates, too weak, or only worth watching.

5
What proof is enough?

Decide which links, quotes, dates, or records prove a fact is real.

6
What matters most?

Say how to rank results so the best items rise to the top.

7
How often should it work?

Choose if this is one time, monthly, or on another schedule.

8
When is it done?

Say what must be checked before the agent can stop.

9
What should it save?

Say which facts, links, notes, checks, and final views must be kept.

10
Who checks it?

Name who reviews, publishes, gets notified, or accepts blockers.

How the search should work
1
Start with the known list

Use the project source list first. These are the places we already trust or know matter.

2
Open one real source

Check one official page, calendar, directory, PDF, portal, or facility at a time.

3
Use web search for missing doors

Google-style search is a way to find leads. It is not proof by itself.

4
Use special places when needed

LinkedIn, event calendars, procurement portals, news pages, and association sites are useful for some jobs.

5
Use APIs or crawlers for scale

A tool like Firecrawl or an official API helps when many pages must be fetched the same way.

6
Save the outcome

Record found, rejected, no result, blocked, or needs recheck before moving to the next link.

The rule is simple: search tools help the agent find doors, but the saved answer should point to the best source page we can inspect. The agent should not skim a pile of links and summarize from memory. It should work one target at a time, save what happened, then choose the next target.

The repeat loop
Repeat
One target at a time
The agent keeps working until every required area is done or a real blocker is written down.
1
Pick one target

Choose one website, region, facility, search group, or other small task.

2
Find proof

Use public sources, keep exact links, and save the search words that found useful results.

3
Save the decision

Mark each item as accepted, rejected, duplicate, blocked, or worth watching.

4
Check and fix

Run the checks, fix bad rows, and only then publish or show the final view.

What must be true before it is done
Scope is clear
The agent knows what counts, what does not count, and which places must be checked.
Proof is saved
Every accepted item has a link people can inspect, plus search words when search found it.
Publish path is clean
Changed files, checks, PR or merge notes, blockers, and cleanup notes are complete.
The view is readable
The final view is easy to read, and internal notes stay out of the front page.
Schedule only after review
Do not turn this into an automatic recurring job until the project rules or owner say it is ready. Review-first PRs are the default.
next run starts from saved data

One GitHub setup that works

This is an example, not the only possible setup. It works well when the research lives in a repo and the owner wants saved data, readable files, checks, and a review trail.

What the bot does
1
Start clean

Run in a clean checkout or worktree, make a dated codex/ branch, and write down the run ID.

2
Save real changes

Update the saved records, logs, readable view, dated brief, and receipts. Do not only change the spreadsheet by hand.

3
Open a PR

Push the branch and open or update a pull request so GitHub can show what changed.

4
Wait for checks

Run project checks locally and wait for GitHub checks before calling the run clean.

5
Merge only safe runs

Merge routine data, workbook, and brief changes only when the project allows it and checks pass.

6
Tell the owner

Comment on the PR or send the configured message with changed rows, blockers, final links, and the requested action.

What to set up in GitHub
1
Protect main

Require a pull request and the project checks before changes can land.

2
Give the bot access

Use a GitHub App or token that can push branches, open PRs, comment, read checks, and merge when allowed.

3
Add the checks

Run the project research checker, tests, workbook readback, and spec checks when specs changed.

4
Choose merge rules

Let routine research data merge after checks, but keep scope, skill, and spec changes open for owner review.

5
Choose messages

Decide where the owner gets notified: PR comment, issue comment, GitHub mention, email bridge, or another receipt.

6
Record blockers

If checks, auth, merge rules, or notifications fail, save the exact blocker instead of pretending the run finished.

When to fix the system

If a run makes a messy view, skips sources, labels things wrong, or stops without proof, fix the smallest part that caused it: the source list, save rules, file maker, checker, grading rule, skill, or this shared job. Do not just patch the next prompt if the same mistake will happen again.
