Foundation Backfill Surface Registry

Surface Registry converts eligible File Registry rows into capability surfaces: routes, screens, APIs, commands, jobs, tables, workflows, infra resources, active behavior docs, and direct external dependencies. This layer answers: What things does this repo expose or depend on?

Status: active
Updated: 2026-05-28
Upstream: File Registry
Next: Capability Matrix

This layer is a reproducible unit in the backfill substrate. It initializes from the previous layer by selecting only files that can plausibly define, expose, configure, or materially document capabilities, fills one eligible upstream file at a time after a complete file read, checks deterministic structure, writes canonical JSONL eval receipts, derives a readable HTML summary, revises failed rows, and records handoff state before the next layer begins.

GBrain's source-of-record lesson applies directly: canonical append-friendly artifacts own state; readable HTML summaries and report tables are derived views. Later agents cite row IDs and evidence links instead of relying on inherited conversation memory.

ScopeFile Registry remains exhaustive. Surface Registry only queues code, runtime/config/deploy files, and active behavior-defining docs. Inert artifacts such as images, PPTX exports, spreadsheets, RTF/CSV/sample fixtures, generated files, lockfiles, archives, delete candidates, and support-only docs are skipped and may be cited only from the eligible file that consumes or defines behavior.
Core rowsurface row with stable surfaceId, run ID, surfaceKind, source category, upstream file IDs, upstream content hashes, simple full-file-read evidence, exposed object, operation, consumer/actor hints, state/rule hints, support reason, status, confidence, and any blocking review flags.

Surface kinds are route, screen, api, command, job, table, workflow, infra-resource, doc, test, generated-artifact, external-dependency, and support-classification. Capability IDs do not belong in this layer. Some legacy surface kinds remain valid for explicit rows, but skipped files are not read or marked merely because they exist in the repo.

Small-loop journey

  1. Initialize: create the canonical Foundation Backfill Surface Registry artifact from Surface Registry-eligible rows in a passing File Registry checker and eval handoff.
  2. Select file: choose exactly one pending or failed eligible upstream File Registry row using deterministic order plus review priority.
  3. Read: agent reads the complete eligible source file named by that row. File Registry kind, path patterns, imports, grep output, generated summaries, and helper-script output are hints, not authority for eligible rows. Inert artifacts are skipped rather than read.
  4. Fill: agent immediately marks every surface for that one file, or records an explicit support classification, with upstream file row IDs, upstream hashes, agent-read-full-file evidence, surface kind, actor or consumer hints, operations, and state/rule hints. The fill command accepts only inline rows for that one reviewed file; generated row files and batch fill modes are outside the process.
  5. Check: run the deterministic checker and record result in the active report and run log.
  6. Evaluate: write canonical JSONL eval receipts and a readable HTML summary.
  7. Revise: route failed rows back to the fill loop with row IDs and findings.
  8. Handoff: advance only when the handoff gate passes or named human decisions block the layer.

Checker and eval contract

MechanismContract
Checkervalidates the upstream File Registry handoff, surface JSONL parsing, surface kind/source/status enums, stable surface ID uniqueness, upstream file row references, upstream content hashes, concrete full-file evidence, scope eligibility, required fields, eligible file coverage, terminal handoff states, blocking flags, and optional report state drift.
Eval receiptsamples surface rows against file registry rows to judge whether exposed and dependent surfaces were missed, over-merged, mislabeled, weakly supported, too vague for Capability Matrix, or detached from full-file evidence. Eval output is canonical JSONL; HTML is a derived summary for human review.
Sandia Oil calibrationThe first real calibration target is Sandia Oil. Its output is not a gold benchmark unless a human-reviewed expected registry is created; it is used to catch obvious misses while context is tractable.

Handoff gate

All Surface Registry-eligible file rows are resolved to surfaces, explicit support classifications, or blocking review rows; inert File Registry rows are skipped; checker passes; eval receipt and HTML summary exist with no revision targets; report names next layer as Capability Matrix.

  • Canonical artifacts exist and share a run ID.
  • Every terminal row has concrete agent-read-full-file evidence for its upstream file; generic evidence that only says the file was read is not accepted.
  • All Surface Registry rows are in valid terminal states: ready-for-capability for surfaces and mapped for support classifications.
  • Checker passes for structure, references, currentness, and required fields.
  • Eval JSONL and HTML summary exist, score at least 96, have no normalized category below 9/10, and list zero revision targets.
  • Active report records counts, artifact paths, checker/eval status, run-log sequence, and next layer.
  • Capability Matrix does not start unless the report state says capability matrix or a named human decision blocks or defers the layer.
