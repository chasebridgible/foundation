Software Design Principles

Durable principles for user-facing software design in Foundation. These principles turn design quality into substrate: they shape descriptive specs, rendered HTML-native intent artifacts, implementation review, and visual verification across web, mobile, dashboards, content pages, and cross-platform apps.

Format: HTML-native
Scope: user-facing software design
Status: Canonical HTML

Foundation position

Design principles are substrate when they change future work. Foundation promotes a design rule only when it survives model churn, framework churn, and design-system churn; improves what agents generate; and gives the system something concrete to verify later.

Design quality is not aesthetic preference. The durable failures are human-factor failures: weak signifiers, unclear hierarchy, hidden state, excessive recall burden, broken focus order, inadequate contrast, undersized targets, incoherent reflow, or layouts that fit pixels while losing meaning.

Visible features need experience contracts. For user-facing work, descriptive specs define the intended experience: what users see, what they can do, what state means, how the interface responds, and how the experience adapts across device and modality.

Operating rule

A visually meaningful feature is ready for implementation only when the descriptive spec gives enough hierarchy, state, copy, responsive, accessibility, and rendered-experience detail for another agent to rebuild the intended user experience without relying on the original implementation.

Interface principles

These are the software-design rules agents should apply when creating or reviewing user-facing specs, rendered UX, and implementation evidence.

01
Hierarchy matches task priority
A screen should show what matters first, second, and third through constrained contrast, scale, grouping, and placement. One dominant purpose anchors the screen.
02
Structure comes before decoration
Semantic grouping, reading order, alignment, and stable spatial relationships carry the experience. Visual treatments earn their place by clarifying hierarchy or affordance.
03
Action and state are signified
Users can tell what is actionable, what is selected, what is loading, what changed, and what happens next from visible cues at the point of use.
04
Recognition beats recall
Options, locations, labels, values, and recovery paths stay visible or easily retrievable. The interface carries memory instead of requiring the user to carry it.
05
Complexity is staged
Primary tasks stay present; advanced filters, secondary controls, extra metrics, and rare settings appear through clear progressive-disclosure affordances.
06
Consistency compounds
The same meaning uses the same vocabulary, component treatment, state pattern, and platform convention across screens, reducing cognitive load over time.
07
Accessibility is correctness
Contrast, visible focus, logical focus order, descriptive names, labels, roles, states, reflow, and target sizing are correctness constraints for visible software.
08
Copy is design material
Headings, labels, button names, empty states, error messages, confirmations, and recovery guidance orient users and belong in the design contract.
09
Responsive adaptation preserves meaning
Breakpoints change navigation, density, grouping, and task flow so the screen still communicates the same intent; fitting pixels is not enough.
10
All visible states are designed
Ready, loading, empty, partial, success, error, permission-denied, offline, and completed states are specified where they affect user understanding or action.
11
Platform parity is task-level
Product intent, domain vocabulary, and state model stay equivalent across platforms while navigation chrome, density, pointer, keyboard, and focus behavior adapt.
12
Experience verification is load-bearing
Code structure alone cannot certify UI quality. Rendered evidence confirms hierarchy, spacing, overflow, states, accessibility-sensitive behavior, and responsive meaning.
Principle family Healthy output Review signal
Hierarchy and structure The screen has one dominant purpose, clear grouping, stable alignment, and a small set of deliberate text styles. Review the rendered screen and identify the first, second, and third priority elements without reading implementation code.
Action and state Controls look actionable; state changes are visible; default, hover, focus, pressed, disabled, loading, success, and error treatments exist where relevant. Exercise the main interaction states and confirm the user can tell what happened and what can happen next.
Recognition and disclosure Important options and labels stay visible; advanced controls are staged behind clear affordances; field labels and navigation names carry user language. Walk the flow from a cold start and confirm the user is oriented without remembering hidden choices from prior screens.
Accessibility and semantics Names, roles, states, headings, labels, contrast-sensitive elements, focus behavior, and target sizing are specified and verified. Keyboard, screen-reader semantics, visible focus, contrast, and target-size evidence support the intended experience.
Responsive meaning Mobile, tablet, desktop, and wide layouts preserve task order and meaning while adapting density and navigation patterns. Rendered evidence at intended breakpoints shows coherent hierarchy, no meaningful overflow, and no hidden primary action.

Platform interpretation

Mobile apps
One dominant purpose per screen
Small screens favor clear top-level navigation, touch-safe controls, short labels, staged complexity, and focused task flow. Dense control matrices and tiny adjacent targets weaken the experience.
Web apps and SaaS
Scanability, orientation, and reversibility
Professional tools can carry density when page regions, selection states, filters, summary-detail structures, and recovery paths stay predictable.
Dashboards
Dashboards are for decisions
Status, change, anomaly, and next action matter more than ornamental metrics. Grouping should match operational domains and de-emphasize metrics that do not change decisions.
Content and product pages
Narrative clarity with constrained emphasis
Strong reading order, concise headings, restrained typography, meaningful sections, and one dominant call to action per narrative step carry the page.
Cross-platform apps
Task-level parity, platform-level adaptation
Intent, vocabulary, state model, and outcomes stay aligned while navigation chrome, pointer affordances, keyboard behavior, focus treatment, and density adapt to the platform.
Agent-generated UI
Variation needs a reason
Agents should reuse established component families, vocabulary, state patterns, and layout grammar unless the spec names a product reason to introduce a new pattern.

Spec implications

Descriptive specs are the design contract for visible software. They should describe the intended user experience with enough fidelity for implementation to remain architecture-flexible while experience intent remains stable.

Visible feature spec area What the spec should capture
Primary hierarchy The screen purpose, first-priority object or action, secondary information, and what is deliberately de-emphasized.
Dominant action The main user action, when it is available, how it is labeled, and how competing actions are ordered.
Component pattern family The intended family of patterns: table, list, timeline, form, map, split view, card collection, command surface, modal, sheet, tab set, or other named structure.
State set Ready, loading, empty, partial, success, error, permission-denied, offline, completed, selected, disabled, and recovering states that affect comprehension or action.
Responsive collapse rules How navigation, density, grouping, columns, filters, panels, maps, media, and actions adapt at the intended breakpoints.
Density expectation Whether the experience is operational, editorial, exploratory, form-heavy, or review-heavy, and how much information belongs above the fold.
Critical copy Headings, labels, button names, error messages, empty states, confirmations, and recovery guidance that carry meaning.
Accessibility semantics Headings, labels, names, roles, states, focus order, visible focus, contrast-sensitive elements, keyboard behavior, and target-size expectations.
Rendered HTML-native intent Interactive or stateful HTML in the descriptive spec for screens, key states, and critical interactions where visual behavior matters.

Rendered experience is not a placeholder. For visible features, the HTML-native rendered experience is a first-class intent artifact. It can be simple, but it should make hierarchy, copy, state, responsive behavior, and critical interactions inspectable.

Verification

No visually meaningful feature is done without rendered evidence. Passing code tests can prove behavior, but rendered UI evidence proves that the intended experience is visible, coherent, accessible, and responsive.

Evidence What it confirms
Stateful screenshots or trace captures Hierarchy, grouping, spacing consistency, overflow behavior, and empty/loading/error/success/permission states.
Responsive breakpoint captures Meaningful reflow, preserved primary action, coherent navigation, and appropriate density at mobile, tablet, desktop, and wide sizes.
Keyboard and focus evidence Logical focus order, visible focus treatment, operability, and no trapped or unreachable controls where keyboard access applies.
Contrast and target-size checks Readable text, non-color-only meaning, usable targets, and sufficient spacing for the platform and interaction mode.
Rendered spec-to-implementation comparison The implementation matches the descriptive spec's hierarchy, states, copy, responsive rules, and critical interactions.
Done condition for user-facing UI

The review evidence should let a reviewer reject a mediocre interface even when the component tree is clean and the tests pass. The rendered result is part of the contract.
