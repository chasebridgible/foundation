Agent Principles

Durable principles for agent-assisted work. These rules apply when a model uses tools, reads and writes durable artifacts, works across context windows, changes software or operating process, evaluates output, or improves the system that will guide future agents.

Format: HTML-native
Authoring: edit this HTML file
Status: Canonical HTML

This document inherits the domain-neutral rules in core-principles.html and specializes them for agents. An agent principle belongs here only if it stays true across models, tools, vendors, harnesses, and repo shapes.

The central rule is simple: useful agency comes from the joint system, not from a prompt alone. The agent, harness, tools, memory, artifacts, validators, roles, approvals, and revision loop are the thing being designed.

Operating posture

Engineer the agent system, not just the prompt. The agent's capability is the combined behavior of the model, instructions, tools, filesystem, memory, validators, permissions, logs, and handoff rules. A better prompt can improve a run; a better harness can improve every future run.

The loop is the kernel, not the whole system. Tool use in a loop is the smallest useful definition of an agent, but the durable advantage comes from context discipline, explicit artifacts, progressive disclosure, role separation, evaluators, and deterministic control.

Prefer process over prose. Agents do better with executable workflows than with reference essays. A usable procedure names the trigger, context boundary, atomic unit, action, check, revision target, and exit condition.

Make invariants mechanical. Anything that must happen every run should move out of best-effort prompt following and into scaffolding: commands, schemas, hooks, tests, permissions, required files, or branch protection.

Keep global doctrine separate from local truth. Shared agent rules belong in Foundation. Product facts, implementation paths, test commands, ADRs, and exceptions belong in the target repo. Local AGENTS.md files adapt the shared substrate to the current workspace.

Context engineering

Curate working context instead of maximizing raw context. More tokens are not automatically more understanding. Context is knowledge prepared for action: the relevant goal, constraints, state, actor, tools, risk, evidence, and next step for the current job.

Separate durable knowledge from executable context. Stored knowledge, run history, specs, logs, and memory are not the live operating context. The live context should be the smallest complete subset needed to act safely right now.

Keep the always-loaded layer short. The root instruction layer should route the agent to deeper material instead of containing it. Long references belong in specs, skills, reports, and artifacts that load only when needed.

Use progressive disclosure as a reliability primitive. Load the entry skill first, then only the source files, specs, examples, or scripts needed for the current step. This reduces context rot and keeps attention on the active unit of work.

Favor recognition over recall. Agents should be able to find the right context by stable IDs, file paths, metadata, registries, links, and state artifacts. Do not rely on the model to remember where the important fact was mentioned.

Memory

Memory is durable substrate plus task-ready context plus revision. A memory system is not a larger archive. It is the structure that stores what matters, retrieves what is relevant, and changes future action when new evidence survives review.

Externalize state into artifacts. The model window is temporary. Long-running state belongs in files, JSON/JSONL artifacts, specs, run logs, session logs, decisions, issue or PR history, databases, and other durable stores that future agents can inspect.

Memory should serve action. A note matters only if it helps a future actor choose, execute, evaluate, or revise. If remembered material does not change a job, process, decision rule, capability map, example, eval, or artifact, it is archive material, not operating memory.

Encoding quality beats accumulation. Typed, bounded, source-backed entries with stable IDs, timestamps, provenance, confidence, and links outperform undifferentiated logs. A memory that cannot be queried or trusted will not compound.

Memory needs consolidation and forgetting. Keep useful lessons alive by promoting them into durable substrate. Keep retrieval usable by pruning, archiving, or de-emphasizing low-value traces. Scale comes from selective strengthening, not keeping everything equally active.

Memory is fallible unless provenance and evaluation are explicit. Retrieved content is evidence-like, not truth-like. The system must preserve source, time, producer, context, and evaluation state so future agents can decide how much to trust it.

Long-running actions

Design for restartability before autonomy. A serious agent system should be able to stop, compact, resume, fork, hand off, and recover without relying on inherited conversation memory.

The unit of progress must be small enough to verify. Long work should be decomposed into rows, files, slices, targets, issues, or tasks whose output can be checked before the agent moves on. If the unit is too large to review, it is too large to trust.

Every long run needs a durable handoff state. A future agent should be able to tell what was attempted, what changed, what passed, what failed, what remains uncertain, and exactly where to continue.

Clean state is part of the deliverable. Long-running agents should leave workspaces mergeable, artifacts current, reports regenerated, checks run, and blockers named. A messy environment is incomplete work, even if the visible output looks plausible.

Parallelism comes after legibility. Multiple agents help only when ownership, locks, state boundaries, merge rules, and judge responsibilities are explicit. Naive concurrency hides failures behind activity.

Harness and tools

The agent-computer interface is a design surface. Tool names, command contracts, file layouts, schemas, logs, and error messages shape agent behavior as much as instructions do.

Tools should make correct work easier than plausible work. Good tools constrain inputs, reject bulk shortcuts when atomic review is required, emit structured receipts, and make stale or unsupported state hard to miss.

Separate roles when context or incentives conflict. Planners, workers, reviewers, judges, memory writers, and approval routers should not always be the same context. Role separation prevents tunnel vision and makes quality easier to inspect.

Use skills as workflow packages, not knowledge dumps. A skill should tell the agent when to use it, what to load, what loop to follow, what artifacts to update, what shortcuts are forbidden, and what checks prove completion.

Prefer deterministic scaffolding for high-risk boundaries. Permissions, destructive actions, production deploys, customer-facing sends, money movement, legal commitments, and durable memory writes need explicit gates. The harness should enforce the gate, not merely recommend it.

AI evals

Agent Principles defines the operating posture; AI Evals Principles defines the specialist evaluation layer. Use AI Evals Principles when designing rubrics, LLM judges, eval suites, trace review, risk-tiered signals, or quality gates for agent output.

Evaluate the whole system, not the model alone. For agent work, the evaluated object is model plus prompt, tools, harness, memory state, source artifacts, permissions, data, and runtime conditions.

Progress is evidence, not self-assessment. An agent is not done because it says it is done. It is done when external evidence, deterministic checks, reviewer judgment, or eval receipts show that the output matches intent.

Use evals to make failure durable. A failed run should produce better future tests, fixtures, rubrics, prompts, skills, schemas, examples, or handoff rules. If a failure leaves no durable learning, the system is not compounding.

Separate generation from evaluation. The producer of work is biased toward accepting it. Use tests, validators, judges, reviewers, golden examples, scorecards, or manual evidence that stand outside the generator's immediate optimism.

Judge against intent, not fluency. Fluent text, long reasoning, and plausible summaries are not acceptance signals. Evals should ask whether the job was completed, the contract was preserved, the risks were handled, and the next actor can continue.

Calibrate cost to risk. Use the cheapest reliable signal that matches the blast radius. Lightweight schema checks are enough for low-risk structure; high-risk behavior needs broader tests, traces, reviewer evidence, or live calibration.

Self-improvement

Self-improvement is promotion through evidence. A system learns when a finding changes future action after review. It does not learn merely because an agent noticed something, summarized it, or stored it.

Watch broadly, adopt narrowly. A capability scout may monitor changing agent tools and practices, but adoption should require relevance, provenance, a grade or recommendation, and a human-readable reason.

Classify feedback before assigning agent work. Production corrections, user edits, or review comments should become agent tasks only after evidence-backed triage separates actionable repeated failures from preferences, unsupported behavior, ambiguous signals, and expected workflow noise.

Do not let the agent silently rewrite its own operating rules. Changes to specs, skills, validators, memory schemas, source allowlists, or workflows should pass through durable artifacts, checks, and reviewable change flow.

Learning artifacts should be queryable. JSON/JSONL run records, source snapshots, findings, grades, decisions, eval receipts, and merge receipts are better operating memory than chat-only summaries.

Separate scouting from acting. Discovering a new capability is different from using it. The scout produces evidence and a triage signal; follow-up work should start only when the owner chooses to act.

Delegation and authority

Agents extend judgment; they do not own it. A responsible person can delegate research, drafting, coding, testing, ranking, and summarization. The responsible person still owns intent, risk, acceptance, and consequences.

Human approval belongs at irreversible or high-risk boundaries. Production deploys, destructive data actions, customer-facing sends, legal commitments, pricing changes, security-sensitive changes, and operating-rule changes should not cross the boundary invisibly.

Borrowed confidence is a failure mode. Agents can transfer fluent confidence faster than they transfer understanding. Review should ask what evidence exists, what changed, what remains uncertain, and whether the human's mental model improved.

Delegation should sharpen the operator. Good agent use leaves the owner with a clearer model of the system. Bad agent use increases output while shrinking comprehension.

Substrate and handoff

Every useful run should leave substrate behind. Specs, skills, examples, tests, eval fixtures, source records, findings, decisions, and reports are the durable assets that let future agents start from a better system.

Handoff is a first-class output. The next actor should not have to reconstruct the run from chat. Handoff names current state, changed files, evidence, checks, blockers, open questions, and the exact continuation point.

Artifact freshness matters. A stale report, registry, eval receipt, or generated map can be more dangerous than no artifact because it looks authoritative while routing future agents incorrectly.

Reviewable change flow protects memory. Durable memory and operating rules should close through branches, PRs, checks, merge receipts, or equivalent reviewable state. Unmerged notes are hints, not substrate.

The best agent system is inspectable. Humans and future agents should be able to see the capability, job, process, artifact, evidence, evaluation, and revision path without relying on a private conversation.
