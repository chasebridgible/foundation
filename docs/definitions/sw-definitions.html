Foundations (the "you have to know these to read anything else" tier)

Agent. A model plus a harness running in a loop: the model decides what to do, calls a tool, reads the result, decides what to do next. A raw chat model is not an agent. See harness-osmani.md.

API (Application Programming Interface). The "back door" of an app. A list of URLs you can send requests to that return data instead of a webpage. GET /charges returns charges; POST /refunds issues a refund. When an agent "calls Stripe," it's hitting Stripe's API.

CLI (Command-Line Interface). A program you run by typing into a terminal. Takes text in, prints text out. git status, npm install, ls. Agent-native CLIs (see Printing Press) are CLIs whose inputs and outputs were designed assuming an agent is the user, not a human.

Compounding system. A system whose value grows with use rather than decays with neglect. Five layers, each with a different durability profile: substrate (durable, repo-shaped artifacts that survive across model generations — specs, capabilities, knowledge layer, conventions, decision records, domain-specific skills); iterating layers (general harness, framework, vendor wrappers rented from frontier labs and replaced on their cadence); operational shape (who does what — humans, agents, routing); intake layers (where work originates — human input, monitoring, user feedback, ambient ingestion); and the delivery loop (the contract that turns intake into shipped work and substrate updates). The pattern is domain-agnostic: a software-delivery compounding system is one application; marketing, sales, knowledge management, and operations are others, each with their own substrate contents but the same five-layer shape. See sw-principles.md §Compounding systems and substrate.

Data layer. Quantitative signals from the systems a business already runs — HubSpot, Stripe, the production database, the client's SQL warehouse, Slack, Linear. Tells you what happened (revenue last month, churn rate, ticket volume), not what the business intends. Accessed by agents through query tools (SQL, API wrappers, Printing Press-style CLIs) rather than markdown reads. Paired with the knowledge layer but distinct from it; quantitative answers come from here.

Delivery loop. The component of a compounding system that turns intake into shipped work and substrate updates. In this repo's vocabulary, the inner loop is spec → test → prompt → execute and the outer loop is review, integration, rollout, observation, and deprecation. Recursive — large goals contain smaller goals, each running its own cycle. The four steps are the contract; the structure inside them is open. See sw-principles.md §The delivery loop and operating-spec.md §2.

Entity propagation. The knowledge-layer pattern where updating one entity (a person, client, deal, project, meeting) automatically walks the graph and updates every related entity. After a meeting, the meeting page is not the end product — the propagation back to each person's page, each company's page, each related deal's page is. Without propagation, a knowledge layer is a filing cabinet; with it, it's a nervous system. (After Garry Tan.)

Frontmatter. A small block of structured metadata at the top of a markdown file, fenced by --- lines, written in YAML. Used to attach machine-readable info (title, tags, when-to-load rules, owner, type) to a human-readable doc. Static site generators (Jekyll, Hugo, Astro) use it for post metadata; Claude Code skills use it to tell the agent when a skill should be loaded into context. See the dedicated entry below.

Harness. Everything wrapped around the model: system prompts, AGENTS.md, skills, tools, MCP servers, sandboxes, hooks, observability, the loop itself. "Agent = model + harness." A thin harness is the design choice to keep this layer minimal — routing logic only, knows nothing about the domain — and invest the difference into skills, code, and structured knowledge; the alternative is a fat harness that tries to be the system itself and gets shipped past by the next frontier release. (Slogan: "fat skills, fat code, fat data, thin harness," after Garry Tan.) See harness-osmani.md.

Intake layer. The components of a compounding system that turn outside-the-system events into work the system can process. Four common sources: human inputs (you, Slack, conversations); monitoring (Sentry-class bug reports, alerts); user feedback (in-app reports, feature requests, support tickets); and ambient ingestion (background pulls of client SaaS, calendar, transcripts — the equivalent of Tan's 100 crons). All ideally land on the same task surface with metadata that lets the rest of the system prioritize, route, and attribute. The intake layer is where the system meets the outside world.

Iterating layer. The components of a compounding system that get replaced on the frontier labs' cadence — the general harness, the runtime, framework choices, standard tool wrappers, vendor SDK wrappers. Useful while it's the best available; rented leverage rather than owned asset. The complement of substrate. The litmus test: would a frontier lab plausibly ship a better version within a year? If yes, iterating layer; treat investments here as short-term advantage with a known maintenance cost.

Knowledge layer. The captured operating mind of a business — what it sells, who it sells to, what it promises, who decides what, what the SOPs are, what the constraints are. Lives in versioned markdown with frontmatter (operating model / control / execution), curated and reviewed on a cadence, accessed by agents through a knowledge MCP server. Answers questions like what's our refund policy? or who approves a deploy? — things the business knows about itself that no external system records. The data layer says what happened; the knowledge layer says what we meant. See knowledge-base/.

MCP server (Model Context Protocol server). A standard way to package a bunch of tools so any agent runtime (Claude, Cursor, Codex) can use them without bespoke integration code. Think "USB-C for agent tools." Write an MCP server for Linear once; every MCP-speaking agent can now use Linear.

Operational shape. The component of a compounding system that names who does what — which humans hold which decisions, which agents do which work, which queues route what to where. Distinct from substrate (the durable artifacts), iterating layers (the rented runtime), intake layers (the surface to the outside world), and the delivery loop (the work contract). For this repo's productized-agency context, the current operational shape is described in operating-spec.md §7.

Registry file. A standalone, structured file (usually YAML or JSON) that lives at a known location in a project and describes "what exists in this system" — clients, dependencies, environments, workflows. Examples: package.json, tsconfig.json, .gitignore, docker-compose.yml, clients.yml. Read by humans, scripts, agents, and CI as the single source of truth for the facts they describe. See the dedicated entry below.

Skill. A markdown file (with frontmatter) that tells the agent, "when the user asks for X, follow these steps." A workflow, not a reference doc. Loaded only when relevant ("progressive disclosure"). See skills-osmani.md.

Skillify. The meta-skill that observes a recent workflow, extracts the repeatable pattern, names the triggers and edge cases, writes a tested skill file (markdown + frontmatter), and registers it where the harness can find it next time. The mechanism that turns one-off work into permanent substrate; without an equivalent move, workflows accumulate rather than compound. The capability-extraction loop (every finished client feature → portable capability) is the same pattern scaled from per-workflow to per-feature. (After Garry Tan.)

Substrate. The durable, repository-shaped component of a compounding system — specs, capability inventory, knowledge layer, repository conventions, domain-specific skills, decision records — whose value compounds across model generations. The defining test (one-shot regeneration): a future model with a better harness, reading only the substrate, could rebuild a better working version of the system from scratch. The substrate iterates — entries are edited, categories are added, the implementation beneath it is replaced — but the substrate is what survives iteration; everything else in a compounding system is part of an iterating, operational, intake, or loop layer. The substrate is the durability lever, not the value generator: the part that makes today's good work still be good work in two years. See the expanded entry below.

Tool (in agent-land). Something an agent can call to do work in the outside world. Reading a file, searching the web, calling Stripe's API. Each tool has a name, a description, and a schema, all of which the model reads on every turn — so tool design is part of harness design.

YAML. A plain-text format for writing structured data (key/value pairs, lists, nested objects) in a way humans can read and edit. The format frontmatter is written in. Also the format of GitHub Actions workflows, Docker Compose files, Kubernetes manifests, most CI configs, and a huge amount of agent / LLM tooling. Indentation-sensitive (two spaces, never tabs), uses key: value for fields and - item for list entries. See the dedicated entry below.

Specs

Three flavors of spec recur in spec-driven development. Each answers a different question, and a complete piece of work touches all three at least once. See operating-spec.md §3 for the long form.

Descriptive spec. How the application should look and feel to the user — the full user flow, conditional behavior, and edge cases. Answers what experience are we building? The place where vagueness gets discovered: "the popup appears when the user saves a clip" is too thin; "a centered modal slides up over 200ms with a confetti burst, dismissed on tap-anywhere or after 1.5 seconds" is a spec.

Technical spec. How the system functions internally — what should happen, in what sequence, with what timing. Answers which implementation contract should the agent follow? Required for every software task. The depth scales with the work: small tasks get a compact technical note; complex systems get deeper coverage of data, contracts, timing, concurrency, integrations, migrations, failure modes, and boundaries. Rule of thumb: never skip the technical spec; scale its detail to the task.

Test spec. Plain-English description of how the system will be verified. The human's seat at the table. Code-level tests (unit, integration, smoke) can be regenerated by the agent at any time; the test spec is what the human reads to decide whether the right things are being checked. When the descriptive spec changes, the test spec changes; when the test spec changes, the code-level tests are regenerated.

Harness concepts

AGENTS.md (also CLAUDE.md, etc.). The rolling rulebook that gets injected into the system prompt every turn. Highest-leverage configuration point. Should be a pilot's checklist (short, every line earned), not a style guide.

Compaction. Summarizing and offloading older context when the window approaches full so the agent can keep working. Necessary but not sufficient for very long runs.

Context rot. The observed degradation in reasoning and task completion as the context window fills. Why "load less, load just-in-time" matters.

Hook. A script that runs at a specific lifecycle point (before a tool call, after an edit, on commit, on session start). Where deterministic enforcement lives. Convention: silent on success, verbose with an actionable error on failure.

Progressive disclosure. Loading instructions, tools, or skills only when the current task calls for them, instead of stuffing everything into context at startup. The mechanism that lets a 20-skill library fit into a small context slot.

Ralph loop. A wrapper that intercepts the agent's "I'm done" signal and re-injects the original goal into a fresh context window. Each iteration starts clean but reads state from disk left by the previous one. Turns a single-session agent into a multi-session one.

ReAct loop. The standard agent loop: model reasons, takes an action via a tool call, observes the result, repeats. "An agent is a system that runs tools in a loop to achieve a goal." (Simon Willison)

Sandbox. An isolated execution environment with allow-listed network and filesystem access, where the agent can run bash and code without breaking the host machine. Hosts the "Hands" half of Brain/Hands/Session.

Sprint contract. A done-condition negotiated and written down before the agent starts. The single highest-leverage move against scope drift on long runs.

Long-running agent concepts

Brain / Hands / Session split (Anthropic). Three independently-replaceable layers: the model + harness loop (Brain), the sandboxed execution environment (Hands), and the append-only event log (Session). Crash a brain, recover from the log; throw away a sandbox, get a clean one.

Cross-modal evaluation. Multi-model scoring pattern: send the same output through several models with different strengths (one tuned for precision, one for recall, one for creativity) and have them score each other on the dimensions that matter. The operational form of separate generation from evaluation (see core-principles.md §Verification) — catches failure modes a single model can't see in itself, including its own confident errors. (Garry Tan's framing; sibling pattern to Cursor's planner / worker / judge.)

Long-horizon reasoning. The model can plan and execute over many dependent steps. A model-quality story.

Long-running execution. The agent's process runs for hours or days; the model is invoked thousands of times. A harness story. (Distinct from long-horizon.)

Memory Bank (Google). Curated long-term memory layer scoped to a user identity, with a search API. The "agent remembers what you usually mean by 'the dashboard'" layer.

METR time horizon metric. An estimate of how long a task a frontier model can complete with 50% reliability. Has been doubling roughly every seven months. Useful for "is this delegable yet" sanity checks.

Planner / Worker / Judge (Cursor). Planners explore and emit tasks; workers execute without coordinating; judges decide when an iteration is finished or must restart. A real architectural pattern even when all three are the same model with different prompts.

Three walls. Finite context, no persistent state, no self-verification. Every long-running architecture is some answer to those three.

Cognitive concepts (the "what's happening to the human" tier)

Borrowed confidence. Adopting the model's certainty as your own without inheriting any of its reasoning. The mechanism by which surrender starts.

Cognitive offloading. Delegating mechanics to a tool while still owning the answer. You judge sensibility, intervene when results are off. Healthy mode.

Cognitive surrender. The point at which you stop constructing an answer at all. The model's output becomes yours by default. Failure mode that looks identical to offloading from the outside.

Comprehension debt. The growing gap between how much code exists in your system and how much any human genuinely understands. The bill for repeated surrender.

Mutual amplification (Andy Clark). A cooperation loop where your prompts sharpen the model's output, which sharpens your next prompts, which sharpens your model of the problem. The "I end the session smarter" outcome.

Scaffolded cognitive friction. Deliberately introduced moments of resistance — design docs, confirmation steps, checklists — that interrupt heuristic acceptance.

Process patterns

Anti-rationalization table. A pre-written list of common excuses for skipping a rigorous step, paired with rebuttals. The point: don't argue with the model (or yourself) on the day; settle the argument in advance. See skills-osmani.md.

Beyoncé Rule. "If you liked it, you should have put a test on it." — Software Engineering at Google. If a behavior matters, encode it as an automated check.

Chesterton's Fence. Don't remove a thing until you understand why it was put there.

DAMP over DRY (in tests). Descriptive And Meaningful Phrases over Don't Repeat Yourself. Tests should read like a specification even at the cost of duplication.

Hyrum's Law. Every observable behavior of your API will eventually be depended on by someone. Design accordingly.

Scope discipline. "Touch only what you're asked to touch." The single biggest determinant of whether an agent's PR is mergeable or has to be unwound.

Vertical slice. A unit of work that owns a coherent concept end-to-end (UI through API through DB), instead of being decomposed by layer (all UI, all API, all DB). Promptable, reviewable, and testable as a unit.

Verification as exit criterion. Every task terminates in concrete evidence — a test that runs, a screenshot, a runtime trace, a reviewer signoff. "Looks right" doesn't close the loop.

Git / version control

Branch. A named pointer to a commit in git history — main, feature-x, chase/refactor-clients. Cheap, logical, lives inside .git/. Switching branches in a normal checkout moves HEAD and rewrites the files in your working directory underneath you. The label on the door, not the room itself.

Diff. The exact set of changes between two versions of code, docs, config, or data — what was added, removed, or modified. In software practice, a good diff is small enough to review, scoped to one intent, free of unrelated formatting churn, and backed by the evidence needed to trust it: tests, screenshots, traces, or a clear explanation of why verification is not needed.

Worktree. A separate working directory on disk tied to a different branch of the same repo. git worktree add ../core-templates-feature-x feature-x gives you a second folder you can cd into, edit, build, and run independently — while your original checkout stays on main, untouched. One repo, many branches; one repo can also have many worktrees, each parked on a different branch. The room; the branch is the label on the door. Why agent harnesses care: spawn five background agents in one repo and they'd fight over the same files. Give each its own worktree and they can work in parallel and merge back via PR — the pattern Cursor background agents and the best-of-n-runner subagent use.

The frontmatter entry, expanded

Frontmatter is a YAML block at the top of a markdown file, fenced by two --- lines. It holds structured metadata — things a human could parse from the prose, but a machine can't be expected to. Example, from a Claude Code skill:

---
name: test-driven-development
description: Use when implementing a new behavior. Write the failing test first.
when_to_use:
  - new feature
  - bug fix where root cause is unclear
owner: chase
---

# Test-Driven Development

(Body of the skill goes here.)

The lines between the two --- are YAML key/value pairs. Everything after the closing --- is the normal markdown body.

Why it exists. Markdown is great for humans but invisible to programs. Frontmatter is the side channel. A static site generator reads title: and date: to render a blog index. A Claude Code agent reads description: and when_to_use: to decide whether to load a skill into context for the current task. A zettelkasten system reads tags: to build a graph between notes.

Why this matters for what we're building. The agent only sees the frontmatter at first — that's how it decides which skills are relevant without loading the whole body. Bad frontmatter, the right skill never activates. Good frontmatter, you get progressive disclosure for free. It's the metadata layer that makes "load just-in-time" work in practice.

Common fields you'll see in agent skills:

  • name — slug used to reference the skill.
  • description — one-line "when does this apply." Critical: this is what the router reads.
  • when_to_use / triggers — list of situations or task types.
  • tags — categorical handles for grouping and search.
  • owner — who maintains it.
  • version — for ratcheting and rollback.

Common fields you'll see in blog/note systems:

  • title, date, author, slug, tags, summary, draft.

TOML and JSON variants. Same idea, different syntax. TOML uses +++ instead of ---; JSON uses {}. YAML is by far the most common in the agent world.

Failure modes worth knowing.

  • Indentation matters in YAML. Two spaces, not tabs. Quoting strings is sometimes required (anything starting with : or containing #).
  • Frontmatter that contradicts the body. description: says one thing, the body does another. The agent reads both at different times and gets confused.
  • Frontmatter as a dumping ground. If every skill has 30 fields and only 4 of them get read, you've recreated the "essay nobody reads" problem at the metadata layer.

The YAML entry, expanded

YAML stands for "YAML Ain't Markup Language" (originally "Yet Another Markup Language" — they renamed it once it caught on). It's a way of writing structured data — the same kind of data you'd put in a JSON file or a database row — but in a format that reads more like a list someone wrote in a notebook than like code.

A small example:

name: chase
role: founder
projects:
  - core-templates
  - sandia-oil
preferences:
  editor: cursor
  shell: zsh

That's three things:

  • A top-level string (name: chase).
  • A list under projects:, where each - item is one entry.
  • A nested object under preferences:, with its own keys indented underneath.

The same data in JSON would be:

{
  "name": "chase",
  "role": "founder",
  "projects": ["core-templates", "sandia-oil"],
  "preferences": {
    "editor": "cursor",
    "shell": "zsh"
  }
}

Both express the same structure. YAML wins when humans need to write or edit it; JSON wins when programs need to exchange it over a network.

Where YAML shows up in your world right now:

  • Frontmatter in markdown files (skills, blog posts, notes).
  • GitHub Actions workflows (.github/workflows/*.yml).
  • Docker Compose files (docker-compose.yml).
  • Kubernetes manifests (every Deployment, Service, ConfigMap).
  • CI configs for CircleCI, Travis, GitLab CI.
  • Most agent/LLM tooling configs (skill definitions, prompt templates, eval harnesses).

If you're going to spend the next two years configuring agent harnesses, you will write more YAML than you do English. Worth getting comfortable with.

The five things to know to read and write it:

  1. Indentation is structural, and must be spaces (two by convention, never tabs). The indentation level is the nesting level. This is the thing that bites everybody — copying YAML from a webpage often replaces spaces with tabs and the file silently breaks.
  2. key: value for fields, - item for list entries. That's 90% of the syntax.
  3. Strings usually don't need quotes. name: chase is fine. You only need quotes when the value contains characters YAML would otherwise interpret — :, #, leading/trailing whitespace, or values that look like other types (yes, no, true, false, numbers).
  4. # starts a comment. Everything after it on that line is ignored.
  5. --- on its own line is a document separator. That's why frontmatter is fenced by --- lines: technically the file is "an empty YAML doc, then your markdown content."

The classic YAML gotchas, in order of how often they bite people:

  • The Norway Problem. country: NO becomes the boolean false (because YAML 1.1 treats no, yes, on, off as booleans). Quote it: country: "NO".
  • Numbers that aren't. version: 1.10 becomes the number 1.1 (because trailing zeros). If it's a version string, quote it: version: "1.10".
  • Indentation drift. Two spaces here, four spaces there, mixed tabs anywhere — the file parses as something other than what you meant. Most editors have a "show whitespace" toggle; turn it on when debugging.
  • Multiline strings. description: | keeps newlines, description: > folds them into spaces. Easy to grab the wrong one.

A heuristic for when to use YAML vs. JSON vs. TOML.

  • YAML when humans write and edit it (config files, frontmatter, manifests).
  • JSON when programs exchange it (APIs, structured logs, anything machine-to-machine).
  • TOML when you want the readability of YAML without the indentation traps (Rust's Cargo.toml, Python's pyproject.toml).

For the knowledge base you're building, YAML is the right default for frontmatter and skill metadata. JSON is what you'll see come back from API calls inside CLIs and MCP servers. TOML you'll only meet if you write Rust or modern Python tooling.

The registry file entry, expanded

A registry file is a standalone, structured file (almost always YAML or JSON) that lives at a known location in a project and answers the question, "what exists in this system?" Every healthy project ends up with a small constellation of them. Examples already familiar from working in any modern repo:

Project-wide
Registry file
Structured data about the whole system, read by humans, agents, scripts, and CI.
One document
Frontmatter
Metadata that describes a single markdown file and helps routers load it.
Explanation
Prose
Narrative context, rules, examples, and rationale that humans need to read.
  • package.json — what npm packages does this project depend on?
  • tsconfig.json — how should TypeScript be configured?
  • .gitignore — what files should git ignore?
  • docker-compose.yml — what containers does this project run?
  • .github/workflows/*.yml — what CI runs on what triggers?
  • clients.yml (in this repo) — what clients does this project serve?

Same shape every time: data the system needs, in a known location, in a machine-readable format, separate from any prose that explains how to use it.

Why registries exist

The alternative is hard-coding facts into prose. The list of clients ends up scattered across CLIENT_SETUP.md, AGENTS.md, three bash scripts, and somebody's memory. Adding a client means editing all of those. Removing one means hoping you found every reference.

A registry collapses that into one file. "The list of clients" lives in exactly one place. Humans, scripts, agents, CI, and future-you all read from the same source. Add a client → edit the registry → everything else picks it up.

When to make something a registry file

Use a registry file when all of these are true:

  • The data describes the project, not a specific document. (If it's about one file, use frontmatter.)
  • The data is structured — fields, lists, nested objects, not freeform prose.
  • The data is referenced from multiple places — scripts, agents, runbooks, other configs.
  • The data is edited by humans more often than by machines. (If machines own it, it's probably state and belongs in a database.)
  • The dataset is small to medium — tens or low hundreds of entries, not thousands.

When any of those is false, pick a different home:

  • "Describes one document" → frontmatter.
  • "Freeform prose" → markdown body.
  • "Only used in one place" → inline constant, env var, or config object in code.
  • "Machines write to it during runtime" → database / state store.
  • "Thousands of rows" → database.
  • "Secrets" → secrets manager (AWS Secrets Manager, 1Password, Doppler). Reference by name from the registry, never embed the secret itself.

The decision rule, said plainly

Note

Three places data can live: standalone registry files (about the whole project), frontmatter (about one document), prose (about whatever the prose is talking about). Pick by scope, not by mood.

What good registry hygiene looks like

  • Known location. clients.yml at the project root, not buried three folders deep. The whole point is that anyone (human or agent) can find it without looking.
  • One job per registry. clients.yml lists clients. Don't also stash default_aws_region or default_branch in there unless it's per-client. System-wide defaults belong in their own file or in env vars.
  • Reference, don't duplicate. If clients.yml has an aws_profile field, every script and runbook should read that field — not hard-code the profile name. The registry is the source of truth precisely because everything else points at it.
  • Validate it. Either with a JSON Schema (for JSON), a YAML schema, or a CI check that loads the file and asserts shape. A registry that drifts out of shape silently is worse than no registry at all.
  • Don't put secrets in it. Reference where the secret lives (secrets_vault: clients/example-client). Never the secret itself. The registry is committed to git; secrets are not.
  • Document the schema once, somewhere obvious. Either via comments in the file itself, a sibling clients.schema.yml, or a section in AGENTS.md / CLIENT_SETUP.md. If anyone has to guess what default_branch means, it's not yet a registry — it's just a YAML file.

What this unlocks for an agent system

Once a registry exists, you can do things you can't do with prose:

  • AGENTS.md can say: "Before doing client work, read clients.yml for the client's aws_profile and secrets_vault." Now every agent that loads that rulebook knows where to look.
  • A bash script can do: yq '.clients | keys' clients.yml and loop over the slugs.
  • A Ralph-loop wrapper can do: read the registry, pick the next unfinished entry, run the agent on that one entry.
  • CI can do: validate that every client referenced anywhere in the repo has a corresponding entry. Catches typos before they hit production.
  • Onboarding becomes: add an entry to the registry, run a setup command, and the system fills in the rest using the runbook as the procedure.

The general pattern: registries are how you stop hard-coding facts into prose. Once the registry exists, every other file in the system can stay generic — and the truth lives in one place that's easy to read, easy to diff, and easy to audit.

The substrate entry, expanded

A substrate is the durable, repository-shaped component of a compounding system that captures the work no frontier lab will ever do for you, arranged so that a more capable model and a better harness can do strictly better work off the same artifacts over time. It's the part of a compounding system whose value compounds across model generations — distinct from the iterating layers (specific harness configs, runtime wiring, framework choices, vendor integrations) that get obsoleted on the labs' cadence.

The substrate is one of five layers in a compounding system. The others are iterating layers (rented from the labs), operational shape (who does what), intake layers (where work originates), and the delivery loop (the contract). All five are necessary for a compounding system to function; only the substrate is what survives across model generations. The substrate is the durability lever, not the value generator.

The substrate iterates too. Entries are edited, categories are added, the implementations beneath them are replaced. But the substrate is what survives the iteration — everything outside it is part of one of the other four layers, each with its own design rule and lifecycle.

The defining test: one-shot regeneration

The substrate is whatever passes this thought experiment:

Note

A future model + future harness, given only the substrate artifacts (specs, capabilities, knowledge layer, conventions, decision records, domain skills), can regenerate a working version of the system from scratch — and a better one than today's setup can produce.

What's outside the substrate is whatever fails that test: the specific runtime wiring, the bespoke harness scaffolding around a current model's weak spots, the integration code for whichever vendor is on top this quarter. All useful at the time. None durable across model generations.

This is not "documentation." Documentation describes what exists. The substrate is the source of truth for what should exist; the implementation is derived from it, not the other way around.

This list is the working answer. It is not closed — categories will be added as new things prove they pass the one-shot test.

  1. Specs. Descriptive, technical, and test specs (see the Specs section above). The precision of intent for each piece of work.
  2. Capability inventory. The shared library of vertical slices (capabilities/), each portable by folder copy. The accumulated answer to "what have we built that's reusable across clients." A capability's implementation is rebuildable; what's substrate is the fact that the capability exists, the shape it has, and the specs it satisfies.
  3. Knowledge layer. The captured operating mind of the business — knowledge-base/ content, the operating model, control, and execution docs. The data layer (HubSpot, Stripe, the SQL warehouse) is not substrate; it's signal flowing through external systems the business doesn't own. One page schema worth flagging as a candidate for entity-style entries (people, clients, ongoing relationships): Garry Tan's brain page shape — compiled truth (current state) at the top, append-only timeline (events in chronological order) below, raw data sidecars for source material. Event-sourced rather than rule-based; complements rather than replaces the current canonical-doc-template.md shape.
  4. Domain-specific skills and tools. Workflows and CLIs that only matter inside our domain — done-with-you-intake, capability-extraction skills, client-specific Printing Press CLIs. Workflows no frontier lab will ever ship because they're too specific to be commodified.
  5. Repository conventions. How files are laid out, how things are named, what folder structure means, what the spec/test/prompt/execute loop expects. AGENTS.md is the rolling rulebook; the conventions encoded around it (vertical-slice folders, the knowledge-base/ layout, frontmatter contracts) are the rest. A new model with a better harness still needs to know "this is how we do things here."
  6. Decision records. The captured why — ADRs and equivalent that record reasoning behind paths taken and roads not taken. Without these, every future agent re-litigates settled decisions. Frontier labs will not write your ADRs.

What's not substrate

  • The model. Always commodity.
  • The general harness. Skill routers, sandboxes, observability, the meta-loop — labs ship these on their cadence.
  • Standard tool wrappers. MCP servers for popular SaaS (Linear, Stripe, Notion) — somebody will ship them better than you can.
  • The data layer itself. HubSpot, Stripe, your SQL warehouse. You don't own these; you query them.
  • Today's framework choices. Expo, Fastify, Postgres, Cognito — replaceable.
  • Specific orchestration wiring. Whichever board, queue, or wrapper happens to run the loop this quarter.

Same test in every case: would a frontier lab plausibly ship a better version within a year? If yes → iterating layer, treat as rented leverage. If no → substrate candidate.

Open: what else might belong

The taxonomy will evolve. Candidates considered but not yet committed:

  • Validation gauntlets — schema validators, capability portability checks, frontmatter validators. They encode "what good looks like," which leans substrate. Tightly coupled to repository conventions; may fold into that category rather than stand alone.
  • Reference artifacts — exemplar specs, capabilities, and knowledge entries that show the shape. Could be its own category; could live alongside each substrate category as "examples of how to do this here."
  • Style and voice guides — for client-facing copy, naming, communication. Domain-specific, not commodity. Probably substrate, not formalized yet.
  • Ambient ingestion pipelines. Background cron jobs that pull client systems (Slack, CRM, calendar, metrics feeds) into each client's knowledge layer on a schedule, not on demand. The thing that keeps the knowledge layer warm between agent runs. Garry Tan's setup runs 100+ such crons; our current operating shape (operating-spec.md §7) is reactive by default. Leans substrate when the ingestion pipeline shape is domain-specific (each pipeline encodes "what to capture from this kind of system, how to normalize it, where to file it"); leans iterating-layer when it's a thin wrapper around a vendor SDK that the labs will eventually ship better. Worth a separate decision once the first such pipeline ships.

The rule for adding a new category is the same as the rule for defining the substrate: does it pass the one-shot regeneration test? If you can throw away the implementation and the substrate alone can rebuild a better version, the substrate is complete. If not, something is missing — and that something is the new category.
